Capstone C1 — Unity Airways Support RAG Knowledge Base · Build after Level 2 / Module 05 · [Project]

Support RAG Knowledge Base

Turn a folder of airline policy documents into a governed, registered RAG chain that answers support questions with citations. Ingest → chunk → index → chain → register — the first capstone, and the one that creates the Unity Airways artifacts every later capstone extends.

01 · Scenario

A support desk drowning in policy PDFs

Unity Airways' support agents field the same policy questions all day — refunds on Basic Economy, checked-bag fees, cancelled connections, booking changes. The answers all exist in official documents, but they are scattered across thousands of files, so agents burn minutes hunting for the exact clause.

The support lead wants a knowledge base an agent can ask in plain language and get back a grounded answer plus the document it came from — so the agent can verify it and paste the citation into the ticket.

The business problem, precisely:

  • The corpus is real-world messy: typed policy PDFs, an FAQ dump, and image-only scanned refund forms.
  • Questions are worded nothing like the documents. "Can I rebook for free if my connection is cancelled?" must find a chunk that says involuntary re-accommodation and never uses the words "rebook" or "free".
  • A confident wrong answer is worse than none. Every response must be traceable to a source chunk, or an agent could relay a policy the airline doesn't have.

Success criteria — what "done" means for the business:

  • An agent asks in natural language and gets a correct, grounded answer in seconds.
  • Every answer names which source document backs it.
  • When a policy PDF changes, refreshing is incremental — re-index what changed, not the whole corpus.
  • The result is a governed, registered UC model, not a notebook someone re-runs by hand.
📌 Important"Answers with citations" is the acceptance bar, not a nice-to-have. A chain that returns fluent text with no traceable source has failed this capstone even if the prose reads well. Groundedness and the source reference are what make it usable at a support desk.
02 · What you'll build

One spine, three named artifacts

Objective: turn a folder of Unity Airways documents into a queryable, registered RAG chain — end to end, on Databricks, with governance and versioning baked in. The concrete target in one line: an ingestion pipeline → AI Search index → a registered, queryable RAG chain that answers policy questions with citations.

unity_airways.rag.ua_rag_chunks
Governed Delta chunk table — one clean, self-contained row per chunk, with metadata and Change Data Feed on. Built in M2 (Module 03.6).
unity_airways.rag.ua_rag_chunks_index
Delta Sync Index with managed embeddings from databricks-gte-large-en, on a Standard endpoint. Built in M3 (Module 04.3).
unity_airways.rag.ua_rag_chain
Registered UC model — the LCEL chain over databricks-claude-sonnet-4-5, logged as Models-from-Code and versioned as a LoggedModel. Built in M4–M5 (Module 05).

One design choice worth naming up front: the chain's prompt is not a string baked into the code. You author it once and register it in the MLflow Prompt Registry (Module 02.5, Beta) as unity_airways.rag.ua_rag_prompt, then the chain loads it by URI — so the prompt is a governed, versioned Unity Catalog artifact from day one. Here you author v1; C2 versions and evaluates it later.

💡 TipBuild in milestone order. Each output feeds the next — the table feeds the index, the index feeds the retriever, the retriever feeds the chain, the chain gets logged and registered. Building the index before Change Data Feed is on the table is the fastest way to a mid-build stall.
03 · Prerequisites

What must be true before you start

  • Modules 00–05 ✅: UC/Volumes/Delta + the Unity Airways use case (00); embeddings (01.3); prompt engineering + Prompt Registry (02, incl. 02.5); parsing & chunking (03); embeddings & AI Search (04); the RAG chain + packaging (05).
  • Workspace: Unity Catalog enabled; rights to create a schema, UC Volume, tables, a Vector Search index, and a UC model in unity_airways.rag.
  • Compute: serverless or a recent DBR-ML runtime. ai_parse_document needs a recent DBR/serverless (docs disagree: DBR 17.3+ on the function reference vs 18.2+/serverless v3+ on the overview — verify in-workspace). Not a Classic/Pro SQL warehouse.
  • MLflow ≥ 3.1 (Models-from-Code, set_active_model/LoggedModel).
  • MLflow Prompt Registry (Beta): mlflow[databricks]>=3.1.0, a UC schema (unity_airways.rag) where you hold CREATE FUNCTION + EXECUTE + MANAGE, and the feature enabled on the workspace Previews page — used to register and load the chain's prompt (02.5).
  • FM endpoints: databricks-gte-large-en (embeddings) and databricks-claude-sonnet-4-5 (chat) — confirm names on the supported-models page; they churn.
  • Packages: databricks-vectorsearch (the VectorSearchClient SDK), databricks-langchain (ChatDatabricks, DatabricksVectorSearch, DatabricksEmbeddings), and mlflow.
⚠️ GotchaThe package is databricks-vectorsearch even though the product is now Databricks AI Search — there is no databricks-ai-search package and no AISearchClient. And the LangChain integration is databricks-langchain, not langchain-databricks or langchain_community. Import errors from getting these wrong are the two most common build-blockers.
04 · Explore the brief

Scenario, architecture, milestones, rubric

Input: a UC Volume of Unity Airways policy PDFs, an FAQ export, and scanned refund forms.

Output: a registered RAG chain that answers support questions in natural language, grounded in the docs, and names its source.

The hard part: questions don't share vocabulary with the documents, the source files are mixed-quality, and the answer has to be traceable — fluent-but-uncited fails.

Everything left of the RAG chain is Modules 03–04; the chain and its registration are Module 05. The dashed line is the incremental refresh — the Delta Sync index reads the table's Change Data Feed and re-embeds only changed rows.
📌 ImportantThe retriever the chain calls is the exact object Module 04 built — Module 05 does not re-create it. The one hand-off from "retrieval" to "application" is DatabricksVectorSearch(...).as_retriever(...). Keep the endpoint name, index name, and returned columns identical on both sides or the chain retrieves nothing.

Five milestones, each mapping to specific module topics, each with an acceptance criterion you check before moving on. Track your build in the checklist under section 05.

CriterionNot yetMeetsExceeds
Retrieval relevance Semantic queries miss obviously relevant chunks; keyword-only matching Top-k returns the correct chunk for paraphrased questions ("rebook for free" → re-accommodation) Adds HYBRID and/or a reranker so the best chunk lands at rank 1; measured on a labeled set (precision/recall/MRR)
Answer groundedness / citations Fluent answers with no source, or unsupported by any retrieved chunk Every demo answer is grounded in a retrieved chunk and names the source document Refuses/hedges when context is missing rather than hallucinating; citation names the specific source + chunk
Chunking quality Fixed-length splits break sentences; boilerplate pollutes chunks; whole docs embedded Content-appropriate chunking (paragraph/sentence + overlap ~200 tokens); noise filtered; one idea per chunk Tuned against retrieval metrics; scanned forms handled cleanly; metadata rich enough for filtered retrieval
Runs from the registered model Only runs in the notebook; not registered, or load_model fails / 401s ua_rag_chain registered; load_model(...).invoke(...) reproduces the answer with signature + resources present Versioned LoggedModels with params; ready to serve; rollback-friendly aliasing considered
Prompt registered & loaded by URI Prompt is an inline string literal in the chain code — no version, no governance RAG prompt registered as unity_airways.rag.ua_rag_prompt (v1); chain loads it by URI (prompts:/unity_airways.rag.ua_rag_prompt/1), not a hardcoded string Prompt carries commit message + tags, loaded by alias (@champion) for promotion — ready for C2 to version + evaluate
Code reproducibility Config hard-coded; artifacts on /mnt/...; no version linkage Config in YAML (ModelConfig); UC-first tables/Volumes; Models-from-Code saves source, not a pickle Version named by Git SHA; DBR + source Delta table version recorded; a fresh run rebuilds the same result

"Meets" is the pass bar. The two criteria that separate passing from excellent are groundedness/citations and reproducibility — a chain that answers well but can't be reloaded from the registry, or that cites nothing, is not production-ready.

    06 · Deliverables

    Three artifacts + a five-question demo

    Submit the three named artifacts (built across M2–M5) plus a short demo that proves the chain works:

    • unity_airways.rag.ua_rag_chunks — governed Delta chunk table (CDF on).
    • unity_airways.rag.ua_rag_chunks_index — ONLINE Delta Sync index (managed embeddings).
    • unity_airways.rag.ua_rag_chain — registered UC model, Models-from-Code, ≥ 1 LoggedModel version.

    Demo — five representative policy questions answered by the registered chain, each with its source cited:

    1. "Can I get a refund on a Basic Economy fare?"
    2. "Can I rebook for free if my connection is cancelled?"
    3. "What's the checked-bag fee?"
    4. "How do I change an existing booking?"
    5. "What happens if my flight is delayed overnight?"

    Each answer must be grounded in a retrieved chunk and name where it came from. An ungrounded or uncited answer fails the demo.

    07 · Stretch goals

    Once the five milestones pass

    • Hybrid search (04.8) — switch queries carrying booking codes, fare classes, or airport codes (UA-8842, fare Q, SFO) to query_type="HYBRID" so exact tokens match alongside semantic recall. ~2x ANN cost — use it selectively.
    • Reranking (04.9) — retrieve a wider candidate set (top 25–50), run a cross-encoder over the (query, chunk) pairs, keep the top 5. Fixes "right chunk retrieved but ranked 4th."
    • Lakeflow / SDP pipeline (03.9) — rebuild M1–M2 as a Lakeflow Declarative Pipeline: bronze (FROM STREAM read_files(...)), silver (ai_parse_document + chunk). Incremental, serverless, UC-governed.
    • Metadata filters (04.4) — add filters="source_doc = 'checked_bag_fee'" so a baggage question can't retrieve refund-policy chunks (source_doc is the PDF filename stem). Match the filter syntax to the endpoint type.

    09 · Field notes

    What to remember

    📌 ImportantKeep the canonical names: catalog unity_airways, schema rag, and the three artifacts ua_rag_chunks / ua_rag_chunks_index / ua_rag_chain. C2, C3, and C4 all extend these exact artifacts — renaming here breaks continuity down the track.
    💡 Tip · fieldThe retriever is the seam between "we have a vector index" and "we have an app." When a served chain works in the notebook but 401s in production, the cause is almost always a missing resources=[...] at log time — declare every endpoint and index the chain touches so auto-auth passthrough can provision credentials.
    ⚠️ GotchaFlavor-logging the chain object hits Failed to save runnable sequence because VectorStoreRetriever isn't natively serializable. Log the .py path with mlflow.models.set_model(chain) (Models-from-Code) — it saves source instead of a pickle and sidesteps serialization entirely. Re-verify the served-model names (databricks-gte-large-en, databricks-claude-sonnet-4-5) at build time; they churn.
