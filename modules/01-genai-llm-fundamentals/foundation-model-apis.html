Foundation Model APIs and External Models

How you actually reach a model on Databricks — hosted or third-party — through one governed serving surface, with a path from prototype to production that barely changes your code.

01

The big idea

Everything on Databricks reaches a model through Mosaic AI Model Serving. Design your app against "a serving endpoint" and you can swap what's behind it without a rewrite.

  • Three families: Foundation Models (Databricks-hosted), External Models (governed proxy to third parties), Custom Models (your own).
  • Two Foundation Model API modes: pay-per-token (start here) and provisioned throughput (production, guarantees, fine-tuned weights).
  • External Models put OpenAI / Anthropic / Google behind one endpoint, with the key in a Databricks secret.
  • Five ways to call it: REST, OpenAI-compatible client, MLflow Deployments SDK, ai_query() in SQL, and ChatDatabricks.
  • Endpoint names churn — treat any name here as a dated snapshot and verify on the supported-models page.

03

Pay-per-token vs provisioned throughput

  • Pay-per-token — Databricks hosts a shared endpoint; billed per token. The docs call it "the easiest way to start" and note it is not designed for high-throughput apps. Best for prototyping and evals.
  • Provisioned throughput — reserved, dedicated capacity with performance guarantees, support for fine-tuned / custom weights, and stronger security (e.g., HIPAA). The docs recommend it for all production workloads. Trade-off: you pay for reserved capacity whether busy or not.
Third-party → External Model. Otherwise pay-per-token unless you need SLAs or fine-tuned weights.
💡 Tip · fieldStart pay-per-token to learn the shape of the traffic, then right-size provisioned throughput against real usage — not a guess.
04

External Models — third parties under governance

An External Model endpoint proxies a provider. You set a provider, a task route, and a config whose API key reads from a Databricks secret — never in code.

from mlflow.deployments import get_deploy_client
client = get_deploy_client("databricks")
client.create_endpoint(
    name="openai-gpt-4o-proxy",
    config={"served_entities": [{
        "name": "gpt-4o",
        "external_model": {
            "name": "gpt-4o", "provider": "openai", "task": "llm/v1/chat",
            "openai_config": {"openai_api_key": "{{secrets/genai_demo/openai_api_key}}"},
        }}]})

Then query openai-gpt-4o-proxy with the same code as a hosted model. provider accepts openai, anthropic, cohere, amazon-bedrock, google-cloud-vertex-ai, and more; task is llm/v1/chat, llm/v1/completions, or llm/v1/embeddings.

⚠️ GotchaDatabricks governs access (auth, rate limits, logging) — not the provider's model. Confirm the provider's data policy and add guardrails for sensitive payloads.
05

Call it — the minimal versions Hands-on

LangChain — ChatDatabricks (package databricks-langchain):

from databricks_langchain import ChatDatabricks
llm = ChatDatabricks(endpoint="databricks-meta-llama-3-3-70b-instruct", temperature=0)
print(llm.invoke("What is a Foundation Model API?").content)

OpenAI-compatible client (FM APIs speak the OpenAI schema):

from openai import OpenAI
from databricks.sdk import WorkspaceClient
w = WorkspaceClient()
client = OpenAI(api_key=w.tokens.create(lifetime_seconds=600).token_value,
                base_url=f"{w.config.host}/serving-endpoints")
r = client.chat.completions.create(
    model="databricks-meta-llama-3-3-70b-instruct",
    messages=[{"role":"user","content":"Say hello in 5 words."}], temperature=0)
print(r.choices[0].message.content)

SQL — ai_query() for batch/inline inference at scale:

SELECT ai_query(
  'databricks-meta-llama-3-3-70b-instruct',
  'Classify sentiment as positive, negative, or neutral: "Delayed but staff were kind."'
) AS sentiment;

How to verify: the Serving UI shows the endpoint Ready; a chat response has a non-empty choices[0].message.content; with AI Gateway logging on, requests land in the usage/inference tables.

06

Worked example — Unity Airways

  • Week 1 (prototype): databricks-meta-llama-3-3-70b-instruct pay-per-token via ChatDatabricks. No infra, per-token cost.
  • Week 3 (compare): an External Model endpoint for GPT to A/B one hard intent; key in a secret; spend visible in one place.
  • Quarter 2 (production): switch to provisioned throughput for peak-hour guarantees and serve fine-tuned weights on the same surface.
📌 ImportantKeep the endpoint name in config. Moving pay-per-token → provisioned throughput → custom weights is then a one-line change, not a rewrite.
