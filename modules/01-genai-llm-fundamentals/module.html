GenAI and LLM Fundamentals

The concept foundation for everything that follows: what an LLM does, how it differs from the ML you know, and how you reach a model on Databricks. One numbered entry per topic.

01

The big idea

Before you build RAG, evaluate an agent, or ship a Genie space, you need a working model of tokens, context windows, embeddings, the cost/latency/quality trade-off, and how a prompt reaches a model here.

  • Generative AI produces new content from a prompt; traditional ML mostly predicts a label or number.
  • An LLM reads tokens up to a context window and predicts the next token; temperature and sampling control randomness.
  • Embeddings turn text into vectors so meaning becomes distance — the engine of semantic search and RAG.
  • Model choice is a three-way trade-off: cost, latency, quality. Read the model card for risk and bias first.
  • You reach models through Model Serving: Foundation Model APIs (pay-per-token / provisioned throughput) and External Models.

03

Generative AI vs traditional ML 01.1 · Theory

Traditional (predictive) ML answers a fixed question about a row and outputs a label or number. Generative AI learns the shape of language so well it can produce new content one token at a time. A foundation model is pre-trained broadly and adapted with prompts instead of retraining.

Traditional MLGenerative AI
OutputLabel or numberNew content (text, code, embeddings)
Built byTraining a model per taskPrompting one foundation model
Databricks toolMLflow + classic MLModel Serving + Foundation Model APIs
💡 Tip · fieldThey are not rivals: a small classifier can route a request and an LLM can respond — the "classify → route → process" pattern.
04

Tokens, context windows, temperature and sampling 01.2 · Theory

  • Token — the unit an LLM reads and generates (~¾ of an English word). You pay per token, input + output.
  • Context window — the max tokens the model reads in one call (prompt + system + retrieved context + answer). Small (512) is fast/cheap but truncates; large (8k–32k+) reasons more but costs more.
  • Temperature — randomness of the next-token pick. 0 is near-deterministic (extraction, JSON, SQL); higher adds variety.
  • Sampling — top-p (nucleus) and top-k trim the candidate set; max_tokens caps length.
Predict → trim → sample → append → repeat. Source: module.md · 📗 B2 Ch2.
📌 ImportantFor anything a system will parse (JSON, SQL, labels), set temperature = 0. The book's structured-output and agent examples use temperature=0.
05

Embeddings — meaning as distance 01.3 · Theory

An embedding turns text into a vector (often 384–1,536+ dimensions) placed so similar meaning sits close together. Keyword search matches letters; embeddings match meaning — the engine behind semantic search and RAG. Closeness is scored with cosine similarity or dot product.

  • An embedding model encodes; an LLM generates. RAG uses both.
  • On Databricks the safe default is databricks-gte-large-en (also databricks-bge-large-en).
⚠️ GotchaQuery and document embeddings must come from the same model, or identical meaning lands in different spaces and retrieval quietly fails (📗 B2 Ch3, "Embedding mismatch").
06

Model tasks and reading model cards 01.4 · Theory

Pick the task type (see the Task types tab above), then read the model card before you deploy. A model card documents how a model was built and where it fails.

  • Training data sources — bias and domain fit.
  • Intended use cases — using a model off-label is risky.
  • Known limitations / biases — weak non-English, hallucination tendencies.
  • Evaluation by language / domain — does it generalize to your users?
📌 ImportantJustify model choice with documented metadata and model-card limitations, not "it looked good in testing." Never deploy in health/finance/legal without reading the card (📗 B2 Ch2, Table 2-7).
07

Balancing cost, latency, and quality 01.5 · Theory

Model selection is a three-way optimization, not a hunt for the single "best" model. Cost (per token or per compute), latency (time to respond), and quality (accuracy, low hallucination) trade off — and model size is the lever (small ~3B cheap/fast; large 13B+ best but slow/pricey).

High-volume + low-risk → small/fast. Customer-facing/regulated → higher quality + fallback. Source: 📗 B2 Ch2.
💡 Tip · fieldUse caching and batching to cut cost, fallback logic for customer-facing quality, and dynamic routing — a cheap model for easy queries, a strong one for hard ones.
08

Foundation Model APIs and external models 01.6 · ★ Cornerstone · T + H

This is how you actually reach a model. Everything runs through Mosaic AI Model Serving, with three endpoint families. The Foundation Model APIs come in two modes — pay-per-token (start here) and provisioned throughput (production). External Models proxy OpenAI / Anthropic / Google under one governance layer.

One endpoint surface, three families. Source: foundation-model-apis.md · 🌐 Model Serving docs.

Full deep-dive with the hands-on lab: foundation-model-apis.html · notebook 01-module-lab.py.

⚠️ GotchaThe LangChain package is databricks-langchain (from databricks_langchain import ChatDatabricks) — NOT langchain-databricks. And databricks-dbrx-instruct is no longer a listed pay-per-token model; treat any endpoint name as a dated snapshot and verify.

10

Field notes

📌 ImportantDesign your app against "a serving endpoint" and keep the endpoint name in config — then you can move pay-per-token → provisioned throughput → custom weights without a rewrite.
💡 Tip · fieldFoundation Model APIs speak the OpenAI schema, so existing OpenAI-client and many LangChain integrations work by changing only base_url and the model name.
⚠️ GotchaSending text longer than the context window causes silent truncation and wrong answers. Check the model's max context; chunk and retrieve instead.

Notes & self-check

Reveal 5-question self-check
  1. How do generative AI and traditional ML differ in what they output and how they are built?
  2. Your prompt + retrieved docs total 10,000 tokens but the window is 8,000. What happens, and what does it do to quality?
  3. You extract invoice_total as JSON for a downstream system. What temperature, and why?
  4. Why must the document and query embeddings come from the same model?
  5. Name the three Model Serving families and the two Foundation Model API modes. When choose provisioned throughput over pay-per-token?
