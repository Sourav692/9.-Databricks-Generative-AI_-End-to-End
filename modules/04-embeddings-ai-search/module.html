Module 04 — Embeddings & Databricks AI Search · Topics 04.1–04.9 · Theory + Hands-on

Embeddings and Databricks AI Search

A clean Delta table can't answer a question. This module embeds the Unity Airways chunks, builds a Databricks AI Search index over them, and hands a retriever to the Module 05 RAG chain — matching on meaning, not keywords.

01 · The big idea

From rows of text to a retriever

Module 03 left one row per chunk in unity_airways.rag.ua_rag_chunks. AI Search embeds each chunk into a vector, stores it on an endpoint, and at query time returns the chunks nearest the question's vector — so a bot answers "rebook for free?" from a clause that says "involuntary re-accommodation."

  • Similarity is measured by cosine (direction) or dot-product (direction + magnitude); normalize the vectors and the two are identical.
  • The embedding model decides quality. Default to databricks-gte-large-en (1024-dim, 8192-token context) and make sure chunks fit its context length.
  • The workhorse index is a Delta Sync Index with managed embeddings — point it at the table; Change Data Feed keeps it current incrementally.
  • Query with ANN or HYBRID, filter on metadata, add a reranker, then hand a DatabricksVectorSearch(...).as_retriever() from databricks-langchain to Module 05.
📌 NamingThe product is Databricks AI Search (formerly Databricks Vector Search / "Mosaic AI Vector Search"), GA. But the SDK package is still databricks-vectorsearch — from databricks.vector_search.client import VectorSearchClient. There is no databricks-ai-search package and no AISearchClient.
02 · The problem

Why keyword search (and naive vectors) fail

A support agent asks "Can I rebook for free if my connection is cancelled?" The answering chunk says "involuntary re-accommodation" and never uses "rebook" or "free." A LIKE '%rebook%' finds nothing. The system must match on meaning, over hundreds of thousands of chunks, in tens of milliseconds, staying fresh as policy changes.

  • Keyword-only → misses paraphrases and synonyms; great for codes, useless for reworded questions.
  • Any embedding model → a 512-token-context model silently truncates a 700-token chunk. Match context length to chunk size.
  • Re-embed everything nightly → slow and costly. Delta Sync re-embeds only the rows that changed via Change Data Feed.
  • One endpoint for all → Standard = low latency; Storage-optimized = billions of vectors, cheaper, higher latency. Wrong choice = overpay or miss latency budget.
  • Top-k is enough → coarse bi-encoder scores can rank the best chunk 4th; the LLM never sees it. A reranker reorders.

04 · Foundations and build

04.1–04.5 — metric, model, index, filters, retriever

04.1 AI Search fundamentals — cosine vs dot-product Theory

Cosine measures the angle (direction) and ignores length: A=(2,2) and B=(4,4) score 1.0. Dot-product measures direction and magnitude: A·B=16 but A·A=8, so a longer vector can outrank a better match. Normalize to unit length and dot-product(Â, B̂) = cosine(A, B). Cosine is the standard for semantic RAG.

04.2 Choosing an embedding model + context length Theory

  • databricks-gte-large-en — 1024-dim, 8192-token context. Safe default; covers 150–500-token chunks whole.
  • databricks-bge-large-en — 1024-dim, 512-token context — truncates larger chunks.
  • databricks-qwen3-embedding-0-6b — Preview; verify specs before use.
  • Query and index must use the same model — same space, same dimension.

04.3 ★ Creating and querying an index Hands-on Cornerstone

Endpoint → Delta Sync index → sync → query. Full walkthrough in the Create & query an index deep-dive.

# %pip install databricks-vectorsearch   # unchanged despite the "AI Search" rebrand
from databricks.vector_search.client import VectorSearchClient   # NOT AISearchClient

vsc = VectorSearchClient()
vsc.create_endpoint_and_wait(name="unity-airways-vs", endpoint_type="STANDARD")

index = vsc.create_delta_sync_index_and_wait(
    endpoint_name="unity-airways-vs",
    index_name="unity_airways.rag.ua_rag_chunks_index",
    source_table_name="unity_airways.rag.ua_rag_chunks",
    primary_key="chunk_id",
    embedding_source_column="content",
    embedding_model_endpoint_name="databricks-gte-large-en",
    pipeline_type="TRIGGERED",
    columns_to_sync=["chunk_id", "content", "source_doc", "chunk_index"],
)

You can also query an index from SQL with the built-in vector_search() function. A Delta Sync index needs Change Data Feed on the source table (Module 03 already enabled it).

04.4 Metadata filtering on queries Hands-on

Restrict which rows are eligible before ranking. Filter syntax depends on the endpoint type: Standard uses dict/filters_json; Storage-optimized uses a SQL-like filters string. You can only filter on columns you synced.

results = index.similarity_search(
    query_text="checked bag allowance",
    columns=["chunk_id", "content", "source_doc"],
    num_results=5,
    filters="source_doc = 'faq' AND chunk_index < 20",  # storage-optimized syntax
)

04.5 Retrievers and embedding models together Theory + Hands-on

The retriever wraps the index behind one call for the chain. It comes from databricks-langchain (not langchain_community, not langchain-databricks).

from databricks_langchain import DatabricksVectorSearch, DatabricksEmbeddings

vs = DatabricksVectorSearch(
    endpoint="unity-airways-vs",
    index_name="unity_airways.rag.ua_rag_chunks_index",
    columns=["chunk_id", "content", "source_doc"],
)
retriever = vs.as_retriever(search_kwargs={"k": 5})   # hand THIS to Module 05
docs = retriever.invoke("Can I rebook for free if my connection is cancelled?")

Managed-embeddings index → the retriever just passes text. Self-managed / Direct Vector Access → embed the query with DatabricksEmbeddings(endpoint="databricks-gte-large-en") so query and docs share one space.

05 · Query, tune, productionize

04.6–04.9 — cost, types, hybrid, reranking

04.6 Tuning for latency and cost Theory + Hands-on

  • Endpoint type is the biggest lever — Standard (low latency) vs Storage-optimized (cheap, huge, higher latency).
  • num_results — return only what the LLM reads; fetch wide only to feed a reranker.
  • columns_to_sync — sync only needed columns → smaller, faster index.
  • Pipeline type — TRIGGERED (cheap, on-demand) vs CONTINUOUS (live, costlier).

04.7 Index and endpoint types Theory + Hands-on

Index typeEmbeddings / syncStatus
Delta Sync (managed)Databricks computes; auto from CDFGA · default
Delta Sync (self-managed)You provide vectors; auto from CDFGA
Direct Vector AccessYou provide; manual CRUDGA
Full-text searchKeyword only, no vectorsBeta

Endpoints: Standard (low latency, up to hundreds of millions of vectors, GA) vs Storage-optimized (>1B vectors, faster indexing, cheaper, GA). Start with Delta Sync managed + Standard.

04.8 Hybrid search — keyword + vector Theory + Hands-on

ANN matches meaning but can miss exact tokens (booking refs, fare classes, IATA codes). HYBRID runs ANN + BM25 and merges — semantic recall plus exact-term precision. GA. Costs ~2x ANN.

results = index.similarity_search(
    query_text="baggage fee for fare class Q on flight UA-8842",
    columns=["chunk_id", "content", "source_doc"],
    query_type="HYBRID",   # "ANN" (default) or "HYBRID"
    num_results=10,
)

04.9 ★ Reranking retrieved results Theory + Hands-on Cornerstone

Bi-encoder retrieval is fast but coarse; the best chunk can land at rank 4. A cross-encoder reranker reads each (query, chunk) pair together and reorders. Full walkthrough in the Reranking deep-dive.

  • Retrieve a wider candidate set (top 25–50) with ANN/HYBRID.
  • Cross-encoder re-scores each pair — accurate, but too slow for the whole corpus, so run it only on candidates.
  • Keep the top k (e.g. 5). Fixes the MRR problem from 03.7 — pushes the right chunk to rank 1.
  • On Databricks: run the reranker behind a Model Serving endpoint or in the chain. Exact managed-reranking names are live re-check pending — confirm at build time.
06 · Worked thread

Unity Airways, table to retriever

  1. Source (03): unity_airways.rag.ua_rag_chunks — CDF on, columns chunk_id, content, source_doc, chunk_index, ingested_at.
  2. Model (04.2): databricks-gte-large-en — 8192-token context covers ~200-token chunks whole.
  3. Index (04.3, 04.7): Delta Sync managed index ua_rag_chunks_index on a Standard endpoint, key chunk_id, embed content, sync source_doc/chunk_index, TRIGGERED.
  4. Query (04.1, 04.8): "rebook for free?" → ANN finds the "involuntary re-accommodation" chunk despite zero shared keywords; a booking-code query switches to HYBRID.
  5. Filter (04.4): baggage how-to adds filters="source_doc = 'faq'".
  6. Rerank (04.9): retrieve top 30, cross-encoder re-scores, keep top 5 — the exact refund clause moves to rank 1.
  7. Hand off (04.5): as_retriever(search_kwargs={"k": 5}) → the Module 05 RAG chain.
07 · Field notes

What to remember

📌 ImportantPackage is databricks-vectorsearch / VectorSearchClient (never databricks-ai-search / AISearchClient). LangChain integration is databricks-langchain (DatabricksVectorSearch, DatabricksEmbeddings, ChatDatabricks). Start with a Delta Sync managed index on a Standard endpoint. Exam trap: normalized dot-product = cosine similarity.
💡 Tip · fieldRight-size before you scale: Standard endpoint + TRIGGERED sync + k=5 is the cheap, fast default for a modest corpus. Plan your filter/return columns into columns_to_sync up front. Hybrid is the safety net for the "exact terms matter" gap from 03.7.
⚠️ GotchaDon't mix models between build and query (wrong space / dimension mismatch), don't embed a 700-token chunk with a 512-token model, and don't mix filter syntaxes (SQL-string on Standard, or JSON dict on Storage-optimized). Reranking fixes ordering, not recall — a chunk never retrieved can't be reranked in.
