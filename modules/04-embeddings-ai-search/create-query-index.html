Module 04 — Embeddings and AI Search · Topic 04.3 · Theory + Hands-on ★

AI Search Index

Turn a governed chunk table into a queryable vector store: one endpoint, one Delta Sync index with managed embeddings, kept fresh by Change Data Feed, queried three ways. The retriever you build here is the "R" in Module 05.

01 · The big idea

One index, three front doors

Unity Airways has its policy chunks sitting in one Delta table, unity_airways.rag.ua_rag_chunks. A table of text is not searchable by meaning — LIKE '%refund%' misses a chunk that says "money back within 24 hours." An AI Search index stores a vector per chunk and returns the ones nearest to the question, and you query that one index from Python, LangChain, or SQL.

  • You build retrieval in two objects: a Vector Search endpoint (the compute that serves indexes) and an index (the searchable structure over one Delta table). One endpoint, many indexes.
  • The default is a Delta Sync Index with Databricks-managed embeddings: name the content column, and Databricks embeds every chunk with databricks-gte-large-en and keeps the index in sync.
  • The source table needs Change Data Feed so the index can re-embed only what changed. pipeline_type is TRIGGERED (you call sync, cheap) or CONTINUOUS (always fresh, always-on compute).
  • The SDK is still databricks-vectorsearch despite the "AI Search" rename. There is no databricks-ai-search package or AISearchClient.

03 · How it works

Endpoint, index, and why CDF is required

Pick the index type by where the vectors live and how the data updates. Start with managed Delta Sync — it is the cheapest, most governed default.

Managed is the default. Self-managed for custom models; Direct Access for real-time vector writes.
  • Endpoint = the server. It hosts indexes. Build one per environment or team and put several indexes on it. Standard for low-latency apps; Storage-optimized for >1B vectors. Creation is async — wait for ONLINE.
  • Managed embeddings guarantee one model. Databricks embeds the chunks and each query with the same databricks-gte-large-en (1024-dim, 8192-token). Mismatched embed models silently wreck relevance; managed mode removes that risk.
  • Change Data Feed is the wire. A Delta Sync Index reads CDF to find which rows were inserted, updated, or deleted, then re-embeds only those. The source table must have delta.enableChangeDataFeed = true or index creation fails.
  • Readiness matters. On creation the index runs an initial sync (read table, embed all rows, build the ANN structure). Query before it finishes and you get empty or partial results — so wait_until_ready().
  • TRIGGERED does not auto-update. After a batch write to the table you must call index.sync(). CONTINUOUS keeps it fresh on standing compute.
04 · Run this

Endpoint, index, sync, query

Prereqs: serverless or DBR ML runtime, MLflow ≥ 3.1; write access to unity_airways.rag; permission to create a Vector Search endpoint. The Module 03 table ua_rag_chunks exists with CDF on.

%pip install -U databricks-vectorsearch databricks-langchain
dbutils.library.restartPython()
CATALOG = "unity_airways"; SCHEMA = "rag"
SOURCE_TABLE   = f"{CATALOG}.{SCHEMA}.ua_rag_chunks"      # chunk_id, content, source_doc, chunk_index, ingested_at
INDEX_NAME     = f"{CATALOG}.{SCHEMA}.ua_rag_chunks_index"
ENDPOINT_NAME  = "unity-airways-vs"
EMBED_ENDPOINT = "databricks-gte-large-en"               # 1024-dim, 8192-token context

Confirm Change Data Feed (a Delta Sync Index needs it):

spark.sql(f"ALTER TABLE {SOURCE_TABLE} SET TBLPROPERTIES (delta.enableChangeDataFeed = true)")
spark.sql(f"SHOW TBLPROPERTIES {SOURCE_TABLE} (delta.enableChangeDataFeed)").show(truncate=False)

Create the endpoint (Standard). create_endpoint is async; the _and_wait helper blocks until ONLINE:

from databricks.vector_search.client import VectorSearchClient   # package: databricks-vectorsearch

vsc = VectorSearchClient()   # picks up notebook auth automatically
if not vsc.endpoint_exists(ENDPOINT_NAME):
    vsc.create_endpoint_and_wait(name=ENDPOINT_NAME, endpoint_type="STANDARD")

Create the Delta Sync Index with managed embeddings — name the text column and the model; Databricks embeds and syncs:

index = vsc.create_delta_sync_index_and_wait(
    endpoint_name=ENDPOINT_NAME,
    index_name=INDEX_NAME,
    source_table_name=SOURCE_TABLE,
    primary_key="chunk_id",                       # id used to add/replace/delete vectors
    embedding_source_column="content",            # THIS column gets embedded
    embedding_model_endpoint_name=EMBED_ENDPOINT, # managed: Databricks calls gte-large-en
    pipeline_type="TRIGGERED",                    # you call .sync(); cheaper than CONTINUOUS
)

Verify online, then re-sync after table changes:

index = vsc.get_index(endpoint_name=ENDPOINT_NAME, index_name=INDEX_NAME)
index.wait_until_ready()             # blocks until ONLINE
print(index.describe()["status"])    # inspect detailed status
# ... after a nightly re-chunk writes new rows:
index.sync()                         # reads CDF, embeds only new/changed chunks

Query A — Python SDK:

results = index.similarity_search(
    query_text="Can I get a refund on a Basic Economy fare?",
    columns=["chunk_id", "source_doc", "content"],   # only requested columns come back
    num_results=5,
)
for row in results["result"]["data_array"]:
    # row = [chunk_id, source_doc, content, score] — score is always last
    print(round(row[-1], 3), "|", row[1], "|", row[2][:120], "...")

Query B — LangChain retriever (the object handed to Module 05):

from databricks_langchain import DatabricksVectorSearch

vector_store = DatabricksVectorSearch(     # managed index: embed model auto-detected
    endpoint=ENDPOINT_NAME,
    index_name=INDEX_NAME,
    columns=["chunk_id", "source_doc", "content"],
)
retriever = vector_store.as_retriever(search_kwargs={"k": 4})
docs = retriever.invoke("Can I change my Basic Economy booking?")

Query C — SQL vector_search() (any SQL surface):

SELECT *
FROM vector_search(
  index => 'unity_airways.rag.ua_rag_chunks_index',
  query_text => 'What is the checked baggage allowance on Basic Economy?',
  num_results => 5
);
-- Returns synced columns plus a similarity score. DBR 15.2 and below use `query =>`.

How to verify retrieval is good (not just working): read the top chunk. If it holds the whole answer — rule plus its condition — your chunking and index are healthy. If the top chunks are near-duplicates or off-topic, fix chunk size/overlap upstream. The index faithfully returns whatever you indexed.

05 · Field notes

What to remember

📌 ImportantAn AI Search index is only as good as the chunks under it. It reliably returns the nearest vectors, so a bad answer is almost always upstream — chunking (Module 03) or the wrong index/columns — not the search. Debug retrieval by reading the top chunk before you touch the model.
💡 Tip · fieldBuild the endpoint once per environment and reuse it. Start every index as managed embeddings, TRIGGERED — the cheapest, most governed default — and move to self-managed or CONTINUOUS only when a concrete requirement forces it. Always put source_doc and chunk_id in your query columns so every retrieved chunk is citable.
⚠️ GotchaThe Study Guide (B2 Ch9) teaches a simplified, conceptual client — client.create_index(index_type="managed", delta_table_name=…) and client.query(index_name=…, k=5) — and admits "exact client APIs may vary." The real SDK is vsc.create_delta_sync_index(source_table_name=…, embedding_source_column=…, embedding_model_endpoint_name=…, pipeline_type=…) and index.similarity_search(query_text=…, columns=[…], num_results=…). The book's k= is num_results=, and you pick the type by calling create_delta_sync_index vs create_direct_access_index — there is no index_type="managed" string.
