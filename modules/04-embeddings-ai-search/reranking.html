Reranking Retrieved Results

Your embedding retriever is tuned for recall, so the best passage is often not rank 1. Two-stage retrieval fixes the order: cast a wide net with the bi-encoder, then let a cross-encoder re-score the candidates and promote the passage that actually answers the question.

01 · The big idea

The right passage came back at rank 7

Unity Airways' assistant answers "Can I get a refund on a Basic Economy fare?" from your indexed policy chunks. It returns the top 5 by similarity and answers from the general refund policy. It is wrong: the Basic Economy exception was retrieved, but at rank 7, so it never reached the prompt. The retrieval was fine. The ordering failed.

  • The embedding model is a bi-encoder: it encodes the query and each passage separately, then compares vectors. That is what makes the index fast — and what makes its ranking a coarse proxy for real relevance.
  • Reranking is a postprocessing step: re-score the top-K candidates with a model that reads the query and passage together, then reorder so the best rises to the top.
  • The shape is two-stage retrieval: stage 1 = fast bi-encoder recall (top-K, K=50); stage 2 = a cross-encoder re-scores each (query, passage) pair and returns top-N (N=5).
  • Retriever owns recall, reranker owns precision, and you only pay the reranker's cost on a small candidate set — never on the whole corpus.

03 · How it works

Why the best passage is often not rank 1

Separate encoding is what makes the index fast, and also what makes its ranking approximate. The cross-encoder gives up precomputation to gain accuracy, which is only affordable on the top-K.

Two vectors compared apart vs one pass that reads query and passage together.
  • Stage 1 tunes recall, not order. Raise K so the good passage is reliably in the pool. Messy ordering is fine — stage 2 fixes it. If the passage never shows up even at K=100, that is a chunking, embedding, or hybrid-search problem, not a reranking one.
  • Stage 2 scores pairs. For each of the K candidates, build a (query, passage) pair, run one forward pass, get a relevance score, sort, keep top-N.
  • Why only top-K. One forward pass per pair means cost scales with candidate count. K=50 pairs is cheap; millions is impossible. Reranking is affordable because stage 1 already narrowed the field.
  • Cross-encoder vs LLM-as-reranker. A small ms-marco-style cross-encoder is the practical default: fast, cheap, no key. LLM-based scoring (prompt an LLM to rate each passage) handles subtler intent but costs more per candidate — reserve it for the hardest queries.
  • Hybrid search and reranking stack. Hybrid (04.8) widens the candidate net for recall; reranking (04.9) sharpens the order for precision. Different halves of the same problem.
04 · Run this

Retrieve top-K, rerank, feed the chain

Prereqs: serverless or DBR ML runtime; read access to the Module 04 Delta Sync index over unity_airways.rag.ua_rag_chunks; a vector search endpoint. The retriever uses DatabricksVectorSearch from databricks-langchain; the cross-encoder is a portable, provider-neutral example served in-process.

%pip install -U databricks-vectorsearch databricks-langchain sentence-transformers
dbutils.library.restartPython()
CATALOG = "unity_airways"; SCHEMA = "rag"
VS_ENDPOINT = "unity-airways-vs"
INDEX = f"{CATALOG}.{SCHEMA}.ua_rag_chunks_index"   # Module 04 index over ua_rag_chunks
K = 50   # stage-1 recall depth
N = 5    # stage-2 passages sent to the LLM
QUERY = "Can I get a refund on a Basic Economy fare?"

Stage 1 — bi-encoder recall via the Module 04 retriever. Tune k for recall, not order:

from databricks_langchain import DatabricksVectorSearch

vs = DatabricksVectorSearch(
    endpoint=VS_ENDPOINT,
    index_name=INDEX,
    columns=["chunk_id", "source_doc", "content"],
)
retriever = vs.as_retriever(search_kwargs={"k": K})   # top-K candidates
candidates = retriever.invoke(QUERY)                  # K LangChain Documents

Stage 2 — cross-encoder rerank (the portable pattern). Score each (query, passage) pair, sort, keep top-N:

from sentence_transformers import CrossEncoder

# Small, well-known open cross-encoder — illustrative. Runs in-process; no external key.
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

pairs  = [(QUERY, d.page_content) for d in candidates]
scores = reranker.predict(pairs)                       # one score per pair
ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
top_n  = [doc for doc, _ in ranked[:N]]                # what the LLM will see

Feed the reranked top-N into the Module 05 RAG chain — wrap both stages in one function and plug it into an LCEL chain:

from databricks_langchain import ChatDatabricks
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableLambda, RunnablePassthrough

def retrieve_and_rerank(question: str):
    cands = retriever.invoke(question)                                  # stage 1
    scores = reranker.predict([(question, d.page_content) for d in cands])
    ranked = sorted(zip(cands, scores), key=lambda x: x[1], reverse=True)
    return [d for d, _ in ranked[:N]]                                   # stage 2

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer using ONLY the context. If unsure, say you don't know.\n\nContext:\n{context}"),
    ("user", "{question}"),
])
llm = ChatDatabricks(endpoint="databricks-claude-sonnet-4-5")   # confirm on supported-models page

chain = (
    {"context": RunnableLambda(retrieve_and_rerank) | format_docs,
     "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
answer = chain.invoke(QUERY)

How to verify it worked: print the naive top-N (candidates[:N]) beside the reranked top-N (ranked[:N]). Success looks like the passage containing the whole Basic Economy refund rule appearing in the reranked list and near the top, even if it was rank 6-7 before. Then measure MRR on a small labeled query set — that is the metric reranking moves most.

05 · Field notes

What to remember

📌 ImportantReranking improves ordering, not coverage. It can only reorder the passages stage 1 already retrieved — if the answer isn't in the top-K, no reranker can put it in the prompt. Tune the retriever for recall first (raise K, fix chunking/embeddings, consider hybrid search), then rerank for precision.
💡 Tip · fieldStart with K=50, N=5, then tune by measuring MRR on a small labeled set. Reranking improves precision significantly but adds latency, so apply it selectively on a small candidate set (the book uses the top ~100). If latency matters, shrink K or move the cross-encoder onto a warm Model Serving endpoint.
⚠️ Gotcha · verify the native optionDatabricks' retrieval stack changes fast. This lesson teaches the portable cross-encoder pattern (retrieve top-K, score (query, passage) pairs, keep top-N) because it works on any stack and matches B2 Ch3. Before production, check the current docs for a native option — whether AI Search exposes a built-in rerank parameter or Databricks hosts a managed reranker endpoint — and prefer it if governed. Do not assume a databricks-*-reranker endpoint name; confirm it. (Live doc re-check pending: authoring-time fetch of the AI Search query docs returned no content.)
⚠️ Gotcha · namingThe product is Databricks AI Search (formerly Vector Search), but the SDK is still databricks-vectorsearch — there is no databricks-ai-search package. The retriever and chat model come from databricks-langchain (from databricks_langchain import DatabricksVectorSearch, ChatDatabricks), not langchain-databricks or langchain_community. Test yourself

Reveal 5-question self-check
  1. Why does a bi-encoder retriever optimize recall but rank imperfectly, and why is that acceptable as stage 1?
  2. What does a cross-encoder do differently from the embedding model, and why can it only run on the top-K rather than the whole index?
  3. You retrieve K=50 and send N=5 to the LLM. What is each number doing, and what breaks if you set K=N=5?
