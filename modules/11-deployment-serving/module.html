Module 11 — Deployment and serving · Topics 11.1–11.13 · Theory + Hands-on

Deployment and Serving

Taking the Unity Airways support agent from "works in my notebook" to a governed production endpoint — packaged with MLflow, served on Model Serving, metered by AI Gateway, run at scale with AI Functions, and promoted safely with Champion vs Challenger.

01 · The big idea

A deployed model is a governed asset, not a script you curl

Module 09 built the engine and Module 10 built the dashboard. Module 11 is the whole delivery system — the assembly line that packages the car, the road it drives on (Model Serving), the tollgate and speed limits (AI Gateway), the test-drive desk (Review App), and the fleet-scheduling office (batch ai_query + Lakeflow Jobs).

  • Package — a ResponsesAgent/PyFunc logged with MLflow 3.1 and registered to Unity Catalog as unity_airways.rag.ua_support_agent (11.4, 11.13).
  • Serve — one of three Model Serving families (custom / foundation / external), created for you by agents.deploy() (11.1 ★, 11.12).
  • Govern the edge — AI Gateway adds rate limits, guardrails, fallbacks, and payload logging with no code change (11.3 ★).
  • Feedback + scale + operate — the Review App (11.2), batch ai_query and AI Functions (11.5, 11.10 ★), Lakeflow Jobs (11.11), and an LLMOps promotion loop (11.6–11.9).
📌 The rule that shapes this moduleEvery deployment follows the same stages, each with a managed Databricks surface: package (MLflow + UC) → serve (Model Serving) → govern (AI Gateway) → clients (App / Review App / ai_query / REST) → log (inference tables) → monitor (Module 13) → promote the next Champion. Name maturity as you go: GA — Model Serving, AI Gateway, Review App, AI Functions, Lakeflow Jobs; Beta — Unity AI Gateway; Preview — AI Gateway PII guardrails.
02 · The problem

Why "it worked in the cell" is not production

A real deployment has to survive things a notebook never sees — and it has to run in two very different modes.

  • Share the notebook or a curl → no version pin, no access control, no rate limit, no logging. The first spike or prompt-injection attempt is an incident you cannot trace to a version.
  • Hard-code the provider key → keys leak and cannot be governed; external providers belong behind an external-model endpoint with the key in a Databricks secret (11.12).
  • Deploy over the old version → no rollback, no A/B, no eval gate. Champion-vs-Challenger exists so a bad version never becomes a bad afternoon (11.8).
  • Loop the real-time endpoint for the nightly batch → slow and expensive. Batch is one ai_query over a table, scheduled by a Lakeflow Job (11.5, 11.11).
  • Ship the endpoint and stop → with no Review App and no monitoring you learn it is wrong only when a customer does.

11.1 ★ — Model Serving endpoints for GenAI

Full deep-dive — the three endpoint families, agents.deploy() vs a raw endpoint, provisioned throughput vs scale-to-zero, payloads, and validation — in the Model Serving deep-dive (model-serving.md / model-serving.html).

11.1 Model Serving Theory Hands-on Cornerstone GA

The home for every GenAI endpoint — three families, all GA:

  • Custom models — your MLflow-packaged ua_support_agent (or an open-source model).
  • Foundation models — Databricks-hosted (e.g. databricks-claude-sonnet-4-5), pay-per-token or provisioned throughput.
  • External models — a governed proxy to OpenAI / Anthropic / Google.
# the easy button for an agent
from databricks import agents
agents.deploy("unity_airways.rag.ua_support_agent", version)
# -> Model Serving endpoint + Review App + feedback model
#    + tracing + inference tables + monitoring, in one call
💡 TipUse scale-to-zero for dev and spiky traffic; reach for provisioned throughput only when production latency/throughput needs a guarantee. Served foundation-model names churn — confirm on the supported-models page at authoring time.
05 · Feedback

11.2 — The Review App for stakeholder feedback

11.2 Review App Hands-on GA

  • What it is: a chat UI agents.deploy() creates via the feedback model, so test users and domain experts try the Unity Airways agent and leave 👍/👎 + comments — no notebook.
  • Where feedback lands: responses and reviewer labels are captured as MLflow traces / labels, ready to seed the Module 08 eval set and the Module 13 improve loop.
  • Structured labeling: mlflow.genai.labeling (create_labeling_session, get_review_app) organizes a formal expert review against a dataset.
📌 ImportantThe Review App is the cheapest way to close the loop with stakeholders before a formal eval — the feedback you collect here becomes labeled evaluation data, not throwaway comments.
06 · Cornerstone

11.3 ★ — AI Gateway: guardrails, rate limits, fallbacks, logging

Full deep-dive — enabling each feature on an endpoint, guardrail/PII config, provider fallbacks, usage system tables, payload logging, and Unity AI Gateway budgets — in the AI Gateway deep-dive (ai-gateway.md / ai-gateway.html).

11.3 AI Gateway Theory Hands-on Cornerstone GA

The control plane in front of a serving endpoint — added with no change to agent code:

  • The governed agent pattern: put rate limiting (per user / endpoint), AI guardrails (safety filtering + PII detection/redaction, Preview), provider fallbacks, and usage tracking (system tables) on the FM / external endpoint the agent calls (ua-support-llm); enable payload logging → inference tables on the agent endpoint (ua-support-agent), which supports only inference tables via AI Gateway. No agent code changes. Providers include OpenAI, Anthropic, Google, and others.
  • Unity AI Gateway Beta — the recommended go-forward: richer observability, MCP-service governance, and budget management (spend thresholds / hard cost caps).
💡 Tip · fieldThis is the answer to "keep it from leaking PII or blowing the budget." Turn on rate limits + payload logging the moment the endpoint exists — they cost nothing and save you during the first spike.
07 · Package the model

11.4 + 11.5 — PyFunc structure, and batch ai_query

11.4 PyFunc model structure; pre/post-processing Hands-on

  • mlflow.pyfunc.PythonModel with predict() (and optional load_context()) packages any Python logic as a servable model.
  • Pre-process the request (normalize the question, inject context) and post-process the output (extract JSON, strip unsafe content) inside the PyFunc.
  • Prefer Models-from-Code (mlflow.models.set_model()) over pickling; a ResponsesAgent is itself a PyFunc-compatible model.

11.5 Batch inference with ai_query Hands-on GA

  • ai_query('<endpoint>', request => <column>) calls a served model for every row of a table — parallelized by Spark / serverless SQL.
  • Unity Airways: one query drafts a category, sentiment, and one-line summary for last night's tickets. Request JSON/structured output for clean typed columns.
  • Wrap it in a Lakeflow Job (11.11); same engine as the wider AI Functions family (11.10 ★).
⚠️ GotchaDo not loop the real-time endpoint row-by-row for batch work — it is slow and expensive. Batch is a single ai_query.
08 · Access + identity

11.6 + 11.7 — Access/version control, and endpoint auth

11.6 Access and version control for endpoints Theory Hands-on

  • Versioning: UC holds model versions and aliases (@champion / @challenger); the endpoint pins a version, so what is live is explicit and auditable.
  • Access control: endpoint permissions (CAN_QUERY / CAN_MANAGE) plus UC grants on the model, functions, and data — least-privilege, granted to users / groups / service principals.
  • Roll forward / back: change the served-model version and traffic split, or flip the UC alias — no code redeploy.

11.7 Authentication for agent endpoints Theory

  • Who is calling: query-time auth is a PAT, OAuth (M2M service principal), or on-behalf-of-user.
  • What the agent uses downstream: agents.deploy() sets up automatic authentication passthrough so the agent's declared resources (AI Search index, UC functions, a foundation model) get scoped credentials — no hand-managed tokens inside the agent.
09 · Operate

11.8 + 11.9 — LLMOps/AgentOps rollout, and Databricks Apps auth

11.8 LLMOps and AgentOps: environments, Champion vs Challenger, rollout Theory

  • Environments: promote a version through dev → staging → prod, tracked by UC aliases rather than copying artifacts.
  • Champion vs Challenger: run the new candidate against current prod on the same eval set (or shadow/split traffic); promote only if quality/latency/cost win.
  • Rollout: canary the challenger on a small traffic percentage, watch monitoring, flip @champion to 100%. Rollback is keeping the previous Champion.

11.9 Databricks Apps authentication and authorization for GenAI Theory

  • A Databricks App (Module 10.5) runs as its own service principal — never the developer's token — and reaches the endpoint through a scoped resource.
  • OAuth scopes limit what the app can do; on-behalf-of-user lets it act with the signed-in user's permissions when the answer must respect that user's data access.
⚠️ GotchaNever overwrite the production version in place — you lose the rollback path. And do not assume the app uses your personal token; it uses its service principal with scoped resources.
10 · Cornerstone

11.10 ★ — AI Functions for GenAI at scale

Full deep-dive — each function's signature, structured/JSON extraction with ai_query, vector_search in SQL, cost/scale patterns, and when to reach for which — in the AI Functions at scale deep-dive (ai-functions-at-scale.md / ai-functions-at-scale.html).

11.10 AI Functions at scale Theory Hands-on Cornerstone GA

GenAI in SQL at Spark scale, governed by Unity Catalog — the family (all GA):

  • ai_query (general — any served model + prompt), ai_classify, ai_analyze_sentiment, ai_extract, ai_gen, ai_mask, ai_summarize, ai_translate, ai_fix_grammar, and vector_search (query an AI Search index from SQL).
  • Unity Airways: ai_classify to route tickets, ai_analyze_sentiment to flag angry passengers, ai_mask to redact PII before analytics, ai_summarize for queue dispositions.
💡 TipAnalysts and pipelines get LLM power without writing agent code — this is the scaled counterpart to the real-time endpoint.
11 · Providers, scheduling, OSS

11.11 + 11.12 + 11.13 — Lakeflow Jobs, external creds, Hugging Face models

11.11 Orchestrating and scheduling with Lakeflow Jobs Hands-on GA

  • Lakeflow Jobs (formerly Databricks Workflows) schedules the multi-task pipeline: ingest → ai_parse_document/chunk → embed → refresh the AI Search index → batch ai_query → mlflow.genai.evaluate.
  • Task dependencies (DAG), triggers (cron / file-arrival), retries, serverless compute; system-table schema moved workflow → lakeflow.

11.12 External model credentials and provider setup Hands-on

  • Put the provider API key (Claude/OpenAI) in a Databricks secret — never plaintext — and reference it from an external-model serving endpoint.
  • Callers use the endpoint name, so swapping providers is a config change; the endpoint sits behind AI Gateway (11.3) and UC access control.

11.13 Deploying open-source / Hugging Face models Hands-on

  • Log a Hugging Face model with the transformers flavor (mlflow.transformers.log_model) — or wrap a pipeline in a custom PyFunc — then register to UC and serve as a custom model endpoint on GPU (B2 Ch5, Example 7-7).
  • For a specialized model (small classifier, domain embedder, fine-tune) not on the Foundation Model APIs list.
⚠️ Gotcha · verify liveRebrands: it is AI Gateway / Unity AI Gateway (not "Mosaic AI Gateway") and Lakeflow Jobs (not "Workflows"); the Vector Search SDK is still databricks-vectorsearch despite the "AI Search" rename. 12 · Worked thread

Unity Airways — deploy to operate, end to end

  1. Package (11.4 / 11.13): log ua_support_agent with Models-from-Code, register to unity_airways.rag.ua_support_agent; log a small OSS ticket-classifier with the transformers flavor.
  2. Serve (11.1 ★): agents.deploy(...) provisions the endpoint, Review App, feedback model + tracing/inference tables/monitoring.
  3. Govern the edge (11.3 ★): enable AI Gateway — rate limits, PII guardrails (Preview), a provider fallback, and usage tracking on ua-support-llm (the endpoint the agent calls); payload logging on the agent endpoint ua-support-agent.
  4. External provider (11.12): the OpenAI key lives in a Databricks secret behind an external-model endpoint, so a GPT fallback is a config change.
  5. Feedback (11.2): support leads use the Review App; labels flow into the Module 08 eval set.
  6. Batch + scale (11.5 / 11.10 ★): a nightly ai_query scores each ticket (category / sentiment / summary); ai_mask redacts PII first.
  7. Schedule (11.11): a Lakeflow Job runs index refresh → batch ai_query → mlflow.genai.evaluate on cron with retries.
  8. Operate (11.6–11.9): endpoint ACLs + UC aliases control access/versioning; the App calls the endpoint as its service principal; a new version ships as a Challenger on 10% canary and is promoted to Champion only after it wins the eval.
13 · Field notes

What to remember

📌 ImportantA deployed model is a governed asset: package (MLflow + UC) → serve (Model Serving) → govern (AI Gateway) → log (inference tables) → monitor (Module 13) → promote. agents.deploy() gives you the endpoint, Review App, feedback model, and tracing/inference-tables/monitoring in one call. Promotion is a traffic shift + alias flip gated on eval.
💡 Tip · fieldTurn on AI Gateway the moment the endpoint exists. Use provisioned throughput only when latency/throughput demands it; scale-to-zero otherwise. Reach for ai_query for batch first. Keep the external-model endpoint name stable so provider swaps are config, not code.
⚠️ GotchaUnity AI Gateway is Beta and PII guardrails are Preview — label maturity, verify enrollment. Served foundation-model names churn (DBRX treated as retired). Never overwrite prod in place. Do not hard-code provider keys — use a Databricks secret + external-model endpoint.
