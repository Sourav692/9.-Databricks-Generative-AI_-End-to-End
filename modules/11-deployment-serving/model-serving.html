Module 11 — Deployment and Serving · Topic 11.1 · Theory + Hands-on ★

Model Serving

A registered agent is a stored artifact, not a service. Model Serving turns unity_airways.rag.ua_support_agent into an autoscaling, governed HTTPS endpoint — one agents.deploy() call that also stands up the Review App, feedback model, tracing, and monitoring. Three endpoint families, one query interface.

01 · The big idea

One governed surface, three sources of model

Your Module 09 agent is registered in Unity Catalog but nothing can call it yet. Model Serving is the managed, serverless service that runs the container, autoscales it, secures it behind workspace auth, and meters usage — so any app, phone system, or batch job hits the same endpoint.

  • Three endpoint families, all GA. Custom models (your MLflow-packaged RAG chain / ResponsesAgent), Foundation Models (Databricks-hosted LLMs), and External Models (a governed proxy to OpenAI / Anthropic / Google).
  • For an agent, one command. from databricks import agents; agents.deploy(uc_model_name, version) creates the endpoint and the Review App and a feedback model, and enables tracing, inference tables, and monitoring.
  • Config is small but load-bearing: workload_size, scale_to_zero_enabled, workload_type (CPU vs GPU), environment vars / secrets, and which served-entity version takes traffic.
  • Same query interface everywhere: REST POST /serving-endpoints/<name>/invocations, the SDK, or SQL ai_query for batch. Agents add stream=True, return_trace, and custom_inputs.
  • Governance carries over. Unity Catalog permissions, audit logs, and lineage follow the model from notebook to endpoint.

03 · How it works

Pick the family, right-size the endpoint

Model selection and endpoint choice is a decision tree, not a default. Serving your own logic is a custom endpoint; the LLM your agent calls is a Foundation Model endpoint; a mandated outside provider goes behind an External Model endpoint.

Agent Framework agents take the agents.deploy() path (endpoint + Review App + monitoring). Non-agent models use the SDK / MLflow deployments / UI.
  • Custom models package your logic — the Unity Airways ResponsesAgent shows up with Task = Agent (Responses). You own its dependencies and behavior.
  • Foundation Models are the shared LLM utility. Two modes: pay-per-token (zero setup, billed per token, great to start and for bursty load) and provisioned throughput (reserved tokens/second, SLA latency, required for fine-tuned / custom weights).
  • External Models exist for governance: one Databricks endpoint fronts OpenAI/Anthropic so keys, spend, and rate limits are centralized (pairs with AI Gateway — 11.3).
  • Right-size the endpoint. scale_to_zero_enabled trades idle cost for cold-start latency; workload_size sets the concurrency band; workload_type is CPU for agents, GPU only for in-process heavy inference.
  • Versions + traffic %. One endpoint can hold several served-entity versions and split traffic — the basis for canary rollout (11.6).
04 · Run this

Deploy the agent, then query it

Prereqs: the agent registered in Unity Catalog (Module 09), mlflow>=3.1, databricks-agents, and rights to create a serving endpoint. Deploy the version you validated in Module 08.

Path A — agents.deploy() (recommended for agents):

from databricks import agents

CATALOG, SCHEMA = "unity_airways", "rag"
UC_MODEL = f"{CATALOG}.{SCHEMA}.ua_support_agent"   # the ResponsesAgent from Module 09
VERSION  = 7                                        # the version you evaluated in Module 08

deployment = agents.deploy(
    model_name=UC_MODEL,
    model_version=VERSION,
    scale_to_zero=True,                 # dev: cheap idling. Prod latency-critical: False.
    environment_vars={                  # config + secret references, never plaintext keys
        "APP_ENV": "dev",
        "API_TOKEN": "{{secrets/ua_scope/api_token}}",
    },
    deploy_feedback_model=True,         # also stands up the Review App feedback loop
)

# Read the REAL endpoint name + Review App URL from the output — do not guess them.
print(deployment.endpoint_name)        # e.g. agents_unity_airways-rag-ua_support_agent
print(deployment.review_app_url)

Path B — SDK, for any (non-agent) UC model:

from databricks.sdk import WorkspaceClient
from databricks.sdk.service.serving import EndpointCoreConfigInput, ServedEntityInput
# NOTE: confirm class names against your installed databricks-sdk version.

w = WorkspaceClient()
w.serving_endpoints.create_and_wait(
    name="ua-support-agent",
    config=EndpointCoreConfigInput(served_entities=[
        ServedEntityInput(
            entity_name="unity_airways.rag.ua_support_agent",
            entity_version="7",
            workload_size="Small",         # concurrency band
            workload_type="CPU",           # agent orchestration -> CPU
            scale_to_zero_enabled=True,
        )
    ]),
)

Invoke it over REST (ResponsesAgent uses an input array):

import requests
from databricks.sdk import WorkspaceClient

w = WorkspaceClient()
ENDPOINT = "agents_unity_airways-rag-ua_support_agent"   # from the deploy output

resp = requests.post(
    f"{w.config.host}/serving-endpoints/{ENDPOINT}/invocations",
    headers={"Authorization": f"Bearer {w.config.token}",
             "Content-Type": "application/json"},
    json={
        "input": [{"role": "user",
                   "content": "My flight is at 4pm — what's the check-in cutoff?"}],
        "databricks_options": {"return_trace": True},    # get the MLflow trace back too
    },
    timeout=120,
)
resp.raise_for_status()
print(resp.json())

Batch inference over a Delta table with ai_query:

SELECT
  ticket_id,
  ai_query(
    'agents_unity_airways-rag-ua_support_agent',
    request => to_json(named_struct('input',
      array(named_struct('role','user','content', question_text))))
  ) AS agent_reply
FROM unity_airways.rag.overnight_tickets;

Verify: the endpoint reads Ready on the Serving page (Task = Agent (Responses)); the REST call returns 200 with an assistant output array; the ai_query job writes one reply per row; every call lands in the endpoint's inference table and in MLflow traces (Module 07).

05 · Field notes

What to remember

📌 ImportantModel Serving has three GA families — Custom models, Foundation Models (pay-per-token vs provisioned throughput), and External Models — behind one query interface and one governance model. For an agent, agents.deploy(uc_model_name, version) is the single call that also creates the Review App + feedback model and enables tracing, inference tables, and monitoring. The custom agent endpoint runs on CPU; the GPU-heavy LLM work sits on a separate Foundation Model endpoint.
💡 Tip · fieldStart pay-per-token; move to provisioned throughput only when traffic is steady/high, you need SLA latency, or you serve custom weights. Keep scale_to_zero=True in dev; turn it off for the customer-facing path so cold starts don't read as "the model is slow." Add "databricks_options": {"return_trace": true} while integrating to see the agent's reasoning next to its answer, and right-size workload_size from the inference table, not a guess.
⚠️ GotchaThe endpoint name from agents.deploy() is auto-generated (pattern agents_<catalog>-<schema>-<model>) — read it from the deploy output. The deploy-agent doc page is JS-rendered, so the exact parameter list and name pattern here are grounded in B1 Ch8 + the naming cheat-sheet — treat as live re-check pending. The book's "Mosaic AI Agent Framework" is now moving to plain "Agent Framework" branding (API unchanged). Foundation-model endpoint names churn (DBRX retired) — confirm on the supported-models page; this lesson uses databricks-claude-sonnet-4-5 per the project's shared conventions.
