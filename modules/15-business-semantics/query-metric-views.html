Querying Metric Views

Two rules — name the dimensions, wrap the measures in MEASURE(). Then pull any slice, including dimensions from joined tables you never touch, with no JOIN in your SELECT.

01 · The big idea

One grammar, any slice

You built one governed metric view. Now every team wants a different cut — revenue by fare class, cancellation rate by route, attach rate by tier, a monthly trend. The skill is querying: a small, consistent grammar that pulls any slice from the one definition, including dimensions that live in joined tables.

  • Two rules: name the dimensions; wrap every measure in MEASURE(). SELECT * is unsupported.
  • GROUP BY ALL groups by the dimensions you selected; order by a measure through its alias.
  • Joins are invisible — ask for Region or Loyalty Tier; the YAML resolves the star/snowflake path.
  • Filter dimensions in WHERE; to filter on a measure, wrap the aggregate query and filter the alias.
📌 ImportantSELECT * and bare measure names fail because a measure has no value until you pick a grain. This is the rule that surprises everyone once.
02 · Why naive fails

The traps

  • Query like a table with SELECT * → rejected; measures need a grain.
  • JOIN the dim tables in your SELECT to get Region → redundant and wrong; the join is already in the YAML.
  • Average per-route cancellation rates to get a region rate → the classic ratio bug; averaging ratios is not the ratio of totals.
  • Filter a measure in WHERE → WHERE MEASURE(...) > x is not allowed; filter measures in an outer query on the alias.

04 · Walkthrough 1

The simplest query

Revenue + volume by fare class Hands-on

SELECT
  `Fare Class`,
  MEASURE(`Total Revenue`) AS total_revenue,
  MEASURE(`Booking Count`)  AS bookings
FROM unity_airways.analytics.bookings_metrics
GROUP BY ALL
ORDER BY total_revenue DESC

GROUP BY ALL groups by Fare Class (the sole non-measure column). Verify: four rows; the summed total_revenue equals the grand total from the same query with no dimension — proof re-aggregation works.

05 · Walkthrough 2

Time dimensions

Monthly trend + "last quarter by fare class" Hands-on

SELECT
  `Booking Month`,
  MEASURE(`Total Revenue`) AS revenue,
  MEASURE(`Booking Count`) AS bookings
FROM unity_airways.analytics.bookings_metrics
WHERE `Booking Month` >= DATE'2025-01-01'
GROUP BY ALL
ORDER BY `Booking Month`

-- "Total revenue last quarter by fare class" (a Module 14 Genie question)
SELECT `Fare Class`, MEASURE(`Total Revenue`) AS revenue
FROM unity_airways.analytics.bookings_metrics
WHERE `Booking Quarter` = DATE'2025-04-01'
GROUP BY ALL
ORDER BY revenue DESC
💡 TipExpose common grains (Booking Month, Booking Quarter) as dimensions in the YAML so consumers never hand-roll DATE_TRUNC — and everyone buckets time the same way.
06 · Walkthrough 3 · the cornerstone

The joins tutorial

Dimensions from joined tables — no JOIN written Hands-on Cornerstone

Loyalty Tier comes from dim_customers; Region comes from dim_airports reached through dim_flights (a snowflake join). Both are just dimension names to the query:

SELECT
  `Region`,
  `Loyalty Tier`,
  MEASURE(`Total Revenue`)      AS revenue,
  MEASURE(`Cancellation Rate`)  AS cancel_rate
FROM unity_airways.analytics.bookings_metrics
GROUP BY ALL
ORDER BY revenue DESC

"Which routes have the highest cancellation rate" — group by origin + destination, and filter on a measure via an outer query:

SELECT * FROM (
  SELECT `Origin Airport`, `Dest Airport`,
         MEASURE(`Cancellation Rate`) AS cancel_rate,
         MEASURE(`Booking Count`)     AS bookings
  FROM unity_airways.analytics.bookings_metrics
  GROUP BY ALL
)
WHERE bookings >= 5
ORDER BY cancel_rate DESC
LIMIT 10
📌 ImportantVerify the join resolved: every row has a non-null Region / Loyalty Tier, and cancel_rate is between 0 and 1. Null Region ⇒ the airport snowflake key is off; an error on Region ⇒ you're likely below DBR 17.1 (nested joins) / 17.2 (v1.1).
07 · Walkthrough 4

Consuming results — BI, alerts, Genie

One definition, every surface Theory Hands-on

  • SQL Editor / notebooks: run the queries directly; spark.sql("...") returns a DataFrame to display().
  • AI/BI dashboards: add the metric view as a dataset; the builder emits MEASURE() SQL. Dimensions become filters/axes, measures the plotted values.
  • Alerts: a threshold on a measure (e.g. weekly Cancellation Rate for a region > 5%) — the same SQL on a schedule.
  • Genie Agent (Module 14): add the metric view as a source; "total revenue last quarter by fare class" becomes MEASURE(`Total Revenue`) ... GROUP BY `Fare Class` — governed, matching the dashboard.
💡 Tip · fieldOn day one, prove BI, SQL, and Genie agree: run Walkthrough 1 in SQL, chart the same breakdown, and ask Genie "revenue by fare class". Identical numbers is the demo that sells the semantic layer. A mismatch means something bypassed the metric view.
08 · Field notes

What to remember

📌 ImportantTwo rules always: name the dimensions, MEASURE() the measures — SELECT * never works. Joins are the view's job, not the query's. Ratios are correct at any grain because they are measures.
💡 Tip · fieldAlias every measure and order/filter by the alias — sidesteps the "can't filter a measure in WHERE" trap. Expose the time grains you use as dimensions.
⚠️ GotchaNeeds a Pro/Serverless SQL warehouse (or DBR 17.2+); nested joins need 17.1+. WHERE MEASURE(...) is not allowed — filter in an outer query. If a joined dimension is null everywhere, the on: join key in the YAML is wrong — fix the definition, not the query.
