Business Semantics — Metric Views

Define each KPI once in YAML, and let every dashboard, SQL query, and the Module 14 Genie Agent re-aggregate it safely. One governed definition of "revenue" — no metric drift.

01 · The big idea

One definition of a metric, many questions

A metric view is a Unity Catalog object defined in YAML that stores dimensions (how you slice) apart from measures (how you aggregate). Because the two are separate, aggregation happens at query time — not baked in at creation the way a regular view fixes its GROUP BY. One view answers "revenue by fare class" and "revenue by month by region", and ratios stay correct at every roll-up.

  • Query it with MEASURE() and GROUP BY ALL — SELECT * is not supported.
  • It kills metric drift — finance, ops, analysts, and Genie all read the one governed definition, so the numbers agree.
  • It models a star / snowflake schema — joins: live in the YAML, so consumers never write a JOIN.
  • 15.6 feeds Module 14 — business-friendly names + comments are the metadata a Genie Agent reads to answer plain-English questions accurately.
📌 The rule that shapes this moduleDefine each metric once, in YAML, and let every tool re-aggregate it safely. The aggregation is a query-time choice via MEASURE() — so one definition serves many questions and ratios (cancellation rate, attach rate) recompute their numerator and denominator at any grain. Metric view unity_airways.analytics.bookings_metrics is the trusted KPI source the Module 14 Genie Agent queries.
02 · The problem

Why "revenue" means four different numbers

Unity Airways has revenue in a finance dashboard, an ops dashboard, an analyst's notebook, and a Genie Agent — each built by a different person. One counts base fare only; one adds ancillary; one drops cancelled bookings; Genie doesn't. Same word, four numbers, no trust. Naive fixes fail:

  • A regular view with the GROUP BY baked in → locks the grain forever; a second grain means a second view, and rolled-up ratios come out wrong.
  • Definitions inside each BI tool → logic lives in N tools, drifts independently, and can't be governed as one UC object.
  • Let Genie infer metrics from raw tables → different phrasings produce different SQL; "revenue" is re-derived every time.
  • Pre-aggregate everything into summary tables → you must predict every slice and still hand-code each ratio.

04 · Concept

15.1 — What metric views are, and why a semantic layer matters

15.1 Semantic layer Theory

A metric view is Unity Catalog's answer to metric drift: a governed object that stores what a metric means in YAML, so every tool computes it the same way. The core split is measures (how to aggregate) apart from dimensions (how to slice) — a regular view fuses them at creation, a metric view keeps them separate so the grain is a query-time choice.

  • Why it matters: it is the single place "revenue", "bookings", and "cancellation rate" are defined — the agreement across finance, ops, analysts, and Genie is the product.
  • What it is not: not a materialized table (though it can be materialized — 15.5), not a BI-tool model. It is a first-class UC object with grants and lineage.
📌 ImportantThe value is governance, not just convenience. "What does revenue mean and where does it come from" has one auditable answer — the YAML plus its lineage back to fct_bookings.
05 · Create + edit

15.2 — Create and edit metric views

15.2 Create & edit Hands-on

Create with one SQL statement whose body is YAML (or the Catalog Explorer visual editor). Editing = re-issue CREATE OR REPLACE VIEW with the updated YAML — there is no partial in-place patch in SQL.

CREATE OR REPLACE VIEW unity_airways.analytics.bookings_metrics
WITH METRICS
LANGUAGE YAML
AS $$
  version: 1.1
  comment: "Unity Airways bookings KPIs — governed semantic layer; trusted Genie source."
  source: unity_airways.analytics.fct_bookings
  dimensions:
    - name: Fare Class
      expr: fare_class
      comment: "Cabin / booking class: Economy, Premium, Business, First."
  measures:
    - name: Total Revenue
      expr: SUM(base_fare_usd + ancillary_usd)
      comment: "Base fare plus ancillary revenue, in USD."
    - name: Booking Count
      expr: COUNT(booking_id)
$$
💡 Tip · fieldKeep the CREATE OR REPLACE ... WITH METRICS YAML in version control. The metric view is code — an edit moves every downstream number, so review it like code. Verify with DESCRIBE EXTENDED + a one-line MEASURE() query.
06 · Cornerstone

15.3 ★ — Query metric views; tutorial with joins

Full deep-dive — MEASURE(), GROUP BY ALL, filtering, time dimensions, star/snowflake joins, and BI/SQL/Genie consumption — in the querying deep-dive (query-metric-views.md / query-metric-views.html).

15.3 Query + joins Hands-on Cornerstone

Two rules: name the dimensions, wrap every measure in MEASURE(). Joins are invisible — asking for Region or Loyalty Tier needs no JOIN because the YAML declared it.

SELECT
  `Fare Class`,
  MEASURE(`Total Revenue`) AS total_revenue,
  MEASURE(`Booking Count`)  AS bookings
FROM unity_airways.analytics.bookings_metrics
GROUP BY ALL
ORDER BY total_revenue DESC
📌 ImportantSELECT * and bare measure references fail — a measure has no value until you pick a grain. Filter dimensions in WHERE; to filter on a measure, wrap the aggregate query and filter the alias in an outer SELECT.
07 · Modeling

15.4 — Modeling metric views; advanced techniques

15.4 Modeling Theory Hands-on

  • Star (one-level) joins — reference the fact as source and each dim by its join name: on: source.customer_id = customers.customer_id.
  • Snowflake (nested) joins, DBR 17.1+ — Region comes from dim_airports reached through dim_flights: nest the airport join under the flight join.
  • Derived dimensions — CASE / DATE_TRUNC for business categories (Booking Month, priority buckets).
  • Filtered measures — COUNT(1) FILTER (WHERE status = 'cancelled'), no second view.
  • Ratio measures — SUM(ancillary_usd) / SUM(base_fare_usd); numerator + denominator stay separate, so the ratio is correct at any grain.
  • Window measures (experimental) — a window: block for running totals and period-over-period; shown with version: 0.1 — verify runtime support.
joins:
  - name: flights
    source: unity_airways.analytics.dim_flights
    on: source.flight_id = flights.flight_id
    joins:
      - name: origin_airport_info
        source: unity_airways.analytics.dim_airports
        on: flights.origin_airport = origin_airport_info.airport_code
⚠️ GotchaA dimension cannot contain an aggregate and a measure must. Slices go in dimensions; anything with SUM/COUNT/AVG goes in measures. This is the most common authoring error.
08 · Materialization

15.5 — Materialization for metric views

15.5 Materialization Theory Hands-on Experimental

By default a metric view computes on read. Materialization pre-computes chosen slices on a schedule (via Lakeflow / Spark Declarative Pipelines) so hot queries hit a cached table. It is a performance choice — the metric definition and query grammar never change.

materialization:
  schedule: every 6 hours
  mode: relaxed
  materialized_views:
    - name: revenue_by_month_class_region
      type: aggregated
      dimensions: [Booking Month, Fare Class, Region]
      measures:   [Total Revenue, Booking Count]
    - name: full_model
      type: unaggregated   # source + joins + filter
  • aggregated = a specific dimension/measure combo (hot slices). unaggregated = the joined-and-filtered model (expensive sources/joins).
  • Requires serverless compute and DBR 17.2+; TRIGGER ON UPDATE is not supported.
💡 TipDon't materialize first. Ship the view, watch which slices are hot, then materialize exactly those. Materializing everything re-creates the summary-table sprawl you were escaping.
09 · Feeds Module 14

15.6 — Agent metadata: feeding Genie/agents

15.6 Agent metadata Theory Hands-on

The bridge to Module 14. A Genie Agent maps plain-English words to metrics using the metric view's metadata:

  • Display names = the name field. Naming a measure Total Revenue (not m1) is what lets "revenue" / "total sales" resolve. Business names are the primary agent metadata.
  • comment on the view and every dimension/measure is the description Genie reads to disambiguate — state meaning, units, and alternate phrasings.
  • Synonyms in the comment are the portable, verified way to teach alternate terms today ("a.k.a. cabin, class"). Whether the YAML also exposes a dedicated synonyms key or a value-format key is evolving — verify against current docs; embedding in comment works regardless.
- name: Cancellation Rate
  expr: COUNT(1) FILTER (WHERE status = 'cancelled') * 1.0 / COUNT(1)
  comment: "Share of bookings cancelled (0-1). Also known as: cancel rate, cancellation percentage."
📌 ImportantNames and comments are the interface between your data and the LLM. A metric view with clear names and rich comments is a better Genie source than raw tables, because the semantics are explicit instead of inferred. In Module 14 you add the metric view as a Genie data source.
10 · Reference

15.7 — YAML syntax reference; manage metric views

15.7 YAML reference Hands-on

The full YAML surface is in the YAML keys tab above. Operate the object like any UC view:

GRANT SELECT ON VIEW unity_airways.analytics.bookings_metrics TO `data-consumers`;
DESCRIBE EXTENDED unity_airways.analytics.bookings_metrics;
-- Learner surface: SQL DDL (above), Catalog Explorer, or the Python SDK.
-- manage_metric_views(create|alter|describe|query|drop|grant) is an agent/MCP build tool, not a public API.
⚠️ GotchaMetric views require a Pro or Serverless SQL warehouse (or DBR 17.2+ for v1.1) — not a basic/classic warehouse. If WITH METRICS errors, check the warehouse type and runtime version first.
11 · Worked example

Unity Airways — the full semantic layer

The consolidated lab builds this end to end, and the three Module 14 Genie questions map straight onto measures + dimensions:

  • "Total revenue last quarter by fare class" → MEASURE(`Total Revenue`) by Fare Class, filtered on Booking Quarter.
  • "Which routes have the highest cancellation rate" → MEASURE(`Cancellation Rate`) by Origin Airport, Dest Airport.
  • "Compare ancillary attach rate across loyalty tiers" → MEASURE(`Ancillary Attach Rate`) by Loyalty Tier.
SELECT `Region`, `Loyalty Tier`,
       MEASURE(`Total Revenue`)     AS revenue,
       MEASURE(`Cancellation Rate`) AS cancel_rate
FROM unity_airways.analytics.bookings_metrics
GROUP BY ALL
ORDER BY revenue DESC   -- no JOIN written: the YAML resolves Region + Loyalty Tier
12 · Field notes

What to remember

📌 ImportantAggregation is a query-time choice (MEASURE() + GROUP BY ALL) — one view, many questions, correct ratios at any grain. The metric view is Genie's trusted source (15.6 → Module 14): names + comments are the metadata. It is one governed UC object — grants, lineage, a single definition.
💡 Tip · fieldVersion-control the YAML; write comments for an LLM reader (meaning, units, synonyms); ship first and materialize only the hot slices.
