Module 07 — MLflow Tracing & Observability · Topics 07.1–07.5 · Theory + Hands-on

MLflow Tracing

When the Unity Airways assistant answers wrong, the output alone can't tell you which step broke — retrieval, prompt, or LLM. This module makes Module 05's RAG chain observable: one Trace per question, nested Spans for every step, queryable evidence you can debug, evaluate, and monitor.

01 · The big idea

From an opaque answer to a step-by-step flight recorder

Module 05 built unity_airways.rag.ua_rag_chain. Module 07 makes each request legible: a Trace records the end-to-end journey, and inside it nested Spans capture the retriever call, the prompt, and the ChatDatabricks LLM call — each with inputs, outputs, and timing.

  • Trace = TraceInfo + TraceData — TraceInfo is searchable metadata (experiment, start time, status, tags); TraceData is the list of Spans in a parent–child tree.
  • Two ways to instrument (the combined cornerstone): automated (mlflow.langchain.autolog(), one line, 20+ libraries) and manual (@mlflow.trace + mlflow.start_span(...)) for custom steps.
  • Query traces with mlflow.search_traces(...) (a pandas DataFrame) or the Traces tab in the MLflow UI.
  • Tracing is OpenTelemetry-compatible; mlflow.set_active_model() links each trace to a LoggedModel version (Module 06). Needs MLflow ≥ 3.1.
📌 The rule that shapes this moduleTraces are the evidence, not a nice-to-have. A GenAI app is a multi-step, probabilistic system — the final output can't tell you which step broke. Instrument early: automated tracing (mlflow.<lib>.autolog()) for supported libraries, manual spans (@mlflow.trace, mlflow.start_span) for custom glue, mlflow.search_traces() to triage — and the same traces feed evaluation (Module 08) and monitoring (Module 13).
02 · The problem

Why print(), eyeballing, and ad-hoc timers fail

A customer asks "Can I get a refund on a cancelled flight?" and the answer is wrong. The chain retrieved chunks, built a prompt, and called the LLM — and the answer doesn't say which step failed. Naive observability breaks:

  • print() / logging → captures discrete events, not the relationships between steps; per-step latency isn't captured and must be derived (Table 5-1).
  • Eyeball the final answer → hides the two things you need most: the retrieved chunks and the exact prompt sent to the LLM.
  • Ad-hoc timers → manual, brittle, throwaway; they don't survive into monitoring or feed evaluation. Tracing captures timing + metadata automatically.
  • "Add observability later" → by prod the failures are subtle and compound: empty context, a swallowed tool error, a dead orchestration branch (📘B1 Ch2).

04 · Concepts

07.1 — Trace & Span

07.1 Tracing concepts — Trace & Span Theory

MLflow Tracing gives detailed tracking + visualization of a request as it propagates through a GenAI app — for a RAG chain that means the input question, the retrieval query, the retrieved chunks, the prompt, and the response. The goal is observability.

Tracing is not logging (📘B1 Ch5, Table 5-1):

LoggingTracing
PurposeDiscrete events in an appEnd-to-end journey of one request
ContextNo relationships between eventsMaintains context across the request
PerformanceLatency must be post-processedKey metrics captured automatically

Anatomy: Trace = TraceInfo + TraceData. TraceInfo is metadata (experiment ID, start time, duration, status, tags) — what you search on. TraceData is a list of Spans linked hierarchically. Each Span captures one step's inputs, outputs, attributes, and events.

📌 ImportantThe RETRIEVER span type is special: it has a required schema so retrieved documents render well in the UI and can be consumed by evaluation (Module 08 reads retrieved context from it). Autolog sets it for known retrievers; for a custom one, use span_type=SpanType.RETRIEVER.
💡 TipMLflow Tracing is OpenTelemetry-compatible with GenAI semantic conventions — if your org already uses OpenTelemetry for microservices, GenAI traces slot into the same trace-with-nested-spans mental model.

07.2 Automated tracing Hands-on Cornerstone

The easiest start: add mlflow.<flavour>.autolog() and MLflow captures spans for you across 20+ libraries (LangChain, LangGraph, OpenAI, LlamaIndex, …).

import mlflow
mlflow.set_experiment("/Users/you@co.com/unity_airways_rag")
mlflow.set_active_model(name="ua_rag_chain")   # link traces to a LoggedModel version (Module 06)

mlflow.langchain.autolog()                      # one line — auto-captures the chain's spans
# mlflow.openai.autolog() / mlflow.anthropic.autolog() for those SDKs

answer = ua_rag_chain.invoke({"messages":[{"role":"user",
          "content":"Can I get a refund on a cancelled flight?"}]})
# -> one Trace: retriever span + ChatDatabricks LLM span, nested under the chain

07.3 Manual tracing / custom spans Hands-on Cornerstone

For steps autolog can't see (a re-ranker, a policy filter, a custom parser). Two Fluent tools: the @mlflow.trace decorator and the mlflow.start_span(...) context manager.

from mlflow.entities import SpanType

@mlflow.trace(span_type=SpanType.PARSER, name="normalize_question")
def normalize(q: str) -> str:
    return q.strip().lower()

with mlflow.start_span(name="policy_filter", span_type=SpanType.CHAIN) as span:
    span.set_inputs({"query": query_text})              # record inputs
    kept = [c for c in chunks if c.policy == "refund"]
    span.set_outputs({"kept": len(kept)})               # record outputs
    span.set_attributes({"filter_rule": "refund_only"}) # arbitrary metadata
⚠️ GotchaThe high-level Fluent APIs (@mlflow.trace, mlflow.start_span) and the low-level Client APIs (manual start_span/end_span with explicit parent IDs) cannot be mixed in one trace — they don't interoperate. Use the Client API only for fine-grained control; Fluent is the recommended path.
💡 TipThey compose: turn on mlflow.langchain.autolog() for the chain's built-in steps, then add a couple of @mlflow.trace spans around custom Python glue — full coverage without hand-instrumenting everything.
06 · Query

07.4 — querying traces

07.4 Querying traces Hands-on

Find the traces worth investigating — the slow, the failed, the wrong. Two surfaces: the Traces tab in the experiment UI, and mlflow.search_traces(...), which returns a pandas DataFrame (one row per trace) or a list of Trace objects.

import mlflow

traces = mlflow.search_traces(
    filter_string="attributes.status = 'ERROR' AND attributes.execution_time_ms > 2000",
)
# traces is a pandas DataFrame — one row per trace
slow_failures = traces[["trace_id", "request", "execution_time_ms"]]

Searchable fields (Table 5-2): metadata, tags, status (OK / ERROR / IN_PROGRESS), name, timestamp (ms), execution time (ms). Syntax rules: table prefix required, backtick dotted fields, single quotes only, case sensitive, AND only.

07 · Discipline

07.5 — trace-first development

07.5 Trace-first development as a discipline Theory

Tracing is a development discipline, not an end-stage add-on. The book frames a five-phase lifecycle — Develop → Evaluate → Deploy → Monitor → Improve (📘B1 Ch2, Fig 2-1) — and tracing keeps the loop honest:

  • Develop from a working skeleton; from the first coherent answer, instrument the key steps so every call records inputs, outputs, timing, and metadata.
  • "The difference between debugging and guessing is whether those traces exist." Development traces become the raw material for an initial evaluation set (Module 08).
  • Monitor: real-world traces let you replay the execution path, spot retries, and pinpoint the prompt or retriever behind slow / off-topic answers (Module 13).
📌 ImportantObservability gets harder as systems become compound — failures turn subtle: retrieval returns empty context, a tool throws an error the model quietly ignores, a branch never executes. Trace-first development is the only way these stay visible instead of silent.
08 · Worked thread

Unity Airways, a wrong answer traced to a fix

  1. A bad answer lands — the assistant wrongly says a cancelled flight isn't refundable.
  2. Find it (07.4): mlflow.search_traces(...) over recent traces; locate the request by its text in the DataFrame.
  3. Open the trace (07.1): expand the tree — chain (root) → retriever → prompt → LLM.
  4. Diagnose (07.2–07.3): the RETRIEVER span pulled the baggage-policy chunk, not the refund chunk — retrieval failed, not the LLM. Autolog captured it; nothing hand-instrumented.
  5. Attribute the version: the trace links via set_active_model() to a LoggedModel version (06.2) — you know which version misbehaved.
  6. Feed evaluation (Module 08): add the question to the eval set; a retrieval scorer reads the context straight from the trace.
  7. Fix and re-observe: bump k or re-chunk the corpus; confirm the new trace's retriever span returns the refund chunk.
  8. Monitor (Module 13): the same traces stream at volume with dashboards + alerts on error rate and per-span latency.
09 · Field notes

What to remember

📌 ImportantTrace = TraceInfo + TraceData; a Span is one step. Two ways to instrument: automated (mlflow.<lib>.autolog()) and manual (@mlflow.trace, mlflow.start_span). Query with mlflow.search_traces(). Use SpanType.RETRIEVER so chunks render and evaluation can read them. Link versions with set_active_model().
💡 Tip · fieldAutolog first, manual spans for custom glue. Instrument from the first working path so every workflow can explain its own results — then the same traces power debugging, evaluation (Module 08), and monitoring (Module 13) with no rework.
⚠️ GotchaDon't mix Fluent + Client tracing APIs. Don't expect OR in search_traces filters (AND only, case sensitive, single quotes). Don't use a plain span for retrieval — use RETRIEVER. Verify the exact search_traces filter prefix against current docs.
