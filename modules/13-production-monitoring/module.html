Module 13 — Production monitoring and continuous improvement · Topics 13.1–13.7 · Theory + Hands-on

Production Monitoring

Taking the Unity Airways support agent from safe and shipped to watched and always improving — see quality drift, get paged before travelers notice, and grow tomorrow's eval set from today's real traffic. The one idea: the monitor → improve loop.

01 · The big idea

Deployment is the start, not the finish

Modules 11–12 got the agent served, guardrailed, and governed. Module 13 keeps it good. A GenAI app in production is a moving target — prompts change, users ask new things, retrieval drifts — so you close a loop and run it forever.

  • Capture — every request becomes an MLflow Trace, an inference-table row, and (when a user reacts) a feedback entry (13.2, 13.3).
  • Assess — score those traces with the same MLflow 3 scorers you used offline in Module 08, plus SQL/NLP analysis for deeper questions (13.4, 13.5 ★).
  • Watch — read the numbers on an AI/BI dashboard and get paged by alerts when something drifts (13.5 ★, 13.6).
  • Improve — curate the failing traces into your eval dataset, re-evaluate v2, redeploy, and watch again (13.7).
📌 The rule that shapes this moduleCapture the trace → assess with reused scorers → watch on a dashboard + alerts → improve by curating failures back into the eval set. Name maturity as you go: GA — MLflow Tracing, inference tables, ai_mask, AI/BI dashboards, SQL alerts; Beta — production monitoring (online scorers on prod traces); Preview — trace-to-Unity-Catalog sync.
02 · The problem

"The endpoint is up" is not "the agent is good"

A week after launch, the steady-state operating questions arrive — none of which a launch checklist answers:

  • Is answer quality the same today as at launch, or has it quietly slipped?
  • A traveler thumbs-downs an answer — where does that signal go, and can we find that trace later?
  • Latency crept from 2 s to 8 s over a week — what changed, and would we notice before customers churn?
  • Users keep asking about a topic the agent was never built for — how would we even know?
  • We want to ship v2 in six weeks — what do we test it against to be confident it's actually better?

These are the questions you can only answer if the loop was built in from day one.

04 · Capture

13.1 — Metric types: operational, quality, business impact

13.1 Metric types Theory

  • Operational — system health: performance (end-to-end + per-component latency, e.g. FAQ AI Search retrieval), reliability (# requests / # failures over 7 days), resource use (LLM in/out tokens, endpoint CPU/memory). Easy to collect; MLflow Tracing adds the per-stage breakdown (B1 Table 9-1).
  • Quality — how good the answers are: response quality (accuracy, right FAQ, on-brand tone), safety and compliance (bias, harmful content, PII), user experience (ratings, CSAT). Unstructured text ⇒ needs NLP / LLM-as-judge (B1 Table 9-2).
  • Business impact — engagement (recurring users, unanswerable topics) and improvement (tickets deflected, time saved, task-completion lift). Some needs an internal study on top of traces (B1 Table 9-3).
💡 Exam angleNo single metric decides model choice — weigh groundedness, task completion, latency, token usage, and stability together (B2 Table 8-2). Don't default to the biggest model.
05 · Capture

13.2 — Inference tables and logs

13.2 Inference tables and logs Theory Hands-on GA

  • Each row is one model invocation — request id, timestamp, prompt/response, prompt_tokens/completion_tokens/total_tokens, latency_ms, model_name. The primary audit trail (B2 Fig 8-3).
  • Source: AI Gateway payload logging (Module 11.3) and, for agents, agents.deploy(). Flow: user request → LLM endpoint → inference log → inference table → dashboard.
  • RAG diagnosis (B2 Table 8-3): logs separate retrieval failure (missing context) from hallucination (good context, wrong answer). Rising tokens with no quality gain ⇒ trim retrieved chunks.
-- last 50 calls (adapt for aggregates)
SELECT request_id, timestamp, prompt_tokens, completion_tokens,
       total_tokens, latency_ms, model_name
FROM   unity_airways.rag.<inference_table>
ORDER BY timestamp DESC
LIMIT 50;
⚠️ GotchaInference tables store raw user content — apply retention policies and redaction, or they become an ungoverned copy of regulated data.
06 · Capture

13.3 — Online monitoring workflow; real-time trace capture

13.3 Online monitoring workflow Hands-on GA

  • No new instrumentation. The Databricks Agent SDK auto-captures MLflow Traces the moment a request hits the endpoint — your Module 07 traces are reused as-is in production.
  • Get the trace_id back so feedback attaches to the right request: query with "databricks_options": {"return_trace": true}.
  • Off-Databricks? Same idea via MLFLOW_TRACKING_URI=databricks + MLFLOW_EXPERIMENT_NAME + host/token.
# link a traveler's thumbs-up/down to a specific trace
import mlflow
from mlflow.entities.assessment import AssessmentSource, AssessmentSourceType

mlflow.log_feedback(
    trace_id=trace_id,                 # returned with the chat response
    name="user_feedback",
    value=satisfied,                   # True = thumbs-up, False = thumbs-down
    rationale=rationale,               # optional short comment
    source=AssessmentSource(
        source_type=AssessmentSourceType.HUMAN,
        source_id=user_id),
)
💡 Verify it workedOpen the trace in the MLflow UI — the feedback appears under Assessments; the Traces tab lists real-time requests with tokens, execution time, and state.
07 · Assess

13.4 — Agent monitoring tools (reuse the Module 08 scorers)

13.4 Agent monitoring tools Theory Hands-on Beta

  • Two complementary views (B2 Table 8-4): inference tables (per-call depth) and agent monitoring (per-workflow context — which step is slow, where an error propagates). A multi-step agent needs both.
  • Production monitoring reuses the Module 08 scorers so dev and prod measure quality identically. Register once, then schedule on a sample of live traces.
  • .start() auto-provisions a Lakeflow Job ("Trace Metrics Computation Job") that runs the judges on the sampled traces, and MLflow creates a default dashboard in the experiment.
from mlflow.genai.scorers import (
    RelevanceToQuery, Safety, RetrievalGroundedness,
    list_scorers, get_scorer, ScorerSamplingConfig)

# register (skip if already on the experiment)
for s in [RelevanceToQuery(), Safety(), RetrievalGroundedness()]:
    if s.name not in [x.name for x in list_scorers()]:
        s.register(name=s.name)

# prerequisite: set_databricks_monitoring_sql_warehouse_id(sql_warehouse_id="<ID>", experiment_id=EXPERIMENT_ID)  # see aibi-dashboard.html
# schedule on live traces — sample_rate=1.0 scores every trace
for name in ["relevance_to_query", "safety", "retrieval_groundedness"]:
    get_scorer(name=name).start(
        sampling_config=ScorerSamplingConfig(sample_rate=1.0))
⚠️ GotchaProduction monitoring is Beta; custom judges via @scorer / make_judge() need MLflow ≥ 3.4. Tune sample_rate down for expensive/less-critical scorers; keep it 1.0 for safety.
08 · Assess + Watch · Cornerstone

13.5 ★ — NLP analysis on traces; custom AI/BI dashboard

Full build — syncing traces to Unity Catalog, the ai_query NLP pass, and assembling the AI/BI (Lakeview) dashboard end to end — in the AI/BI monitoring dashboard deep-dive (aibi-dashboard.md / aibi-dashboard.html).

13.5 NLP on traces + AI/BI dashboard Hands-on Cornerstone ai_query GA Trace-sync Preview

  • Why go past the defaults: the auto dashboard covers standard metrics, but questions like "what are the most common question categories, and what can't we answer?" need custom analysis.
  • Sync traces to UC: from the MLflow Traces UI enable Sync traces to Unity Catalog → a Delta table like unity_airways.rag.ua_trace_text (creates a "Trace Archive Job"). Preview.
  • NLP with ai_classify (preferred): pull the question text with get_json_object(request, '$.request.input[0].content'), classify intent via ai_classify(text, ARRAY(...)) for fixed labels (book fallback: ai_query(...) + a responseFormat JSON schema), then aggregate.
  • AI/BI dashboard: create a dataset from that SQL and add widgets (intent-mix pie, latency percentiles, pass-rate trends). Lakeview / AI-BI — a UI + SQL surface, no invented dashboard API.
📌 ConsistencyThe hub and the cornerstone share the same table unity_airways.rag.ua_trace_text, the same endpoint ua-support-agent, and the same scorer/tracing APIs. Ad-hoc alternative: Genie Code on the experiment or the MLflow MCP server for natural-language trace questions.
09 · Watch

13.6 — Metric alerts and anomaly detection

13.6 Metric alerts and anomaly detection Hands-on GA

  • Dashboards are passive; alerts page you. Set a Databricks SQL alert on a query over the traces/inference table — e.g. count of the "Others" intent bucket (a topic-drift signal) crossing a threshold → email/notification (B1 Fig 9-14).
  • Alert shape: a SQL query + a condition (Count(intent_count) > 5), a schedule, and a destination. Combine static thresholds (catch spikes) with statistical / baseline-deviation alerts (catch slow drift) — B2 Fig 8-4.
  • Anomaly pipeline: inference table + metric baselines → anomaly detection → alert trigger. Baselines often come from your controlled MLflow eval results.
💡 Tip · fieldMake every alert actionable — name the responsible team, response time, and first investigation step. Keep separate categories for performance, cost, and reliability, or alerts become noise.
10 · Improve

13.7 — The "Improve" loop: expanding eval sets from production

13.7 The "Improve" loop Theory Hands-on

  • The point of monitoring is the next version. Real production traces are the best ground truth — fold failing/interesting ones into the Module 08 eval dataset so v2 is tested against reality (B1 Ch9).
  • The whole loop (B1 Fig 9-17): domain experts + production feed the eval dataset → evaluate in dev with the shared scorers → deploy → monitor in prod → curate failures back → repeat.
  • Redeploy discipline: promote v2 by moving an alias (Module 11.8 / 12.7) so the endpoint URL stays stable and the audit trail holds.
import mlflow

ds = mlflow.genai.datasets.get_dataset(
    f"{catalog}.{schema}.{eval_dataset_table_name}")

traces = mlflow.search_traces(          # ⚠️ experiment_ids, NOT experiment_names
    experiment_ids=[experiment_id],
    filter_string="attributes.status = 'OK'",   # in practice: negative feedback / failed scorers
    order_by=["attributes.timestamp_ms DESC"],
    max_results=10)

ds = ds.merge_records(traces)           # closed feedback loop
⚠️ Gotchamlflow.search_traces takes experiment_ids= (a list), not experiment_names= — there is no experiment_names argument (passing it raises TypeError). And filter on negative feedback / failed scorers, not just status='OK' — you want the regressions.

12 · Field notes

What to remember

📌 ImportantThe monitor → improve loop is the whole module: capture (traces / inference tables / feedback) → assess (reused Module 08 scorers + NLP) → watch (AI/BI dashboard + alerts) → improve (curate failing traces into the eval set) → redeploy. The trace is the atom — it is the signal, the scorer input, the dashboard row, the alert trigger, and the next eval-set record.
💡 Tip · fieldPlan what you want to see in prod while instrumenting in dev (Module 07) — traces carry over unchanged. Pair every Module 12 guardrail/rate limit with a monitor here. Filter the improve-loop query on negative feedback / failed scorers, not status='OK'. Use Genie Code or the MLflow MCP server for one-off questions instead of a new dashboard.
⚠️ GotchaProduction monitoring is Beta; some APIs need MLflow ≥ 3.4 and trace-to-UC sync is Preview — label maturity. mlflow.search_traces uses experiment_ids=. The books use their own naming (B1 uses workspace.unity_airways.production_traces, judge model databricks-gpt-oss-120b); this hub standardizes on unity_airways.rag.* + chat model databricks-claude-sonnet-4-5 — APIs are identical.
