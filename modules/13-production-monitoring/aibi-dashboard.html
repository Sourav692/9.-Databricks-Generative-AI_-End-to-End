Module 13 — Production Monitoring · Topic 13.5 · Hands-on ★

NLP on Traces + AI/BI Dashboard

Your agent already emits the signal: traces and an inference table. This lesson turns that raw stream into a page a director can read — run NLP and quality analysis over it, land one metrics table, and publish an AI/BI dashboard: volume, latency, quality, sentiment, topics.

01 · The big idea

Raw signal in, a dashboard a stakeholder trusts out

Topic 13.3 gets traces flowing and 11.3 turns on the AI Gateway inference table. 13.5 is where that signal becomes a standing, refreshing view of how the Unity Airways agent is doing.

  • Two raw signals, already flowing. MLflow traces (Module 07 — latency, tokens, status, text) and the AI Gateway inference table (11.3 — request/response JSON, guardrail outcomes).
  • Two enrichment engines. AI Functions in SQL (ai_classify for topic, ai_analyze_sentiment, ai_summarize) turn text into columns; MLflow 3 scorers running as monitors (Safety, RelevanceToQuery, RetrievalGroundedness) turn quality into columns. Production monitoring is Beta.
  • One flat metrics table. Everything lands in unity_airways.rag.ua_request_metrics — one row per request, every metric a column. That table is the contract the dashboard reads.
  • An AI/BI dashboard on top. Lakeview tiles: volume, latency p50/p95, quality trend, sentiment mix, top topics. Built in the UI; the work is the dataset SQL.
  • Read traces with mlflow.search_traces(experiment_ids=[...]) — a list of IDs, never experiment_names=.

03 · How it works

Structured for free, unstructured with AI

Some metrics are already structured fields on the trace. The rest live in free text and need either an AI Function or an LLM scorer to become a column.

MetricWhere it comes fromMaturity
latency_mstrace.info.execution_time_ms — structured, no workGA
tokensLLM span token-usage attribute (token_usage — confirm the key in your traces)GA
topicai_classify(question, ARRAY(...)) — fixed labels + OtherGA
sentimentai_analyze_sentiment(question)GA
relevance / safety / groundednessMLflow scorers running as production monitorsBeta
user_feedbackmlflow.log_feedback from the app (13.3)GA
  • Prefer task-specific AI Functions. ai_classify for fixed-label topics, ai_analyze_sentiment for sentiment. Reach for ai_query only when you need custom structured JSON.
  • Reuse the Module 08 scorers. The same Safety, RelevanceToQuery, RetrievalGroundedness you evaluated with offline now watch production — one metric definition, dev to prod.
  • Both .register() and .start(). Registering alone activates nothing. ScorerSamplingConfig(sample_rate=...) caps LLM-judge cost; safety at 1.0, others lower.
  • Enabling monitoring provisions a Lakeflow Job that runs the scorers on a schedule and writes results back onto the traces.
04 · Run this

Pull, score, enrich, land, publish

Prereqs: the deployed agent from Module 11 (ua-support-agent) with traces flowing (13.3), payload logging on (11.3), a SQL warehouse, MLflow 3.x recent enough for production monitoring (verify — around 3.4+), and CAN USE on the warehouse + CAN EDIT on the experiment.

1) Pull traces into a DataFrame — experiment_ids=, never experiment_names=:

import mlflow

EXPERIMENT_ID = "3430349293389553"   # the experiment your endpoint logs to

traces_df = mlflow.search_traces(
    experiment_ids=[EXPERIMENT_ID],              # a LIST of IDs
    filter_string="attributes.status = 'OK'",    # attributes. prefix, single quotes, AND only
    order_by=["attributes.timestamp_ms DESC"],
    max_results=5000,
)
print(len(traces_df), "traces", traces_df.columns.tolist())

2) Start scorer monitors (Beta) — register AND start:

from mlflow.genai.scorers import (
    Safety, RelevanceToQuery, RetrievalGroundedness, ScorerSamplingConfig,
)
from mlflow.tracing import set_databricks_monitoring_sql_warehouse_id

mlflow.set_experiment("/Shared/ua-support-agent")
set_databricks_monitoring_sql_warehouse_id(
    sql_warehouse_id="<SQL_WAREHOUSE_ID>", experiment_id=EXPERIMENT_ID,
)

JUDGE = "databricks:/databricks-gpt-oss-120b"
safety = Safety(model=JUDGE).register(name="prod_safety").start(
    sampling_config=ScorerSamplingConfig(sample_rate=1.0))      # every trace
relevance = RelevanceToQuery(model=JUDGE).register(name="prod_relevance").start(
    sampling_config=ScorerSamplingConfig(sample_rate=0.5))
grounded = RetrievalGroundedness(model=JUDGE).register(name="prod_groundedness").start(
    sampling_config=ScorerSamplingConfig(sample_rate=0.5))

3) Enrich the text with AI Functions (SQL over the trace/inference text):

SELECT
  trace_id, request_time, user_question, agent_response, latency_ms, total_tokens,
  ai_classify(user_question,
    ARRAY('Check-in','Cancellation','Baggage','Refunds','Loyalty','Other')) AS topic,
  ai_analyze_sentiment(user_question)  AS sentiment,
  ai_summarize(agent_response, 20)     AS answer_summary
FROM unity_airways.rag.ua_trace_text
WHERE user_question IS NOT NULL;

4) Land one flat metrics table (join structured + scorer + AI-Function columns on trace_id):

(metrics_df.write.format("delta").mode("overwrite")   # or MERGE on trace_id
   .option("overwriteSchema","true")
   .saveAsTable("unity_airways.rag.ua_request_metrics"))

5) Build the dashboard — New → Dashboard, add a dataset per tile, drop widgets, Publish. The dataset SQL does the work:

-- daily_volume_latency
SELECT DATE(request_time) AS day,
       COUNT(*) AS request_count,
       percentile_approx(latency_ms, 0.5)  AS p50_latency_ms,
       percentile_approx(latency_ms, 0.95) AS p95_latency_ms
FROM unity_airways.rag.ua_request_metrics
WHERE request_time >= date_sub(current_date(), 30)
GROUP BY DATE(request_time) ORDER BY day;

-- top_topics (bar) / sentiment_mix (pie)
SELECT topic, COUNT(*) AS n FROM unity_airways.rag.ua_request_metrics GROUP BY topic ORDER BY n DESC;
SELECT sentiment, COUNT(*) AS n FROM unity_airways.rag.ua_request_metrics GROUP BY sentiment;

How to verify it worked: list_scorers() shows your monitors with their sample rates; after the monitoring Lakeflow Job runs, traces carry scorer assessments. SELECT COUNT(*) and a few rows from ua_request_metrics confirm topic/sentiment/quality columns are populated. In the published dashboard, the volume and latency tiles render with 30 days of data and the topic bar shows your fixed label set plus Other.

05 · Worked example

Unity Airways, one Monday morning

The metrics job ran overnight over the weekend's traces. The dashboard tells the whole story at a glance:

  • Volume: 4,120 requests over 7 days, up 18% week over week. Adoption is real.
  • Latency: p50 1.9s, p95 6.4s — the p95 line ticked up Saturday, one for 13.6 to watch.
  • Quality trend: avg_safety flat at 0.99; avg_groundedness dipped 0.94 → 0.88 midweek. A retrieval regression, caught before any ticket.
  • Top topics: Baggage 38%, Check-in 24%, Cancellation 16%, Refunds 12%, Loyalty 6%, Other 4%. Sampling that Other slice shows seat-upgrade questions — a feature request nobody had logged.
  • Sentiment mix: 61% neutral, 27% positive, 12% negative, concentrated in Refunds.

Every number came from signal already flowing. The only new work was the enrichment and the tiles.

06 · Field notes

What to remember

📌 Important13.5 is a pipeline: traces + inference table → enrich (AI Functions for text, scorer monitors for quality) → one flat ua_request_metrics Delta table → an AI/BI dashboard. Read traces with mlflow.search_traces(experiment_ids=[...]).
💡 Tip · fieldTurn on 2–3 scorer monitors and the metrics job on launch day, not after the first incident — a quality trend is only useful with history. Keep the metrics table flat and push all business logic (percentiles, buckets, ratios) into the dataset SQL; AI/BI widgets are deliberately dumb.
⚠️ Gotchaexperiment_names= does not exist on search_traces — use experiment_ids. A scorer needs both .register() and .start(). Widgets can't run percentile_approx — alias it in the dataset SQL. 📘B1 Ch9 classifies intent with ai_query + JSON schema; prefer ai_classify for fixed labels. Native trace-to-UC sync is private preview; exact synced-trace and inference-table names depend on your 13.3/11.3 config — verify them.
