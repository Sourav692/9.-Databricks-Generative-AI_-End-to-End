Module 08 — Evaluating GenAI applications · Topic 08.1 · Theory ★

The Evaluation Harness

One call runs your app over a dataset, grades every row, and logs it all with lineage: mlflow.genai.evaluate(data=, predict_fn=, scorers=[...], model_id=). This is the architecture the whole evaluation module plugs into.

01 · The big idea

Evaluation is an experiment, not a string diff

A GenAI answer can be wrong for many reasons — bad retrieval, bad prompt, hallucination, a formatting step. Judging it means judging the whole execution, repeatably, with the result tied to a version. The Evaluation Harness makes that one function call.

  • One entry point: mlflow.genai.evaluate(data=, predict_fn=, scorers=[...], model_id=). It runs your app over the dataset, scores each row, aggregates, and logs an MLflow evaluation run (📘B1 Ch6).
  • Three inputs: data (the dataset — rows of inputs / outputs / expectations), predict_fn (a callable wrapping your app, e.g. the RAG chain's .invoke), and an explicit scorers list — 3.x auto-selects nothing.
  • Two outputs: per-row traces + feedback (assessments) and aggregate metrics (pass rates, means, distributions), all inside one run.
  • Wired for drill-down: the run links to the LoggedModel version (model_id, Module 06) and to traces (Module 07), so a low aggregate score leads straight to the failing span.
  • Never get wrong: there is no agents.evaluate(), and mlflow.evaluate(model_type="databricks-agent") is the former MLflow-2 path.

03 · How it works

What the harness writes, and why the links matter

The harness logs one evaluation run with four parts. Aggregate metrics tell you when something regressed; the linked traces tell you what went wrong on which row — that drill-down is the whole point.

Run Info + Traces + Feedback + Metrics + Metadata, all logged together and anchored to the LoggedModel (model_id).
  • Traces — one per dataset row: request, response, retrievals, intermediate steps, timings, token counts, tool calls (Module 07).
  • Feedback — the per-row assessments your scorers produce, attached to each trace (e.g. correctness: Fail, safety: Pass).
  • Metrics — aggregates computed from the feedback: pass rates, means, distributions (e.g. correctness_mean 0.18).
  • Metadata / params — dataset name + version, model/chain version, scorer definitions, environment.
  • The two modes. Pass predict_fn and the harness runs your live app per row (direct mode); pass precomputed outputs/traces instead and it just scores them (answer-sheet mode — good for regression tests and curated production traces). Same output either way.
  • Score the trace, not the text. Groundedness needs the retrieved context, safety reads the response, latency/cost come from span timings and tokens — all live in the trace (📘B1 Ch6 NOTE).
04 · Assemble it

The three inputs, then run

[Theory] Illustrative of the harness shape, not a runnable flow (that is 08.3–08.5). Assume MLflow ≥ 3.1, CATALOG="unity_airways", SCHEMA="rag", chain unity_airways.rag.ua_rag_chain.

1. Load the three inputs — the app, the dataset, the scorers:

import mlflow
from mlflow.genai.datasets import get_dataset
from mlflow.genai.scorers import Correctness, RelevanceToQuery, RetrievalGroundedness, Safety

model_uri = f"models:/{CATALOG}.{SCHEMA}.ua_rag_chain@champion"
ua_rag_chain = mlflow.langchain.load_model(model_uri)

eval_dataset = get_dataset(name=f"{CATALOG}.{SCHEMA}.eval_dataset")   # data=

ua_scorers = [Correctness(), RelevanceToQuery(), RetrievalGroundedness(), Safety()]  # explicit

2. Wrap the app in a predict_fn — the harness calls this per row (direct mode):

from typing import Dict

def rag_chain_predict_fn(question: str) -> Dict[str, str]:
    return ua_rag_chain.invoke({"messages": [{"role": "user", "content": question}]})

3. Name the version, then run the harness — results link to the LoggedModel + a run:

mlflow.set_active_model(name="ua_rag_chain_v1")   # anchor to a LoggedModel

with mlflow.start_run(run_name="ua_rag_eval_baseline"):
    mlflow.set_tag("model_version", "v1")
    eval_results = mlflow.genai.evaluate(
        data=eval_dataset,               # rows of inputs / (outputs) / (expectations)
        predict_fn=rag_chain_predict_fn, # direct mode: run the live chain per row
        scorers=ua_scorers,              # MUST be explicit — 3.x auto-selects nothing
    )
# -> one evaluation run: per-row traces + feedback, and aggregate metrics.

Verify it worked — one trace per dataset row (zero traces means the row schema does not match predict_fn — align column names/types); aggregate metrics per scorer; and you can open any row's trace to see retrievals + per-scorer feedback.

05 · Field notes

What to remember

📌 ImportantThe Evaluation Harness is one call — mlflow.genai.evaluate(data=, predict_fn=, scorers=[...], model_id=). data is the dataset (inputs/outputs/expectations), predict_fn wraps your app (the RAG chain's .invoke), scorers is an explicit list from three families (mlflow.genai.scorers built-ins, mlflow.genai.judges / make_judge(), and @scorer), and model_id anchors results to a LoggedModel version. There is no agents.evaluate().
💡 Tip · fieldDesign the predict_fn and the dataset schema together before you run anything. The most common failure is a run that finishes with zero traces because the row schema does not match what predict_fn expects. Validate one row locally (rag_chain_predict_fn(sample_question)) before evaluating 200.
⚠️ GotchaIn MLflow 3.x the harness scores the trace, not the raw answer string — groundedness needs the retrieved context, and latency/cost need span timings + token counts. Evaluate a bare string and you silently lose those dimensions. Also: the built-in scorer set and production monitoring evolve fast (monitoring is Beta) — confirm scorer names on current docs before a customer demo.
