Module 08 — Evaluating GenAI · Topic 08.4 · Theory + Hands-on ★

LLM-as-a-Judge

Exact-match can't grade open-ended answers, so you hand the grading to an LLM. The one split you must not get wrong: Correctness and RetrievalSufficiency need a ground-truth answer — the rest are reference-free.

01 · The big idea

Grade meaning, not tokens

Your Unity Airways RAG chain answers free text: refunds, rebooking, baggage. Two correct answers can share almost no words, so exact-match scores a perfect answer as wrong. An LLM judge reads the trace and scores for meaning — is the answer correct, relevant, safe, grounded? — and returns a verdict plus a rationale.

  • Judges are scorers backed by an LLM. Import from mlflow.genai.scorers, instantiate, and pass to mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]). Each returns a per-row Feedback logged to the trace.
  • The load-bearing fact. Correctness and RetrievalSufficiency need a ground-truth answer (expected_facts/expected_response) in each row's expectations. ExpectationsGuidelines needs per-row guidelines. Everything else is reference-free.
  • Custom judges for the bespoke stuff. Guidelines(...) turns a plain-English rule into a judge; make_judge(...) gives a full custom prompt with graded verdicts.
  • A judge is itself an LLM — it costs tokens, adds latency, and can disagree with a human. Calibrate it (08.8) and check it against human feedback (08.6) before trusting it in a quality gate.

Three buckets: reference-free (no labels), per-row criteria (ExpectationsGuidelines), and reference-based needing a factual answer (Correctness, RetrievalSufficiency). The one people forget is RetrievalSufficiency — it does need ground truth.

Context + criteria in, a scored Feedback with a rationale out. Because it is an LLM, it costs tokens and can be wrong — so you calibrate it.
03 · How it works

Which judges need ground truth

This is the crux of the topic. A judge is reference-free when it can score an answer with no labels — it only needs the question and the reply (and, for retrieval judges, the trace). A judge is reference-based when it must compare the answer against a labeled expected answer. Get this split wrong and either your Correctness scores are meaningless or you waste labeling effort on judges that never read a label.

  • Reference-free (no labels): RelevanceToQuery, Safety, RetrievalGroundedness, RetrievalRelevance, plain Guidelines. Run these on unlabeled data and on production traffic where no gold answers exist.
  • Needs per-row criteria (not a gold answer): ExpectationsGuidelines — the pass/fail rule is supplied per row inside expectations.guidelines.
  • Needs a ground-truth answer: Correctness and RetrievalSufficiency — each row's expectations must carry expected_facts or expected_response.
  • Ground truth lives in the row. Same dataset, one expectations block per row. Reference-based judges read it; reference-free judges ignore it. Rows without it are simply skipped by the reference-based judges — that skip is the visible proof of the requirement.
  • Read the rationale, not just the number. Every verdict is a Feedback with a written rationale on the trace. That sentence localizes the fault — off-topic, ungrounded, or a missed fact — which is what makes judges beat a bare accuracy number.
⚠️ GotchaMany summaries list only Correctness and ExpectationsGuidelines as "needing expectations" and forget RetrievalSufficiency. The live Databricks reference is explicit: RetrievalSufficiency requires ground truth because it checks whether retrieval fetched enough to support the expected facts. And ExpectationsGuidelines needs the expectations field — but populated with guidelines, not a factual answer.
04 · Run this

Import, run reference-free, then add labels

Prereqs: serverless or DBR ML runtime, MLflow ≥ 3.1 (≥ 3.4.0 for make_judge); the registered chain from 05.6 (unity_airways.rag.ua_rag_chain); a judge LLM endpoint (databricks-claude-sonnet-4-5). Retrieval judges need the chain to emit retrieval spans in its trace (Module 07).

%pip install -U "mlflow[databricks]>=3.4" databricks-langchain
dbutils.library.restartPython()

0. Imports, variables, and a pinned judge model:

import mlflow
from mlflow.genai.scorers import (
    Correctness, RelevanceToQuery, Safety,
    RetrievalGroundedness, RetrievalRelevance, RetrievalSufficiency,
    Guidelines, ExpectationsGuidelines,
)

CATALOG, SCHEMA = "unity_airways", "rag"
UC_MODEL   = f"{CATALOG}.{SCHEMA}.ua_rag_chain"
eval_model = "databricks:/databricks-claude-sonnet-4-5"   # pin the judge LLM
mlflow.set_registry_uri("databricks-uc")

1. Wire predict_fn to your Module 05 chain:

chain = mlflow.langchain.load_model(f"models:/{UC_MODEL}@champion")

def predict_fn(query: str) -> str:
    # inputs keys become predict_fn kwargs; here inputs = {"query": ...}
    return chain.invoke(query)

2. A small dataset — some rows with ground truth, some without:

eval_df = [
    {
        "inputs": {"query": "Can I get a refund on a Basic Economy fare?"},
        "expectations": {"expected_facts": [
            "Basic Economy fares are generally non-refundable.",
            "A full refund is available if cancelled within 24 hours of booking.",
        ]},
    },
    {"inputs": {"query": "My connection was missed — what are my options?"}},
    {"inputs": {"query": "How many carry-on bags are included?"}},
]

3. Reference-free judges — run on everything, no labels needed:

free_run = mlflow.genai.evaluate(
    data=eval_df, predict_fn=predict_fn,
    scorers=[
        RelevanceToQuery(model=eval_model),
        Safety(model=eval_model),
        RetrievalGroundedness(model=eval_model),
        RetrievalRelevance(model=eval_model),
        Guidelines(name="professional_tone",
                   guidelines="Professional, courteous airline-support tone; no slang.",
                   model=eval_model),
    ],
)

Verify: in the MLflow Evaluations UI (or free_run.metrics), every row has a Feedback per judge — a score and a readable rationale.

4. Reference-based judges — they only score rows that have expectations:

gt_run = mlflow.genai.evaluate(
    data=eval_df, predict_fn=predict_fn,
    scorers=[
        Correctness(model=eval_model),          # needs expected_facts / expected_response
        RetrievalSufficiency(model=eval_model), # needs expected_facts / expected_response
    ],
)
# The refund row gets a verdict; the unlabeled rows are skipped for these judges.

5. A custom judge for a bespoke rule:

from mlflow.genai.judges import make_judge   # requires MLflow >= 3.4.0

cites_policy = make_judge(
    name="cites_ua_policy",
    instructions=("Question: {{ inputs }}\nAnswer: {{ outputs }}\n"
                  "Does the answer cite a specific Unity Airways policy? "
                  "Categorize as 'cited', 'vague', or 'missing'."),
    model=eval_model,
)
custom_run = mlflow.genai.evaluate(data=eval_df, predict_fn=predict_fn,
                                   scorers=[cites_policy])

05 · Field notes

What to remember

📌 ImportantThe whole topic reduces to one fork and one habit. The fork: does this judge need a ground-truth answer? Correctness and RetrievalSufficiency do (expected_facts/expected_response); everything else in the core set is reference-free. The habit: read the rationale, because the sentence the judge writes is what turns a red number into a fix.
💡 Tip · fieldSplit your judges into two lists in code — a reference_free list you can run anywhere (dev, staging, and production monitoring) and a reference_based list you only run on labeled rows. It answers "do I need labels here?" structurally, and it mirrors how you reuse these scorers for production monitoring later.
⚠️ GotchaA judge is an LLM, so it has real cost and latency and can disagree with a human — especially near the pass/fail boundary. Pin the judge model, sample its verdicts against human labels (08.6), and calibrate borderline cases (08.8) before you gate a release on a judge score. make_judge needs MLflow ≥ 3.4.0; verify its exact kwargs against current docs.

Test yourself

Reveal 5-question self-check

  2. Name the two core built-in judges that need ground truth and the exact field their labels go in. Which retrieval judge is the one people forget belongs in this group?
  3. You have 500 unlabeled production traces. Which judges can you still run, and why don't they need expectations?
  4. When would you use Guidelines vs ExpectationsGuidelines vs make_judge? Give a one-line Unity Airways example for each.
  5. A judge scores a row fail. What single artifact do you read to find out why, and why is that better than a bare accuracy number?
