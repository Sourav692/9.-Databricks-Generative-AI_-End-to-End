Module 08 — Evaluating GenAI Applications · Topics 08.1–08.10 · Theory + Hands-on

GenAI Evaluation

You changed the retrieval, the prompt, or the model — did the Unity Airways assistant get better or worse? This module replaces "looks good to me" with evidence: a versioned dataset, a fixed set of scorers, and one call to mlflow.genai.evaluate(...) that scores the trace and links results to the app version.

01 · The big idea

From "seems fine" to a number you can defend

Module 05 built unity_airways.rag.ua_rag_chain and Module 07 made it observable. Module 08 makes every change decidable: run the app over a curated dataset, apply scorers, log per-row scores + aggregates with lineage to the LoggedModel version — then promote or discard on evidence.

  • Four building blocks — datasets (inputs/outputs/expectations), scorers (code + LLM judges), evaluation runs, and feedback. The Evaluation Harness ties them together (08.1 ★).
  • One entry point: mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]). Two modes: direct (predict_fn calls the app) and answer-sheet (precomputed outputs / existing traces).
  • Scorers are code-based (@scorer → Feedback) or LLM judges (RelevanceToQuery, Safety, Correctness, RetrievalGroundedness, …). Some judges need ground-truth expectations; many do not (08.4 ★).
  • Compare rag_chain vs the llm_only baseline on the same dataset; open the traces that flipped to diagnose (08.5). Needs MLflow ≥ 3.1 (make_judge ≥ 3.4).
📌 The rule that shapes this moduleIn MLflow 3.x you name the scorers, the harness never guesses. The entry point is mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]) — scorers=[...] is required. There is no agents.evaluate(), and mlflow.evaluate(model_type="databricks-agent") is the retired MLflow-2 path. It scores the trace, not just the string, so retrieval judges read the RETRIEVER span from Module 07.
02 · The problem

Why eyeballing and one accuracy number fail

A passenger asks "My flight UA217 was delayed in Tokyo — am I eligible for a refund?" You bump the retrieval k. Better or worse? Naive evaluation breaks:

  • Eyeball a few answers → not reproducible; two reviewers disagree and there's no record to point to.
  • One accuracy number → GenAI quality is multi-dimensional (grounded, relevant, safe, on-brand, fast, cheap); a single score hides the trade-offs.
  • Reuse mlflow.evaluate(model_type="databricks-agent") → superseded; the MLflow 3 path is mlflow.genai.evaluate(...) with explicit scorers=[...].
  • Score the text string → retrieval quality, groundedness, latency, and cost live in the trace. MLflow 3.x scores the trace so a groundedness judge reads the exact retrieved chunks.
  • Trust an LLM judge blindly → a judge is a model with opinions; uncalibrated it over/under-scores. Align it with human labels first (08.8).

04 · Cornerstone

08.1 ★ — the MLflow 3.x stack & the Evaluation Harness

Full deep-dive — the four building blocks, harness internals, direct vs answer-sheet modes, and "evaluate the trace, not the text" — in the Evaluation Harness deep-dive (eval-harness.md / eval-harness.html).

08.1 The evaluation stack Theory Cornerstone

MLflow 3.x makes evaluation a governed workflow of four building blocks: datasets (mlflow.genai.datasets), scorers (mlflow.genai.scorers + mlflow.genai.judges), evaluation runs (mlflow.genai.evaluate()), and feedback (assessments on traces). The Evaluation Harness automates supplying inputs, invoking the app, and computing metrics — the same logic runs in a notebook or a production job.

import mlflow

eval_results = mlflow.genai.evaluate(
    data=eval_dataset,                # inputs / outputs / expectations
    predict_fn=rag_chain_predict_fn,  # direct mode: harness invokes the app
    scorers=unity_airways_scorers,    # REQUIRED — you name them; no auto-select
)
📌 ImportantMLflow 3.x scores the trace. Groundedness and retrieval relevance need the retrieved context; safety/guidelines use the response + metadata; latency and cost come from span timings and tokens. Direct mode creates a fresh trace via predict_fn; answer-sheet mode uses existing traces.
05 · Datasets

08.2 — building & managing evaluation datasets

08.2 Evaluation datasets Theory Hands-on

Build from three sources: production/dev traces, expert-labeled cases, and curated edge cases. The schema is inputs (required), outputs (answer-sheet mode), and expectations (optional ground truth). Reserved expectations keys: expected_facts / expected_response (Correctness), guidelines (Guidelines), expected_retrieved_context (retrieval).

from mlflow.genai.datasets import create_dataset, get_dataset

CATALOG, SCHEMA = "unity_airways", "rag"
eval_dataset = create_dataset(name=f"{CATALOG}.{SCHEMA}.eval_dataset")

# every merge auto-creates a new version with UC lineage
eval_dataset = eval_dataset.merge_records([
  {"inputs": {"question": "My flight UA217 was delayed in Tokyo. Refund?"},
   "expectations": {"expected_facts": ["Refund depends on fare type",
                                       "Waiver WX-2025 applies only to affected segment"]}},
])
⚠️ Gotcha · verify liveSDK limits: max 2,000 records, ≤ 20 expectation fields per entry, no CMK yet. Prefer archiving over hard-delete — a hard purge leaves the lineage reference but you can't resolve it back to the table. (Exact current limits: live re-check pending.)
💡 TipKeep the seed set small on purpose so it's cheap to run frequently and catches regressions early. Grow it as feedback arrives; retire cases that no longer reflect policy.
06 · Scorers

08.3 — code-based scorers

08.3 Code-based scorers Hands-on

A scorer turns a raw prediction into a measurable Feedback (value + rationale). Code-based scorers are @scorer-decorated Python functions — deterministic, no token cost, ideal for continuous checks: schema validity, exact match, phone/date presence, flight-code format, business rules.

from mlflow.genai.scorers import scorer
from mlflow.entities import Feedback

@scorer(name="response_length")
def ua_response_length_scorer(outputs) -> Feedback:
    response = outputs if isinstance(outputs, str) else outputs.get("response", "")
    n = len(str(response).split())
    if n < 5:   return Feedback(value=0.0, rationale=f"Too short ({n} words)")
    if n > 120: return Feedback(value=0.5, rationale=f"Quite long ({n} words)")
    return Feedback(value=1.0, rationale=f"Appropriate length ({n} words)")

Design principles: single point of truth (one scorer, one thing); stable interface (deterministic, normalize text, parse dates); interpretable 0.0–1.0 scale; deterministic first, judgment second — "a regex should not be replaced by an LLM to find a phone number."

💡 TipCode-based scorers can emit operational signals too — read span timings/tokens from the trace and report a normalized latency or cost value that feeds SLO dashboards.
07 · Cornerstone

08.4 ★ — LLM-as-a-Judge & which judges need ground truth

Full deep-dive — built-in judges, Guidelines vs ExpectationsGuidelines, custom make_judge(), model pinning, and the ground-truth decision — in the LLM-as-a-Judge deep-dive (llm-as-judge.md / llm-as-judge.html).

08.4 LLM judges Theory Hands-on Cornerstone

LLM judges grade nuance — relevance, safety, correctness — reading the full trace and returning a score + rationale. Built-ins import from mlflow.genai.scorers; pass model= to pin a judge model. For full control, build a custom judge with make_judge().

from mlflow.genai.scorers import (
    RelevanceToQuery, Safety, RetrievalGroundedness, Correctness, Guidelines)

eval_model = "databricks:/databricks-claude-sonnet-4-5"   # override + pin for stability
ua_relevancy    = RelevanceToQuery(model=eval_model)
ua_safety       = Safety(model=eval_model)
ua_groundedness = RetrievalGroundedness(model=eval_model)   # hallucination signal
ua_correctness  = Correctness(model=eval_model)             # needs expectations
ua_tone = Guidelines(name="professional_tone",
    guidelines="Professional airline-support tone; reference 'Unity Airways'.",
    model=eval_model)

from mlflow.genai.judges import make_judge   # custom judge (MLflow >= 3.4)
ua_coherence = make_judge(name="coherence", instructions=(
    "Evaluate coherence and policy-adherence.\n"
    "Question: {{ inputs }}\nResponse: {{ outputs }}\n"
    "Categorize as 'coherent', 'somewhat coherent', or 'incoherent'."))
📌 Important · the ground-truth split (three buckets)(1) Needs a ground-truth answer (expected_facts / expected_response): Correctness and RetrievalSufficiency — the retrieval judge people forget requires ground truth. (2) Needs per-row guidelines (criteria, not a factual answer): ExpectationsGuidelines. (3) Reference-free (no labels): RelevanceToQuery, Safety, RetrievalGroundedness, RetrievalRelevance, plain Guidelines. So run reference-free judges on every trace; invest in an expected_facts/expected_response answer key only where Correctness and RetrievalSufficiency matter most.
⚠️ GotchaJudges default to Databricks-hosted LLMs — pass model= and pin it. make_judge() needs MLflow ≥ 3.4 (older used deprecated custom_prompt_judge). Don't use MLflow-2 names (groundedness, chunk_relevance, …).
08 · Run & compare

08.5 — running, comparing, diagnosing with traces

08.5 Evaluation runs Hands-on

An evaluation run binds one dataset version + one candidate + one scorer set, logging traces (one per row), feedback, aggregate metrics, and metadata. Load the Module 05 chain, wrap it in a predict_fn, and run.

loaded_chain = mlflow.langchain.load_model(
    f"models:/{CATALOG}.{SCHEMA}.ua_rag_chain/{MODEL_VERSION}")

def rag_chain_predict_fn(question: str) -> dict:
    return loaded_chain.invoke({"messages": [{"role": "user", "content": question}]})

with mlflow.start_run(run_name="ua_rag_eval"):
    mlflow.set_tag("candidate", "rag_chain")   # tag so runs stay comparable
    res_rag = mlflow.genai.evaluate(
        data=eval_dataset, predict_fn=rag_chain_predict_fn,
        scorers=unity_airways_scorers)

Compare rag_chain vs llm_only: re-run the same dataset + scorers against the Module 06 llm_only baseline, tag candidate=llm_only. Rule: fix the dataset, pin versions, change one variable per run, tag it. In the UI filter by tag, select both runs, read the headline metrics — then open the traces that flipped.

💡 Tip · diagnoseZero traces → data schema vs predict_fn mismatch. Scorer crashes → defensive checks + neutral value. Output shape varies → normalize in a post-processor. Slow → inspect span timings (retrieval vs judge calls). Layer on difficulty bands, bootstrap CIs, and a persisted failure table.
09 · Human loop

08.6 — capturing human-in-the-loop feedback

08.6 Human feedback Theory Hands-on

Scorers miss nuance — tone, empathy, subtle gaps. MLflow stores human judgment as an Assessment on a trace: Feedback (what the app produced) and Expectations (what correct looks like). Every Assessment records a source (human / judge / code).

from mlflow.entities.assessment import AssessmentSource, AssessmentSourceType

mlflow.log_feedback(
    trace_id=trace_id, name="user_feedback", value=satisfied, rationale=note,
    source=AssessmentSource(source_type=AssessmentSourceType.HUMAN, source_id=user_id))

# Domain-expert review at scale — Labeling Sessions (Review App)
from mlflow.genai.labeling import create_labeling_session
session = create_labeling_session(name="refund_policy_review",
    label_schemas=[accuracy_schema.name, ideal_schema.name])
session.add_traces(mlflow.search_traces(
    filter_string="assessments.user_feedback = false", max_results=20))
print("Share with experts:", session.url)
⚠️ GotchaTreat user comments as personal data — if policy forbids raw text, store a category code + hashed session id. Developer feedback is opinionated; treat it as advisory until a designated reviewer signs off.
10 · Rubrics & calibration

08.7 + 08.8 — qualitative rubrics and calibrating metrics

08.7 Qualitative rubrics Theory

Quality, safety, and hallucination are judged against rubrics an LLM judge applies consistently. Quality → a Guidelines judge (tone, completeness). Safety → the Safety judge + domain non-negotiables ("never invent a fee waiver"). Hallucination → this is groundedness: RetrievalGroundedness checks the answer against retrieved chunks; a low score is your hallucination signal. Book 2's RAG dimensions: relevance, grounding (reduces hallucination), consistency, retrieval quality — all application-driven.

08.8 Calibrating metrics Theory Hands-on

A judge is a model with opinions; a threshold is a product decision. Calibrate both against human labels. For a threshold: label ~100 past answers, plot, set bands (the book lands on Pass = 40–110 words, Review = 25–39 / 111–140, Fail = <25 / >140) — recorded with the scorer version, validated per slice. For a judge (groundedness): compare judge scores to human labels, measure agreement, refine the rubric / pin the model / move the threshold, re-check.

📌 ImportantA rubric is only as good as its wording. Vague criteria ("be helpful") produce noisy scores. Write it like a brief for a new support agent: concrete, testable, tied to a real outcome. "Calibrate with small human-reviewed samples so you don't over-trust a single model's opinion."
11 · Discipline

08.9 — evidence-driven development & the MEP

08.9 Minimum Evaluable Product Theory

The Develop phase ends not at the first green check but when one realistic path runs end to end and leaves the scaffolding to measure the next change. The MEP is the smallest slice that is both useful and measurable — one representative journey plus the scorers to judge future changes against it.

  • For Unity Airways: answer "Can I change my flight tomorrow?" — one booking-lookup tool, one citation step, one grounded response. Trace it, export a few requests, attach two scorers: structure validity + groundedness with citation.
  • Every run tests a claim you can say out loud: "tightening the refund prompt raises groundedness on refund intents while leaving helpfulness flat."
  • The loop: hypothesis → representative scenarios → inspect traces → evaluate on a fixed seed set → compare to a baseline → promote or discard on evidence. Set stop-the-line thresholds (safety violations, schema breakage, key-scorer drops).
📌 ImportantEvidence-driven development is a culture, not a tool. Teams that adopt the MEP stop arguing from anecdotes and start pointing to trace evidence, controlled comparisons, and domain-aligned metrics. The goal: a loop that is "intentionally boring."
12 · Traditional metrics

08.10 — BLEU, ROUGE, perplexity, exact-match (vs judges)

08.10 Traditional text-eval metrics Theory

Reference-based NLG metrics are cheap, deterministic, and comparable — but blind to grounding and hallucination.

MetricMeasuresBlind to
ROUGERecall n-gram overlap w/ reference (summarization)Meaning; paraphrase
BLEUPrecision n-gram overlap (translation)Meaning; paraphrase
BERTScoreSemantic similarity via embeddingsFactual grounding
PerplexityHow well an LM predicts text (lower better)Task correctness; grounding
Exact-matchBinary string equality (extractive QA)Anything phrased differently

Prefer LLM judges for open-ended RAG answers where "correct" can be phrased many ways and grounding/safety matter. Log traditional metrics like any other: mlflow.log_metric("rouge_score", 0.78). The exam point: MLflow enables consistent tracking + comparison of any metric.

⚠️ GotchaA high BLEU/ROUGE against one reference can still be a hallucination if the reply invents a policy the reference happened to phrase similarly. Use these as a cheap first filter, never the promotion gate for a policy-answering RAG app.
13 · Worked thread

Unity Airways — promoting a retrieval change with evidence

  1. Hypothesis (08.9): raising k 3→5 lifts groundedness on refund intents without hurting relevance or latency.
  2. Dataset (08.2): 40 rows from Module 07 traces + expert-labeled refund cases with expected_facts, at unity_airways.rag.eval_dataset.
  3. Scorers (08.3–08.4): RelevanceToQuery, Safety, RetrievalGroundedness, Correctness + code-based response_length + a Guidelines tone judge — passed explicitly.
  4. Baseline run: mlflow.genai.evaluate(...) on ua_rag_chain; tag candidate=rag_chain, k=3.
  5. Compare vs llm_only (08.5): same dataset + scorers on the Module 06 llm_only baseline — groundedness collapses (no context to ground in). Retrieval earns its keep.
  6. Candidate run: re-run with k=5 — groundedness 0.71 → 0.89, relevance flat, latency +40ms (in envelope).
  7. Diagnose the flip (08.5): open the one regressed row's trace — RETRIEVER pulled a baggage chunk. Add to the failure table.
  8. Calibrate (08.8): confirm the groundedness judge agrees with human labels on the refund slice before trusting 0.89.
  9. Promote: hypothesis held — promote the k=5 LoggedModel version, linked to dataset + runs + traces.
  10. Monitor (Module 13): the same scorers now watch production; a groundedness drop alerts before a customer complains.
14 · Field notes

What to remember

📌 ImportantEntry point mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]) — scorers is required, no auto-select, no agents.evaluate(). Scorers = code-based (@scorer→Feedback) + LLM judges (RelevanceToQuery, Safety, Correctness, RetrievalGroundedness, …). Dataset fields inputs/outputs/expectations. It scores the trace.
💡 Tip · fieldFix the dataset, pin versions, change one variable, tag the run. Reference-free judges everywhere; expectations only where correctness matters. Calibrate a judge against human labels before trusting its number. The same scorers you register here monitor production (Module 13) — build the instrument once.
⚠️ GotchaDon't use mlflow.evaluate(model_type="databricks-agent") or MLflow-2 judge names. Don't expect Correctness without expectations. Don't trust an uncalibrated judge. Don't treat BLEU/ROUGE as the promotion gate for a RAG policy assistant — they miss hallucination.
