Chunking Strategies

Splitting documents into retrievable units is the first, mostly irreversible quality lever in RAG. Match the strategy to the document, add overlap where meaning spans boundaries, and size chunks against two context windows.

01 · The big idea

The retriever can only return what you chunked

Unity Airways wants an assistant that answers "Can I get a refund on a Basic Economy fare?" from its own policy PDFs. You cannot paste 400 pages into the prompt, so RAG retrieves the relevant pieces. But retrieval can only ever return the pieces you decided to create when you chunked. Split a refund rule in half and the model gets half a rule.

  • Chunking divides documents into small units that are embedded, indexed, and retrieved on their own. It happens before embeddings and caps everything downstream.
  • Five strategies matter: fixed-length, sentence-level, paragraph-based, sliding-window (overlap), and semantic.
  • Two dials: granularity (chunk size) and overlap (shared text between neighbors). Smaller raises precision and can drop recall; overlap protects boundary-spanning ideas at a storage cost.
  • Size against two ceilings: the embedding window per chunk (gte-large-en = 8192 tokens) and the generation window for k chunks plus prompt.

03 · How it works

Granularity, overlap, and two context windows

Pick a strategy by the shape of the document. A real corpus almost always needs more than one.

There is no universal best strategy. Match it to the document type.
  • Granularity is a continuum. As chunk size grows, retrieval precision tends to fall while context coverage rises and per-query latency drops. Medium chunks balance all three and are the usual default.
  • Overlap protects boundary-spanning ideas. Fixed overlap repeats a set number of tokens (a 100-token window with a stride of 80 repeats 20 tokens). Dynamic overlap is content-aware, and in modern practice you get it from semantic splitting rather than literal token repetition.
  • Ceiling one — the embedding window. A chunk longer than databricks-gte-large-en's 8192 tokens is silently truncated at embed time; its tail never gets represented. databricks-bge-large-en caps at 512, which forces smaller chunks.
  • But the ceiling is not the target. Oversized chunks cause semantic dilution — one vector blurs several ideas and precision drops. Target roughly 200–500 tokens with a wide margin below the ceiling.
  • Ceiling two — the generation window. Retrieving k = 5 chunks of 500 tokens is 2500 tokens before the system prompt, question, and answer room. That sum must fit the generation model. Chunk size, overlap, and k are tuned together.
04 · Run this

Split, size-check, write to Delta, index

Prereqs: serverless or DBR ML runtime; a Unity Catalog CATALOG/SCHEMA you can write to; a vector search endpoint. Splitters come from the open-source langchain-text-splitters package; DatabricksEmbeddings comes from databricks-langchain; the index uses the databricks-vectorsearch SDK.

%pip install -U langchain-text-splitters langchain-experimental \
    databricks-vectorsearch databricks-langchain tiktoken
dbutils.library.restartPython()

Recursive splitting with overlap — tries paragraph, then line, sentence, word; chunk_overlap is the sliding window:

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1200,     # chars (~300 English tokens at ~4 chars/token)
    chunk_overlap=200,   # overlap carries context across boundaries
    separators=["\n\n", "\n", ". ", " ", ""],
)
chunks = splitter.split_text(policy_text)

Size-check against the 8192 ceiling (this is how you tie chunk size to the embedding model):

import tiktoken
enc = tiktoken.get_encoding("cl100k_base")   # close proxy for budgeting
lengths = [len(enc.encode(c)) for c in chunks]
print("max:", max(lengths), "avg:", round(sum(lengths)/len(lengths)))
assert max(lengths) < 8192, "chunk will be truncated at embed time"

Semantic chunking with Databricks-hosted embeddings (no external key):

from langchain_experimental.text_splitter import SemanticChunker
from databricks_langchain import DatabricksEmbeddings

embeddings = DatabricksEmbeddings(endpoint="databricks-gte-large-en")
semantic = SemanticChunker(embeddings, breakpoint_threshold_type="percentile")
docs = semantic.create_documents([long_transcript])

Write chunks to a Delta table, then build the AI Search index:

from pyspark.sql import functions as F
from databricks.vector_search.client import VectorSearchClient

rows = [("conditions_of_carriage", i, c) for i, c in enumerate(chunks)]
df = (spark.createDataFrame(rows, ["doc_id", "chunk_index", "chunk_text"])
        .withColumn("chunk_id",
            F.sha2(F.concat_ws("::", "doc_id", F.col("chunk_index").cast("string")), 256))
        .withColumn("source", F.lit("conditions_of_carriage")))

table = f"{CATALOG}.{SCHEMA}.policy_chunks"
df.write.mode("overwrite").saveAsTable(table)
spark.sql(f"ALTER TABLE {table} SET TBLPROPERTIES (delta.enableChangeDataFeed = true)")

vsc = VectorSearchClient()
vsc.create_delta_sync_index(
    endpoint_name="unity-airways-vs",
    index_name=f"{CATALOG}.{SCHEMA}.policy_chunks_index",
    source_table_name=table,
    primary_key="chunk_id",
    embedding_source_column="chunk_text",           # this column gets embedded
    embedding_model_endpoint_name="databricks-gte-large-en",
    pipeline_type="TRIGGERED",
)

How to verify it worked: after the size-check, confirm max is well under 8192. After indexing, run a similarity_search for a real question and read the top chunk: if it contains the whole refund rule (condition included), size and overlap are right. If the condition is missing, add overlap or move to a coarser, boundary-aware strategy for that document type.

05 · Field notes

What to remember

📌 ImportantChunking is the ceiling on RAG quality. The retriever can only return units you create here, so no reranker, prompt, or bigger model can recover an idea you split in half. Fix retrieval at the chunking step first.
💡 Tip · fieldStart medium — around 200–300 tokens with 10–20% overlap — then measure retrieval with precision, recall, or MRR and adjust. Smaller (100–200) for FAQ facts; larger (300–500) for contracts and policies. Match the dial to the document, not to a global default.
⚠️ GotchaThe book's Example 3-1 (fixed-length) imports from langchain.text_splitter import CharacterTextSplitter; current LangChain moved splitters to langchain_text_splitters. Its semantic examples (3-2 and 3-4) use a different library — LlamaIndex (llama_index's SemanticSplitterNodeParser) with OpenAIEmbedding(). On Databricks, swap the embedder for DatabricksEmbeddings(endpoint="databricks-gte-large-en") from databricks-langchain. Books lag the product; verify imports against current docs.

Test yourself
