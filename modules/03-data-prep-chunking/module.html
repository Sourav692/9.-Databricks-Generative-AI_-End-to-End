Data Prep and Chunking for RAG

RAG quality is decided before a single vector exists. This module turns raw, messy documents into small, clean, well-labeled chunks in a query-ready Delta table — all on one use case: Unity Airways customer support.

01 · The big idea

Get the chunks right first

Chunking, cleaning, and structuring the source data set the ceiling on everything downstream. A retriever can only return a chunk that exists — and only help if that chunk is the right shape and free of noise.

  • Chunking breaks big documents into small, self-contained units. The strategy (fixed, sentence, paragraph, sliding-window, semantic) and the size + overlap are the two biggest levers.
  • Real docs are messy: scanned PDFs, boilerplate headers, cookie banners. You extract text with the right library, then filter the noise before chunking.
  • The output is an embed-ready Delta table — one row per chunk, plus metadata — which becomes the source for the Module 04 vector index.
  • On Databricks you can do most of this SQL-natively with AI Functions (ai_parse_document, ai_extract) and run the whole ingestion with Lakeflow Declarative Pipelines (built on the open-source Spark Declarative Pipelines framework, SDP).
02 · The problem

Why "just embed the documents" fails

Unity Airways loads 4,000 policy PDFs, scanned refund forms, and an FAQ export into a RAG chatbot. In the demo it nails "what's the checked-bag fee?" In production, "Can I get a refund if I miss my connection?" returns a 12-page fare-rules chunk that buries the answer, or a chunk that stops mid-sentence. Retrieval is "working" — but the chunks are the wrong shape.

  • Embed the whole document → overflows the context window; retrieval returns the document, not the passage.
  • Split every N characters → cuts definitions and procedures in half. "Step 2: Check the" is useless alone.
  • Feed raw PDF text → garbled OCR and repeated headers/footers dominate similarity search.
  • One chunk size fits all → tiny = precise but blinkered; huge = full but noisy. There's a trade-off to tune, not guess.

04 · From raw docs to clean chunks

03.1–03.5 — ingest, extract, filter, chunk

03.1 RAG pipeline overview Theory

A RAG system has distinct stages: ingestion → content preparation (clean + chunk) → embedding → vector retrieval → response generation. Chunking happens in content preparation, before embeddings, so it has a cascading effect on retrieval accuracy, latency, and answer quality.

03.2 ★ Chunking strategies Theory + Hands-on Cornerstone

Five strategies, trading simplicity against meaning. Full walkthrough in the Chunking strategies deep-dive.

StrategyBest forWatch out
Fixed-lengthLogs, structured dataBreaks context mid-thought
Sentence-levelArticles, FAQsLoses broader context
Paragraph-basedReports, blogsSizes vary a lot
Sliding-windowQA, legal/technicalMore chunks to store
SemanticBooks, transcriptsComplex; needs NLP

03.3 Overlap and granularity Theory

  • Granularity (chunk size): too large → distracting noise; too small → missing context. Mind the model context window.
  • Overlap: repeating a slice of one chunk in the next preserves meaning that spans a boundary. Fixed (repeat N tokens) or dynamic (content-aware / semantic).
  • Retrieval trade-off: smaller → higher precision, lower recall, more vectors; larger → more coverage, more noise. Medium is the common default.

03.4 Content extraction from PDFs and images Hands-on

Get clean text out first. The production pattern is layered: text-based parsing first, OCR fallback only when the text is empty.

import pdfplumber, pytesseract
from PIL import Image

def extract_text_from_pdf(pdf_path):
    text = ""
    with pdfplumber.open(pdf_path) as pdf:
        for page in pdf.pages:
            page_text = page.extract_text()
            if page_text:                      # digital PDF: keep the clean text
                text += page_text + "\n"
        if not text.strip():                   # scanned PDF: no text -> OCR
            text = "\n".join(
                pytesseract.image_to_string(page.to_image().original)
                for page in pdf.pages)
    return text

03.5 Content filtering — noise and redundancy Theory + Hands-on

Remove redundancy (boilerplate, repeated headers, disclaimers) and noise (page numbers, HTML tags, timestamps, nav menus). If every article carries the same nav menu, it appears in nearly every chunk and precision collapses.

import re
def clean_text(text):
    text = re.sub(r'<[^>]+>', '', text)               # strip HTML tags
    text = re.sub(r'Page \d+ of \d+', '', text)                # page numbers
    text = re.sub(r'Last updated: .*?\n', '', text)            # timestamps
    return text

03.6 Converting to Delta for querying Hands-on

Each chunk becomes a row with metadata (source doc, chunk index, timestamp). Delta gives query efficiency, incremental upserts, auditability/time-travel, and is the source AI Search syncs from.

from pyspark.sql.functions import current_timestamp, lit

df = (spark.createDataFrame([(i, c) for i, c in enumerate(chunks)],
                            ["chunk_id", "content"])
        .withColumn("source_doc", lit("unity_airways_policy.pdf"))
        .withColumn("ingested_at", current_timestamp()))

# Governed UC managed table, not a DBFS mount
df.write.format("delta").mode("overwrite").saveAsTable(f"{CATALOG}.{SCHEMA}.ua_rag_chunks")

03.7 Evaluating retrieval quality + corrective techniques Theory + Hands-on

Measure against a labeled validation set. Each metric points to a specific fix.

Metric lowLikely causeCorrective action
PrecisionNoisy / too-broad chunksBetter filtering; reranking; tighter chunks
RecallChunks too small; coverage gapsAdjust size; improve ingestion; align embeddings
MRRPoor ranking of top resultsTune ranking; cross-encoder reranking (04.9)

Corrective levers: revisit chunking (semantic/overlap), align/fine-tune embeddings, fix indexing so tables/figures aren't dropped, query expansion/rewriting, and hybrid search (BM25 + dense) as a safety net. Evaluate offline before deploying.

03.8 ★ Parsing with AI Functions Theory + Hands-on Cornerstone

-- Parse Unity Airways policy PDFs straight from a UC Volume
SELECT path, ai_parse_document(content) AS parsed
FROM READ_FILES('/Volumes/unity_airways/rag/landing/policies/', format => 'binaryFile');

  • ai_extract — pull named fields (booking_reference, refund_amount) with a schema. GA.
  • ai_prep_search — parsed output → search-ready chunks for AI Search. Beta.

03.9 RAG ingestion as a Lakeflow Declarative Pipeline Hands-on

Run the whole ingestion as a declarative, incremental medallion pipeline (Lakeflow Declarative Pipelines is the successor to DLT).

CREATE OR REFRESH STREAMING TABLE ua_docs_bronze
AS SELECT *, _metadata.file_path AS source_path, current_timestamp() AS ingested_at
FROM STREAM READ_FILES('/Volumes/unity_airways/rag/landing/', format => 'binaryFile');

CREATE OR REFRESH STREAMING TABLE ua_docs_parsed
AS SELECT source_path, ai_parse_document(content) AS parsed, ingested_at
FROM STREAM(ua_docs_bronze);
06 · Worked thread

Unity Airways, end to end

  1. Ingest (03.1, 03.9): PDFs, scanned refund forms, FAQ export land in a UC Volume; a Lakeflow Declarative Pipelines bronze table streams them in with Auto Loader.
  2. Parse/extract (03.4, 03.8): typed PDFs parse cleanly; scanned forms need OCR — or one ai_parse_document call handles both.
  3. Filter (03.5): strip nav, "last updated" lines, page numbers so they don't pollute every chunk.
  4. Chunk (03.2, 03.3): FAQ → sentence/paragraph chunks; dense fare-rules → paragraph chunks with sliding-window overlap. Start ~200 tokens.
  5. Store (03.6): one row per chunk in unity_airways.rag.ua_rag_chunks with source + chunk_id + timestamp.
  6. Index + evaluate (03.7 → Module 04): AI Search Delta Sync index on the table; run a labeled set; low precision on "miss my connection → refund?" traces back to noisy/coarse chunks — tune and re-measure.
07 · Field notes

What to remember

📌 ImportantRetrieval can only return a chunk that exists. If a fact is split across two chunks or buried in a noisy one, no downstream embedding-model or LLM tuning fully recovers it. Fix the shape of the data first. For Lakeflow Declarative Pipelines use CREATE OR REFRESH STREAMING TABLE (not CREATE OR REPLACE) and from pyspark import pipelines as dp (not import dlt).
💡 Tip · fieldStart chunks at 150–300 tokens with small overlap, then tune with precision/recall/MRR. Always eyeball source docs before writing noise-removal rules, and include metadata columns (source doc, chunk number, URI) for debugging and metadata-filtered retrieval.
⚠️ GotchaThe book writes chunks to a DBFS mount (/mnt/delta/...) and wires semantic chunking to OpenAI — on current Databricks, use a UC managed table + Volume and the in-platform databricks-gte-large-en embeddings. And the two live docs disagree on ai_parse_document's minimum runtime (overview says DBR 18.2+/serverless v3; the function reference says DBR 17.3+) — verify for your workspace.
