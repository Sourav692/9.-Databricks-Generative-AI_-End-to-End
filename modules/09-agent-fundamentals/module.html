Module 09 — Agent Fundamentals and Tools (Mosaic AI Agent Framework) · Topics 09.1–09.11 · Theory + Hands-on

AI Agents and Tools

The Unity Airways assistant grows up. The Module 05 RAG chain becomes one tool among several, and an LLM decides when to retrieve policy, look up a booking, or call an API. You wrap it with ResponsesAgent, register it to unity_airways.rag.ua_support_agent, and ship it with one agents.deploy(...) call.

01 · The big idea

From a fixed chain to a system that decides

Module 05 built unity_airways.rag.ua_rag_chain — it always retrieves, then answers. An agent hands the wheel to an LLM: on each turn it decides whether to retrieve, look up a booking, or call an API, and loops until it can reply. Same MLflow lifecycle, tools in the middle.

  • Three levels of autonomy (09.1) — a traditional agent follows if-else rules; an AI agent lets an LLM pick tools; an agentic AI system routes between specialist agents. More flexible means less predictable.
  • Tools are how it acts (09.3 ★): a VectorSearchRetrieverTool (retrieval), a UC CREATE FUNCTION via UCFunctionToolkit (structured lookup), and a StructuredTool (API calls) — all from databricks-langchain.
  • Package once, govern everywhere (09.6 ★): wrap in ResponsesAgent, mlflow.models.set_model(), register to Unity Catalog, then agents.deploy(...) → Serving endpoint + Review App.
  • MCP standardizes tools across agents (09.8); managed MCP servers (Public Preview) expose Vector Search, UC functions, and Genie behind one UC-governed URL (09.11).
📌 The rule that shapes this moduleAuthor with ResponsesAgent, the recommended MLflow interface (OpenAI Responses schema; auto-tracing, streaming, multi-agent history). ChatAgent is being superseded; ChatModel is legacy. Migration path: ChatModel → ChatAgent → ResponsesAgent. SplitChatMessagesRequest/StringResponse are deprecated. Log from code with mlflow.models.set_model(); tools via UCFunctionToolkit; deploy with agents.deploy → endpoint + Review App + feedback model.
02 · The problem

Why the RAG chain and more if-else branches both fail

A passenger asks "My flight UA217 to Tokyo is delayed — can I make my connection, and is my booking refundable?" That needs policy (retrieval), the live booking record (a lookup), and maybe a status API — chosen on the fly. Naive fixes break:

  • The Module 05 chain → always retrieves, has no path to the booking table; "wouldn't be able to answer… as it is not connected to flight data" (📘B1 Ch7).
  • More if-else branches → the traditional-agent era; brittle rules that snap when a user rephrases, and a maintenance nightmare as intents grow.
  • Answer from LLM memory → no grounding; it invents a refund policy or a flight time. Tools exist so the agent acts on real data.
  • Vague/overlapping tool descriptions → the LLM picks the wrong tool; "having similar tools can confuse the supervisor agent about which one to pick."
  • Pickle the LangGraph object → the plain LangChain flavor "would need a lot of input/output custom functions" for deployment. ResponsesAgent handles the contract.

04 · Definitions

09.1 + 09.2 — agents vs AI agents vs agentic AI, and the lifecycle

09.1 Agents vs AI Agents vs Agentic AI Theory

Three terms, one axis — how much the system decides for itself (📘B1 Ch7, Fig 7-1).

LevelWho decides the flowUnity Airways example
Traditional agentHardcoded if-else rules and menus"Press 1 for schedule, 2 for reservation" → fixed query
AI agentAn LLM reads free text and picks a tool"Flight schedule for Singapore to Japan today?" → LLM calls schedule tool
Agentic AIA supervisor routes between specialist agents"Best flight for Singapore to Japan?" → delegates to Flight Recommender

Agents vs chains (Table 7-1): agents have dynamic control flow, higher autonomy and adaptability, but lower consistency and higher latency (10 seconds to minutes, since the supervisor can loop). Chains are deterministic, consistent, fast — better for predictable tasks.

09.2 Agent development lifecycle Theory

Same MLflow lifecycle as any GenAI app. Pick an orchestration framework (MLflow 3.x integrates LangChain, LangGraph, DSPy, LlamaIndex; the book uses LangGraph + databricks-langchain), iterate with tracing and the Prompt Registry, evaluate every layer with mlflow.genai.evaluate(), and swap models via the AI Gateway. Start small: "start with a small, well-defined toolset… later iterate."

⚠️ GotchaMore autonomy means less predictability — agents "can become less controlled or predictable, potentially allowing overly permissive behaviors. It is crucial to design appropriate guardrails." Pick the simplest pattern that solves the task; a chain if a chain will do.

09.3 ★ — creating tools: retriever, structured lookup, API

09.3 Creating tools Hands-on Cornerstone

Tools let the agent "do more than just respond… using knowledge that [the] LLM was trained on." The Unity Airways support agent needs three, all from databricks_langchain.

from databricks_langchain import VectorSearchRetrieverTool, UCFunctionToolkit
from unitycatalog.ai.core.databricks import DatabricksFunctionClient

# 1) vector retriever — wrap the Module 05 AI Search RAG chunks index
vector_retriever_tool = VectorSearchRetrieverTool(
    index_name="unity_airways.rag.ua_rag_chunks_index", num_results=5,
    tool_description="Search Unity Airways FAQ: cancellation, policies, baggage.")

# 2) structured-data lookup — a UC CREATE FUNCTION, wrapped as a tool
client = DatabricksFunctionClient()
toolkit = UCFunctionToolkit(
    function_names=["unity_airways.rag.get_flight_status"], client=client)

# 3) API tool — a StructuredTool with a Pydantic input schema (09.4)
tools = list(toolkit.tools) + [vector_retriever_tool, weather_tool]
💡 TipThe tool_description (and SQL COMMENT) is the agent's instruction manual — write it like a brief for a new rep: what it does, when to call it, key keywords. Vague or overlapping descriptions are the #1 cause of wrong-tool bugs.
06 · Safety

09.4 + 09.5 — tool governance, and multi-stage reasoning

09.4 Tool design for safety and governance Theory

A tool is executable and governed; a prompt instruction is advisory. A UC function runs under Unity Catalog permissions and a WHERE clause the LLM cannot cross — a boundary, not a suggestion. Constrain inputs with a structured schema ("secure the input type… especially when using LLM as they could hallucinate"), handle failure inside the tool, and keep the toolset small and non-overlapping.

09.5 Multi-stage reasoning and tool ordering; planning agent chains Theory Hands-on

Break a task into focused stages — extract → reason → respond (📗B2 Ch2). Two implementations: tool ordering (fixed sequence — SimpleSequentialChain/SequentialChain/RouterChain) for known steps, and agent planning (dynamic — ReAct, plan-and-execute) when "you can't predict how many steps are needed or which tools to use." Fig 2-3 contrasts the two; the LangGraph loop (should_continue routes back to tools or ends) is the planning case.

📌 ImportantAgentic systems are less controlled than chains, so safety is designed, not assumed. Restrict the tool surface, validate inputs, scope UC grants to the agent's service principal, and add AI Gateway guardrails at serving time (Module 11/12). A tool the agent cannot misuse beats a prompt asking it not to. Default to tool ordering; reach for a planning agent only when the path truly varies — and cap the loop.

09.6 ★ — packaging with ResponsesAgent, register in UC

09.6 Packaging with ResponsesAgent Hands-on Cornerstone

ResponsesAgent is "a specialized interface for serving generative AI models that handle structured responses with tool calling" — framework-agnostic via the OpenAI Responses schema. Wrap the LangGraph agent, set it as the model from code, log with resources for auth passthrough, then register to UC.

from mlflow.types.responses import ResponsesAgentRequest, ResponsesAgentResponse
from mlflow.pyfunc import ResponsesAgent
import mlflow

class LangGraphResponsesAgent(ResponsesAgent):
    def __init__(self, agent): self.agent = agent
    def predict(self, request: ResponsesAgentRequest) -> ResponsesAgentResponse:
        outputs = [e.item for e in self.predict_stream(request)
                   if e.type == "response.output_item.done"]
        return ResponsesAgentResponse(output=outputs, custom_outputs=request.custom_inputs)

mlflow.langchain.autolog()
mlflow.models.set_model(LangGraphResponsesAgent(agent))   # Models-from-Code

mlflow.set_registry_uri("databricks-uc")                  # register to UC, not workspace
mlflow.register_model(model_uri=logged.model_uri,
                      name="unity_airways.rag.ua_support_agent")
⚠️ GotchaLog with mlflow.pyfunc.log_model + ResponsesAgent and a resources list (auth passthrough), not the plain mlflow.langchain flavor. Set mlflow.set_registry_uri("databricks-uc") so the three-level catalog.schema.model name lands in Unity Catalog. Don't fall back to ChatAgent/ChatModel.
08 · Context

09.7 — context engineering

09.7 Context engineering Theory

An LLM has a fixed context window, and agent conversations fill it fast. Deciding what to put in front of the model each turn — system prompt, trimmed history, retrieved knowledge, user metadata — is context engineering. Pass the whole history every turn and you hit "token limit errors or very long latencies."

  • Truncate to recent turns — invoke on state[-5:] instead of the full state.
  • Pass just the last query to a tool — works for retrieval and text-to-SQL tools.
  • Summarize periodically — replace old turns with a running summary + latest message.
  • LLM query rewriting — turn a vague follow-up into a standalone query.
  • Keep endpoints stateless — the app backend fetches the right history and passes it in.
💡 TipDifferent tools want different context. Give a retrieval tool just the reformulated query; give the composing LLM the trimmed history plus retrieved chunks. Tailoring context per tool beats one global truncation rule.
09 · MCP

09.8 + 09.11 — MCP servers as tools, and managed MCP

09.8 MCP servers as tools Theory Hands-on

Model Context Protocol (MCP) is "an open standard… introduced by Anthropic in 2024" for exposing tools/resources to any agent in a discoverable way. Use an MCP server "when you want a standardized and reusable interface… that many agents and platforms can consume"; use a LangChain tool for one app or prototype (Table 7-2). Managed servers are platform-hosted with secure defaults; external ones are self-hosted for flexibility.

09.11 Managed MCP servers on Databricks Theory Hands-on

Databricks hosts managed MCP servers for Vector Search (AI Search), UC functions, Genie, and Databricks SQL — UC-governed, authenticated, auto-scaled. The URL points at a UC namespace:

from databricks.sdk import WorkspaceClient
from databricks_mcp import DatabricksMCPClient

ws = WorkspaceClient()
MCP_URL = f"{ws.config.host}/api/2.0/mcp/vector-search/unity_airways/rag"
mcp_tools = DatabricksMCPClient(server_url=MCP_URL, workspace_client=ws).list_tools()
⚠️ Gotcha · verify liveManaged MCP is Public Preview (some surfaces GA) — label it as such, verify the exact server URLs and enrollment on current docs, and confirm the catalog/schema in the URL points where your index/function actually lives. (Live re-check pending.)
10 · Multi-agent

09.9 + 09.10 — agents-as-tools, and testing tools

09.9 Multi-agent orchestration and agents-as-tools Theory Hands-on

A deployed agent can be a tool for another agent. A supervisor delegates sub-tasks to specialists (each an endpoint) exposed as tools. Because the specialist is a Serving endpoint, you call it with ChatDatabricks(endpoint=...) and wrap it in a Tool — the DIY version of Agent Bricks' Multi-Agent Supervisor (Module 10.3, referenced only).

from databricks_langchain import ChatDatabricks
from langchain.tools import Tool

recommender_tool = Tool(
    name="flight_recommendation_agent",
    func=get_flight_recommendation,   # calls ChatDatabricks(endpoint="...")
    description="Get real-time flight recommendations based on the user input.")

09.10 Testing agent tools before packaging and deploy Hands-on

Test tools before packaging (pairs with 09.3). Unit tests for wrappers, API handlers, and transforms; then LLM-judge evaluation with a validation set that records expected docs, expected inputs, and the expected set of tools per query. A right_tool_usage_scorer (using meets_guidelines from mlflow.genai.judges) checks tool choice — run it through mlflow.genai.evaluate() so tool tests live in the same harness as answer-quality tests.

📌 ImportantTest the tool layer and the orchestration layer separately. A tool can be perfect while the agent still calls it at the wrong time — right-tool-usage scoring catches that, and it must pass before you package with ResponsesAgent.
11 · Worked thread

Unity Airways — from RAG chain to deployed support agent

  1. Frame the leap (09.1): the FAQ chain can't answer "is my delayed booking refundable?" — it isn't connected to booking data.
  2. Design a small toolset (09.3 ★): VectorSearchRetrieverTool over the RAG chunks index, a UC get_flight_status via UCFunctionToolkit, a weather StructuredTool.
  3. Govern the tools (09.4): a WHERE clause scopes rows; Pydantic schemas type inputs; each tool handles its own errors.
  4. Build the loop (09.2, 09.5): a LangGraph tool-calling agent — agent node calls the LLM, should_continue routes to tools or ends (planning-agent reasoning).
  5. Engineer context (09.7): pass only the reformulated last query to the retriever; trim history for the composing LLM; keep the endpoint stateless.
  6. Test before packaging (09.10): unit-test each tool, then mlflow.genai.evaluate() with a validation set; the right_tool_usage_scorer must pass.
  7. Package with ResponsesAgent (09.6 ★): wrap the agent, mlflow.models.set_model(...), log with resources, run the mlflow.models.predict() pre-check.
  8. Register to UC (09.6 ★): mlflow.set_registry_uri("databricks-uc") → register_model(name="unity_airways.rag.ua_support_agent").
  9. Deploy (Module 11): agents.deploy("unity_airways.rag.ua_support_agent", version) → Serving endpoint + Review App + feedback model, tracing + inference tables + monitoring on.
  10. Extend (09.8, 09.9, 09.11): swap hand-wrapped tools for managed MCP, and delegate "best flight" queries to a deployed Flight Recommender agent — AI agent → agentic AI.

What to remember
