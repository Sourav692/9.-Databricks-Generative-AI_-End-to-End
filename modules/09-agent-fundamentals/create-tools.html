Module 09 — Agent fundamentals · Topic 09.3 · Hands-on ★

Agent Tools

An LLM can only talk until you give it tools. Build the three that a Unity Airways support agent needs — a retriever over your AI Search index, a Unity Catalog function for exact records, and a Python API tool — then let the model choose. Name, description, schema — that is the whole interface.

01 · The big idea

Three tool types, one list, the model chooses

Your RAG chain from Module 05 can search policy docs. But customers also ask "is flight UA123 delayed?" (a table row) and "will weather delay it?" (a live API). One retrieval step can't do that. You give the model discrete, described tools and let it pick per question.

  • A tool is a described function the LLM may call. It reads the name + description to choose, and the input schema to fill the arguments. Description quality is what makes selection work.
  • Retriever tool — VectorSearchRetrieverTool(index_name="unity_airways.rag.ua_rag_chunks_index", ...) from databricks_langchain wraps your Module 04/05 index for unstructured search.
  • Structured-data lookup — write CREATE FUNCTION unity_airways.rag.get_flight_status(...), then expose it with UCFunctionToolkit(function_names=["unity_airways.rag.get_flight_status"]). Governed, reusable, testable. The built-in system.ai.python_exec plugs in the same way.
  • API-calling tool — a Python function wrapped as a LangChain StructuredTool with a Pydantic args_schema. External calls leave the trust boundary, so they need governance (09.4).
  • Assemble → test → package. Collect all three into one tools list, bind_tools it to databricks-claude-sonnet-4-5, test each tool alone (09.10), then hand the list to a ResponsesAgent (09.6). Every call shows up in MLflow Traces.

03 · How it works

Match the tool type to where the answer lives

Documents, a table, or an outside system — pick the tool type accordingly, describe it well, and let the LLM choose. Getting the description and schema right is what makes selection reliable.

All three share the same discipline: one clear purpose, tested alone (09.10), added to the list → ResponsesAgent (09.6).
  • Retriever tool. Wraps the AI Search index you built in Modules 04–05. Instead of a fixed retrieval step, the agent calls it when it decides a policy/FAQ answer is needed. The tool_description tells it when that is.
  • UC function tool. An exact table lookup as a governed CREATE FUNCTION. Parameter and function COMMENTs become the description the LLM reads. UCFunctionToolkit returns a list of tools — one per function name.
  • Built-in code tool. system.ai.python_exec runs LLM-generated Python in a sandbox. Add it exactly like your own function when the agent needs to compute, not look up.
  • API tool. A plain Python function wrapped as a StructuredTool. The Pydantic args_schema secures input types; a try/except keeps one API outage from crashing the whole agent.
  • Assemble + bind. One tools list, then llm.bind_tools(tools) makes them visible to databricks-claude-sonnet-4-5. Start with 2–4 sharp tools, not a dozen fuzzy ones.
04 · Run this

Build the three tools, assemble, bind

Prereqs: serverless or DBR ML runtime; the AI Search index unity_airways.rag.ua_rag_chunks_index (Modules 04–05); rights on unity_airways.rag to CREATE FUNCTION and EXECUTE; and the served model databricks-claude-sonnet-4-5.

1. Retriever tool — wrap the index. The description is the API the LLM sees:

from databricks_langchain import VectorSearchRetrieverTool

retriever_tool = VectorSearchRetrieverTool(
    index_name="unity_airways.rag.ua_rag_chunks_index",
    num_results=5,
    tool_description=(
        "Search Unity Airways policy and FAQ documents about cancellations, "
        "refunds, baggage rules, and travel policies. Use for 'what is the policy' "
        "or 'am I allowed to' questions."
    ),
)
# verify in isolation
retriever_tool.invoke({"query": "refund on a Basic Economy fare?"})

2. Structured-data lookup — a Unity Catalog function; the COMMENTs become the tool description:

CREATE OR REPLACE FUNCTION unity_airways.rag.get_flight_status(
  in_flight_number STRING COMMENT 'IATA flight code, e.g. UA123',
  in_flight_date   STRING COMMENT 'Scheduled departure date, YYYY-MM-DD (UTC)'
)
RETURNS TABLE
COMMENT 'Return the live status, gate, and times for one Unity Airways flight on a date. Use for "is flight X delayed / cancelled" questions.'
RETURN (
  SELECT flight_number, status, departure_gate,
         scheduled_departure, estimated_departure
  FROM   unity_airways.rag.flight_status_records
  WHERE  flight_number = in_flight_number
    AND  flight_date   = in_flight_date
);

3. Test the function, then wrap it with UCFunctionToolkit:

from unitycatalog.ai.core.databricks import DatabricksFunctionClient
from databricks_langchain import UCFunctionToolkit

client = DatabricksFunctionClient()
client.execute_function(
    "unity_airways.rag.get_flight_status",
    {"in_flight_number": "UA123", "in_flight_date": "2026-07-20"},
)  # confirm the row before trusting an agent with it

uc_toolkit = UCFunctionToolkit(
    function_names=["unity_airways.rag.get_flight_status"],  # can list several
    client=client,
)
flight_tool = uc_toolkit.tools[0]   # .tools is a LIST — one per function

4. API-calling tool — typed inputs + error handling + StructuredTool:

import requests
from typing import Any
from pydantic import BaseModel, Field
from langchain.tools import StructuredTool

class AirportWeatherInput(BaseModel):
    latitude: float  = Field(description="Airport latitude in decimal degrees")
    longitude: float = Field(description="Airport longitude in decimal degrees")

def get_airport_weather(latitude: float, longitude: float) -> dict[str, Any]:
    """Fetch the short-term weather forecast for an airport location."""
    try:
        r = requests.get("https://api.open-meteo.com/v1/forecast",
                         params={"latitude": latitude, "longitude": longitude,
                                 "hourly": "temperature_2m,rain,precipitation_probability"},
                         timeout=10)
        r.raise_for_status()
        return r.json()["hourly"]
    except Exception as e:
        return {"error": f"weather lookup failed: {e}"}   # LLM can recover

weather_tool = StructuredTool(
    name="get_airport_weather",
    func=get_airport_weather,
    description="Get the hourly weather forecast for an airport by lat/long. "
                "Use to check whether weather may delay a flight.",
    args_schema=AirportWeatherInput,
)

5. Assemble the tool list and bind it to the model:

from databricks_langchain import ChatDatabricks

tools = []
tools.extend(uc_toolkit.tools)   # structured lookups
tools.append(retriever_tool)     # unstructured retrieval
tools.append(weather_tool)       # external API

llm = ChatDatabricks(endpoint="databricks-claude-sonnet-4-5")
llm_with_tools = llm.bind_tools(tools)   # tools now visible to the model

6. Test end-to-end and read the trace — this tools list is what 09.6 packages into a ResponsesAgent:

resp = llm_with_tools.invoke(
    "Is flight UA123 on 2026-07-20 delayed, and can I get a refund if it is?"
)
print(resp.tool_calls)   # which tools the model chose, and with what args
# open MLflow Traces: each tool call is a span (inputs, outputs, latency)
05 · Field notes

What to remember

📌 ImportantA tool is only as good as its name + description + input schema — that trio is the entire interface the LLM sees. Match the tool type to where the answer lives: documents → retriever tool, a table → UC function, an outside system → API tool. Then let the model choose, and confirm the choice in the trace.
💡 Tip · fieldPut the effort into the COMMENTs (UC functions) and tool_description / Field(description=...) text. When an agent misbehaves in the field, fix the descriptions and schema before touching the model or prompt — that is usually the real bug. Start with 2–4 sharp tools, not a dozen fuzzy ones.
⚠️ GotchaPackage names trip people up: the integration is databricks-langchain (from databricks_langchain import ...), not langchain-databricks; the Vector Search SDK stays databricks-vectorsearch despite the "AI Search" rebrand. The VectorSearchRetrieverTool class + kwargs here come from 📘B1 Ch7 (an O'Reilly Early Release) — the pattern is right, but confirm the exact current signature in docs before asserting it to a customer.
