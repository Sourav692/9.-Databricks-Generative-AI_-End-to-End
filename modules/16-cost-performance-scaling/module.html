Cost, performance and scaling

The Unity Airways agent works, is deployed, and is monitored. Then traffic triples and the bill doubles. This module puts the architect and operator lens on the stack: how Mosaic AI serves at scale, which levers move latency and cost, and how to profile before you scale. Cost, latency, and quality are one three-way trade-off.

01 · The big idea

Tune a few named levers; do not buy a bigger box

Cost, latency, and quality compete. You meet a target by moving the right lever, not by scaling everything. The levers: context length, embedding dimension, batch size / concurrency, endpoint sizing, and batch-vs-real-time execution.

  • Serving at scale (16.1 ★) is stateless, horizontally scalable endpoints with an autoscaler that watches throughput. GPU plus dynamic batching for heavy inference. Full deep-dive in mosaic-ai-architecture.html.
  • Context length and embedding dimension (16.2) set quality against latency, storage, and cost. Bigger is not automatically better.
  • Batch ai_query() (16.3) is the engine when no human is waiting — the same surface you used in 11.10.
  • Profiling (16.4) splits end-to-end latency so you fix the real bottleneck, not the guessed one.
  • Cost control (16.5): right-size the model, cache and batch, scale-to-zero vs provisioned throughput, AI Gateway budgets on the Foundation Model endpoint.
  • Fine-tuning and provisioned throughput (16.6) live in Mosaic AI Model Training — flagged beyond the exam blueprint.

03 · The six topics

One numbered entry per topic

  • 16.1 ★ cornerstone · Theory
    Mosaic AI architecture; model serving at scale
    Model Serving (custom / Foundation Model / external), AI Gateway, AI Search, Agent Framework, Unity Catalog. Serving at scale = stateless, horizontally scalable replicas + an autoscaler on throughput; GPU + dynamic batching for heavy inference. The agent endpoint is CPU orchestration; the LLM runs on a separate Foundation Model endpoint. Deep-dive: mosaic-ai-architecture.html.
  • 16.2 · Theory
    Context length and embedding-dimension trade-offs
    Context length caps how many retrieved chunks fit per request (4k ≈ two docs; 16k–32k = several, at higher latency/cost; overflow truncates). Embedding dimension (384 vs 1024+) trades semantic precision against storage and search speed. Model size is the sibling lever. Start moderate; raise only when profiling proves it.
    ⚠️ GotchaChanging embedding dimension forces a full re-embed and index rebuild — it is not a live config flip. Check the embedding model's max input and the LLM's context window together.
  • 16.3 · Theory + Hands-on
    Batch inference workloads suitable for ai_query()
    Suitable when three things hold: large datasets, no synchronous interaction (nightly/hourly OK), and a consistent SQL schema. Fits: embedding generation, scheduled enrichment, RAG-quality eval. Not for chatbots / live agents / dashboards. No conversational state, tool-calling, or multi-step workflow. Same surface as 11.10.
  • 16.4 · Hands-on
    Performance testing and profiling; high-throughput retrieval
    Simulate realistic traffic, then split end-to-end latency into vector-search / model-execution / queue-wait / orchestration. Match the bottleneck to a fix (Table 9-11): long context → trim; low GPU util → raise concurrency; queue wait → lower batch timeout; weak retrieval → raise embedding dim; slow index → rebuild/repartition. AI Search Standard vs Storage-optimized; hybrid search. Profile before scaling.
  • 16.5 · Theory
    Controlling LLM / GenAI costs with Databricks features
    Token-based vs compute-based pricing. Right-size the model and context; cache and batch repeated work; batch ai_query() over looping a real-time endpoint; scale_to_zero for dev, provisioned throughput for steady prod; AI Gateway usage tracking / rate limits / budgets.
    ⚠️ GotchaAI Gateway on the agent endpoint (ua-support-agent) supports inference tables only. Put rate limits and budgets on the Foundation Model endpoint the agent calls.
  • 16.6 · Theory · beyond the exam
    Fine-tuning and provisioned throughput (Mosaic AI Model Training)
    Provisioned throughput reserves tokens/second with guaranteed latency; recommended for production, required for custom/fine-tuned weights, available with compliance certs. Fine-tuning is now part of Databricks Model Training (old standalone page deprecated). Fine-tune only when volume justifies it; RAG + right-sizing ships first.
04 · Field notes

What to remember

📌 ImportantCost, latency, and quality are one trade-off, and "is a human waiting?" is the biggest decision (real-time serving vs batch ai_query()). Profile before you scale — split end-to-end latency into vector-search, model-execution, queue-wait, and orchestration, then move the one lever the bottleneck points at.
💡 Tip · fieldRead real latency and token cost from the inference table (Module 13), not a stopwatch on one request. Develop batch jobs on a LIMIT sample first. Keep the Foundation Model endpoint name stable so you can move budgets and provisioned throughput without touching agent code.
⚠️ GotchaAI Gateway on an agent endpoint = inference tables only (rate limits/guardrails need an FM or external endpoint). Provisioned throughput regions/certs change — live re-check. Fine-tuning is now Model Training. Served-model names churn (confirm databricks-claude-sonnet-4-5; DBRX retired). SDK is still databricks-vectorsearch despite the AI Search rename.
05 · Notes & self-check

Test yourself

Reveal 5-question self-check
  1. Your agent endpoint is slow under load. Before adding replicas, what do you do, and what four time components do you split end-to-end latency into?
  2. Give one advantage and one disadvantage each of a long context window and large embedding dimensions. Why is changing the embedding dimension expensive?
  3. State the three characteristics that make a workload suitable for batch ai_query(), and name two workloads that are not suitable.
  4. List four Databricks tactics for controlling GenAI cost. Which endpoint type must AI Gateway rate limits sit on, and why not the agent endpoint?
  5. What is provisioned throughput, when do you choose it over pay-per-token, and which Mosaic AI product now owns fine-tuning?
