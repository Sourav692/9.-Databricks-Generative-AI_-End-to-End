Module 16 — Cost, performance and scaling · Topic 16.1 · Theory ★

Mosaic AI architecture

You cannot size or price a system you cannot decompose. Mosaic AI is layered and independently scalable — retrieval, orchestration, model, and serving, all governed by Unity Catalog. The agent endpoint is CPU orchestration; the LLM runs on a separate GPU Foundation Model endpoint. Serving at scale means stateless replicas plus an autoscaler.

01 · The big idea

A layered factory, not one box

Think of the deployed agent as "one box that runs the model" and every scaling decision is wrong. Mosaic AI is really a set of separately governed, separately scalable services — so you can point at the exact layer a bottleneck lives in and move exactly one lever.

  • Five layers, one governance model. Unity Catalog (governance) under AI Search (retrieval), the Agent Framework (orchestration), Foundation Model APIs (model), and Model Serving + AI Gateway (serving + edge).
  • Independently scalable. Retrieval scales separately from the agent, which scales separately from the served LLM. One slow layer does not force you to overpay for the others.
  • Serving at scale = stateless + autoscaling. Replicas hold no session state, so any request hits any replica; the autoscaler watches throughput and adds/removes replicas.
  • The CPU/GPU split is the money fact. The ua_support_agent endpoint orchestrates on CPU and calls a separate Foundation Model endpoint (GPU) for the heavy inference.
  • Provisioned throughput reserves capacity for guaranteed latency and for custom/fine-tuned weights (16.6).

03 · How it works

Serving at scale — the mechanism

Scaling is not "a bigger machine." It is more stateless replicas, an autoscaler that reacts to throughput, and dynamic batching on the model layer.

  • Stateless + horizontal. Every replica is interchangeable; the router sends each request to any free replica. Adding capacity = adding replicas, not enlarging one box.
  • Autoscaling on throughput. The autoscaler grows the replica pool under load and shrinks it when idle. It reacts with a short delay — the autoscaler-delay bottleneck.
  • Cold start vs warm. With scale_to_zero an idle endpoint drops to zero replicas (cheap) but the next request pays a cold start. Latency-critical paths turn it off or use provisioned throughput.
  • Dynamic batching on the model layer. The Foundation Model endpoint groups concurrent requests into batches for GPU efficiency; larger batches raise throughput but add queue-wait.
  • Independent scaling. Retriever and model endpoints scale separately, so a retrieval-bound spike and a generation-bound spike get different fixes.
📌 ImportantWhen a customer asks "how many GPUs?", answer at the model layer (the Foundation Model endpoint), sized from token throughput — not at the agent endpoint, which is CPU orchestration.
04 · Run this

Read the endpoint config to reason about scale

Prereqs: the deployed agent endpoint ua-support-agent (Module 11), databricks-sdk, and CAN_VIEW on the endpoint. Confirm class/attribute names against your installed SDK version.

from databricks.sdk import WorkspaceClient

w = WorkspaceClient()
ep = w.serving_endpoints.get("ua-support-agent")   # the deployed agent endpoint (11)
print(ep.state)                                    # READY when serving

# Inspect sizing per served entity: the agent should be CPU-appropriate.
for se in ep.config.served_entities:
    print(se.entity_name, se.entity_version,
          se.workload_size, se.scale_to_zero_enabled)

Verify: the agent endpoint shows CPU-appropriate sizing and is READY; the GPU-heavy generation lives on a separate Foundation Model endpoint you size independently by observed token throughput (read it from the inference table, Module 13).

05 · Field notes

What to remember

📌 ImportantMosaic AI is layered and independently scalable under one Unity Catalog governance model. The agent endpoint is CPU orchestration; the LLM is a separate GPU Foundation Model endpoint. Serving at scale = stateless replicas + autoscaler (plus dynamic batching / provisioned throughput). Scale out, not up.
💡 Tip · fieldKeep the Foundation Model endpoint name stable so you can switch it to provisioned throughput without touching agent code. Size workload_size and top-k from the inference table's observed peak, not a single test request.
