Module 02 — Prompt Engineering · Topics 02.1–02.8 · Theory + Hands-on

Prompt Engineering

A prompt is a control surface, not a magic phrase. This module goes from prompt craft to prompt operations, all on one use case: Unity Airways customer support.

01 · The big idea

Prompts are production assets

Good prompts set the role, the constraints, and what to do when information is missing. The moment a prompt touches customers, you version it, review it, and roll it back like code.

  • For anything a machine parses, make the output format a contract (JSON, fixed keys, "return only that").
  • On Databricks the lifecycle tool is the MLflow Prompt Registry (mlflow.genai, Beta): immutable versions, mutable aliases, governed in Unity Catalog.
  • Promotions are driven by evaluation on a fixed dataset, not by "I like version 2 better." Optimization proposes candidates; they still earn promotion through the same checks.
02 · The problem

Why "just write a better prompt" fails

A support assistant works in the demo, then invents refund eligibility in production. The deployed prompt is whatever someone last pasted into a notebook. Nobody can say which prompt is live, who changed it, or what the last known-good version was. These are prompt lifecycle problems, not model problems.

  • One clever prompt still fails when the user is vague, angry, or missing details.
  • Free-text output breaks downstream parsers on a missing key or stray sentence.
  • Prompts as repo strings lose history, comparison, and safe rollback.
  • "Do not hallucinate" is weak; models follow "do X instead" far better.

04 · Prompt craft

02.1–02.4 — writing prompts that survive real users

02.1 Core techniques Theory

  • Zero-shot: no examples. Make it work by defining role/scope, giving measurable constraints, and saying what to do when info is missing.
  • Few-shot: a few short examples — one direct answer, one that asks for a missing detail, one that declines to speculate.
  • Reasoning: ask for deliberation, but keep it silent in production and output only the final answer.
  • Set a priority order: safety/correctness > policy > clarity > tone. Promote "be cheerful" above "don't invent policy" and you get friendly hallucinations.

02.2 Structured output + templates Hands-on

When output feeds a parser, the objective is predictability. Declare the exact schema, isolate input with delimiters, forbid extra text explicitly.

from langchain.prompts import PromptTemplate

fraud_prompt = PromptTemplate.from_template(
    "You are a fraud analysis system.\n"
    "Analyze the claim below and return ONLY a JSON object with this schema:\n"
    '{{"fraud_flag": true | false, "risk_score": number}}\n'
    "Do not include explanations, comments, or additional text.\n"
    "CLAIM:\n{claim_text}"
)

02.3 Prompt injection with templates Theory

  • Injection overrides your instructions ("Ignore previous instructions..."). In RAG it can arrive through a retrieved document, not just the user.
  • Treat all external input as untrusted; wrap it in delimiters and tell the model not to follow instructions inside them.
  • The template is one layer. Combine with input sanitization, output validation, and fallback handling.

02.4 Metaprompts Theory

A metaprompt is a persistent system-level instruction governing how the model answers. It redefines success: "answer only when supported by context; otherwise say you can't." A correct refusal is a success, not a failure.

METAPROMPT = """
You are an internal knowledge assistant.
Rules:
- Use only the information provided in the retrieved context to answer questions.
- Do not use prior knowledge or make assumptions beyond the retrieved context.
- If the retrieved context does not contain enough information, respond with:
  'I do not have sufficient information in the provided documents to answer this question.'
- Do not fabricate examples, policies, or steps.
"""
05 · Prompt operations

02.5–02.8 — keeping prompts alive in production

02.5 ★ MLflow Prompt Registry Hands-on

A Git-like model: prompts are named entities in Unity Catalog, versions are immutable, aliases are mutable pointers, tags are version metadata. Full walkthrough in the Prompt Registry deep-dive.

import mlflow
v1 = mlflow.genai.register_prompt(
    name="main.default.unity_airways_customer_support",
    template="You are a customer support assistant for Unity Airways.\n"
             "Customer question: {{question}}\nWrite a concise answer (max 120 words).",
    commit_message="v1: baseline support answer",
    tags={"use_case": "customer_support"})
mlflow.genai.set_prompt_alias(
    name="main.default.unity_airways_customer_support", alias="production", version=1)
prompt = mlflow.genai.load_prompt(
    "prompts:/main.default.unity_airways_customer_support@production")
text = prompt.format(question="Can I change my flight tomorrow?")

02.6 Evaluate and compare versions Hands-on

Keep the dataset fixed. Score each version with mlflow.genai.evaluate and the Correctness scorer; read correctness/mean; promote the winner.

from mlflow.genai.scorers import Correctness
results = mlflow.genai.evaluate(
    predict_fn=make_predict_fn(prompt_name, version),
    data=eval_dataset,
    scorers=[Correctness()])   # dataset fields: inputs, expectations (expected_facts)

02.7 Prompt optimization Hands-on

One unified API, mlflow.genai.optimize_prompts, proposes a candidate template you still review and promote through the same lifecycle.

from mlflow.genai.optimize import GepaPromptOptimizer
result = mlflow.genai.optimize_prompts(
    predict_fn=predict_fn, train_data=train_dataset, prompt_uris=[baseline.uri],
    optimizer=GepaPromptOptimizer(
        reflection_model="databricks:/databricks-claude-sonnet-4-5", max_metric_calls=20),
    scorers=[Correctness()])
candidate = result.optimized_prompts[0]

02.8 Anti-patterns and debugging Theory

  • Kitchen-sink prompt → define one primary task; make the rest testable constraints.
  • Changing many things at once → one change per version + an intent-rich commit message.
  • Hard-coding versions → load by alias; move the pointer.
  • Log prompt name + version at runtime so "it said I'd get a refund" traces to a version.
06 · Field notes

What to remember

📌 ImportantA prompt that works once is not the goal. A prompt that works when the user is vague, upset, or missing details is the goal. Design for the messy case first. And in evaluation, a correct refusal is a success, not a failure.
💡 Tip · fieldLoad by alias in production and by version in development. Pin versions for reproducible tests; use the alias as your controlled release channel. One change per version keeps iteration debuggable.
⚠️ GotchaUse mlflow.genai.evaluate (MLflow 3), not mlflow.evaluate(model_type="databricks-agent"). Registry templates use {{double}} braces — convert with to_single_brace_format() for LangChain. The Prompt Registry is Beta on Databricks (needs mlflow[databricks]>=3.1.0, a UC schema, and CREATE FUNCTION + EXECUTE + MANAGE).
