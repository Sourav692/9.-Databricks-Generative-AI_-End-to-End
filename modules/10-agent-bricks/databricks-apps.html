Module 10 — Agent Bricks and low-code agents · Topic 10.5 · Hands-on ★

Databricks Apps

A serving endpoint is an API, not a product. Put a real chat UI in front of the Unity Airways agent and ship it on Databricks Apps — a managed platform that runs your app.py next to your models, as a scoped service principal, behind SSO. Two files, one CLI, no secrets in code.

01 · The big idea

A source directory the platform runs for you

Your Module 09 agent (unity_airways.rag.ua_support_agent) already sits behind a serving endpoint after agents.deploy(). Databricks Apps is the last mile: a thin, branded chat UI that queries that endpoint, hosted in the workspace so it never leaves Unity Catalog governance.

  • Databricks Apps hosts interactive Python apps (Streamlit, Dash, Gradio, Flask, FastAPI) inside your workspace, next to data and models, each with a URL behind SSO.
  • An app is a directory. Two files matter: app.py (your chat UI) and app.yaml (the command to start it + env vars, including valueFrom resource references).
  • Resources + a service principal. Declare the serving endpoint as a resource with Can Query; the app runs as its own service principal and the SDK's Config() reads the auto-injected creds. No PAT, no secret in code.
  • The UI is thin. It calls the agent endpoint with an OpenAI-compatible client (client.chat.completions.create(model=ENDPOINT, ...)) or a REST POST /invocations. The agent does the work.
  • Ship with the CLI: databricks apps create → upload source → databricks apps deploy. databricks apps get returns the URL; share with CAN USE.

03 · How it works

Scoped identity, declared resources, one thin UI

The platform runs your directory as a service principal and injects scoped credentials. You declare the endpoint as a resource, reference it with valueFrom, and call it from app.py. Pick the deployment surface by audience.

The Review App is for reviewing quality during eval; the Databricks App is the product. They can point at the same endpoint.
  • The directory. app.py + app.yaml (+ requirements.txt for extras). Runtime is Python 3.11, ~2 vCPU / 6 GB, with Streamlit/Dash/Gradio/Flask/FastAPI pre-installed. Bind DATABRICKS_APP_PORT (default 8000); never 8080.
  • app.yaml. command (e.g. ["streamlit","run","app.py"]) plus env entries that are literal values or valueFrom resource references — so no IDs or secrets in the repo.
  • Resources + SP. Attach the serving endpoint as a resource with Can Query. The app runs as its own service principal; DATABRICKS_CLIENT_ID/_SECRET are auto-injected and the SDK Config() uses them.
  • Per-user governance (optional). Turn on user auth and read x-forwarded-access-token so queries respect the signed-in user's Unity Catalog permissions. That trade-off is 11.9.
  • Query the endpoint. OpenAI-compatible client or REST /invocations. The LLM work stays on the endpoint; the app stays thin.
04 · Run this

Build app.py + app.yaml, then deploy

Prereqs: a deployed agent/chain serving endpoint (Module 09 agents.deploy("unity_airways.rag.ua_support_agent", version), a Module 05 chain, or a 10.2 Knowledge Assistant); the databricks CLI (v0.294.0+); rights to create an app and grant it Can Query on the endpoint.

1. app.py — a Streamlit chat UI that queries the agent endpoint:

import os
import streamlit as st
from databricks.sdk import WorkspaceClient

st.set_page_config(page_title="Unity Airways Support", layout="centered")  # must be first

# Injected from app.yaml (valueFrom: serving-endpoint) — the name of the endpoint
# that agents.deploy() created for unity_airways.rag.ua_support_agent.
ENDPOINT = os.environ["SERVING_ENDPOINT"]

@st.cache_resource                      # one client across reruns
def get_client():
    # WorkspaceClient() auto-detects the app SP creds via SDK Config().
    # get_open_ai_client() returns an OpenAI-compatible client on /serving-endpoints.
    return WorkspaceClient().serving_endpoints.get_open_ai_client()

client = get_client()
st.title("Unity Airways — Support Assistant")

if "messages" not in st.session_state:
    st.session_state.messages = []
for m in st.session_state.messages:
    with st.chat_message(m["role"]):
        st.markdown(m["content"])

if prompt := st.chat_input("Ask about a flight, refund, or baggage rule…"):
    st.session_state.messages.append({"role": "user", "content": prompt})
    with st.chat_message("user"):
        st.markdown(prompt)
    with st.chat_message("assistant"):
        resp = client.chat.completions.create(
            model=ENDPOINT, messages=st.session_state.messages,
        )
        answer = resp.choices[0].message.content
        st.markdown(answer)
    st.session_state.messages.append({"role": "assistant", "content": answer})

2. app.yaml — start command + the endpoint injected via valueFrom (no hardcoded ID):

command:
  - "streamlit"
  - "run"
  - "app.py"

env:
  - name: SERVING_ENDPOINT
    valueFrom: serving-endpoint

3. requirements.txt — only the extras (frameworks are pre-installed):

databricks-sdk
openai

4. Create, upload, deploy with the databricks apps CLI:

# name: <=26 chars, lowercase letters/numbers/hyphens, no underscores
databricks apps create ua-support-chat

databricks workspace mkdirs /Workspace/Users/me@unity.com/apps/ua-support-chat
databricks workspace import-dir . /Workspace/Users/me@unity.com/apps/ua-support-chat

databricks apps deploy ua-support-chat \
  --source-code-path /Workspace/Users/me@unity.com/apps/ua-support-chat

5. Add the endpoint resource, then verify — in the UI (Configure → + Add resource → serving endpoint → Can Query), redeploy so valueFrom resolves:

databricks apps get ua-support-chat    # app_status.state: RUNNING + the app URL
databricks apps logs ua-support-chat   # [SYSTEM]/[APP] lines; "App started successfully"
# Share: grant CAN USE to the support group, CAN MANAGE to developers.

Alternative query pattern — dependency-light REST call (or when your agent uses the Responses schema):

import os, requests
from databricks.sdk.core import Config

cfg = Config()                                  # picks up the app SP credentials

def ask_agent(messages):
    headers = cfg.authenticate()                # adds the Authorization bearer header
    headers["Content-Type"] = "application/json"
    endpoint = os.environ["SERVING_ENDPOINT"]
    r = requests.post(
        f"https://{cfg.host}/serving-endpoints/{endpoint}/invocations",
        headers=headers, json={"messages": messages}, timeout=120,
    )
    r.raise_for_status()
    return r.json()
05 · Field notes

What to remember

📌 ImportantA Databricks App is just a source directory (app.py + app.yaml) that the platform runs as a scoped service principal. The two rules that keep it safe and portable: declare the serving endpoint as a resource (permission Can Query) and reference it with valueFrom — never a hardcoded ID or token. The app is a thin UI; the agent endpoint does the work.
💡 Tip · fieldStart from the AI Playground → "Export to Databricks Apps" (10.1) to get an app.py + app.yaml already wired to your endpoint, then customize. Iterate with databricks apps run-local before you deploy, and watch databricks apps logs on first boot — the "App started successfully" line is your green light. Keep the app at CAN USE for the team, CAN MANAGE for yourself.
⚠️ Gotchadatabricks apps logs needs OAuth auth (not a PAT) — use databricks apps get for status if you used a token. The endpoint name from agents.deploy() is auto-generated (commonly agents_<catalog>-<schema>-<model>, e.g. agents_unity_airways-rag-ua_support_agent) — read it from the deploy output, don't guess. The SDK helper get_open_ai_client() and the exact serving-endpoint resource key are grounded in the databricks-apps skills + docs; the doc pages are JS-rendered, so treat the precise strings as live re-check pending and confirm against current docs before asserting to a customer.
