Module 10 — Agent Bricks · Topic 10.2 · Theory + Hands-on ★

Knowledge Assistant

In Modules 04–05 you hand-built a cited RAG bot. Here you point Databricks at your documents and it stands up the same thing — retrieval, a quality loop, and citations, all managed — from the console, with no agent code. Same outcome, a fraction of the build.

01 · The big idea

A managed, cited Q&A chatbot over your docs — no code

Most "answer questions over our documents" requests don't need anything bespoke. They need retrieval, a grounding prompt, citations, and a quality signal. A Knowledge Assistant gives you all four from the console — you supply the documents and a few settings.

  • Knowledge Assistant (Agent Bricks) — a managed, cited Q&A chatbot over your documents. Build it in the console: attach a knowledge source, name it, describe it, click create. No agent code. GA (Jan 2026).
  • Knowledge source = one or both: a UC Volume folder of documents (PDF / TXT / MD), or an existing AI Search index — reuse your Module 04 unity_airways.rag.ua_rag_chunks_index.
  • Databricks manages the hard parts: chunking + indexing (when the source is a Volume), retrieval, generation, a judge-backed quality loop, guardrails, and inline citations back to the source document.
  • Same outcome as the hand-coded RAG chain (Module 05), far less code. You trade fine control over chunking/retriever/prompt for speed and a managed quality loop.
  • Consume it via the serving endpoint it creates, the built-in tile chat, or Export to a Databricks App — the hand-off to Topic 10.5.

03 · How it works

Pick a source, and Databricks manages the rest

Point the assistant at a UC Volume folder or an AI Search index, add a description, instructions, and sample questions — then Databricks handles retrieval, generation, the quality loop, and citations. The decision that matters is when this is enough versus when you need the custom chain.

Both roads reach a governed, cited Q&A endpoint. Low-code for standard doc Q&A; custom when you need control the console doesn't expose.
  • Knowledge source. A UC Volume folder (Databricks chunks + indexes for you) or an existing AI Search index (retrieves against what you already built). Attach one or both. Both are Unity Catalog objects, so your grants carry over.
  • Managed retrieval + generation. At question time the KA fetches relevant chunks and generates a grounded answer with a managed foundation model. You don't write or tune the retriever or the prompt.
  • Judge-backed quality loop. An LLM-as-judge scores answers — the same idea you formalize in Module 07 with mlflow.genai. Your sample questions seed it.
  • Citations. Every answer points back to the source document/chunk. This is the trust feature: a human can verify, and a wrong answer is traceable rather than a silent hallucination.
  • What you give up. Chunk size, retriever k, embedding model, prompt internals, judge config — all managed. Need those? Drop to the hand-coded chain (Module 05) + ResponsesAgent (Module 09).
04 · Run this

Build it in the console, then consume the endpoint

Prereqs: no cluster needed to author (it's console-driven), but you need a knowledge source in Unity Catalog — a Volume of documents or the AI Search index unity_airways.rag.ua_rag_chunks_index — and permission to create Agent Bricks. Provisioning uses serverless capacity.

1–6. Console build (no code):

# Console-driven — nav labels can shift between releases; confirm in the current console.
1. Left nav → Agent Bricks → Knowledge Assistant → New
2. Name        : Unity Airways Policy Assistant
3. Description : Answers customer questions about refunds, baggage,
                 cancellations, and travel policy.
4. Instructions: Always cite the specific policy document.
                 If the answer is not in the docs, say so plainly.
                 Use short bullet points for multi-part answers.
5. Knowledge source (pick one or both):
     - UC Volume folder : /Volumes/unity_airways/rag/ua_policy_docs   (managed chunk + index)
     - AI Search index  : unity_airways.rag.ua_rag_chunks_index       (reuse Module 04)
6. (Optional) Sample questions — seed the quality loop, onboard users:
     "What is the checked-bag allowance in Economy?"
     "How do I rebook after a cancellation?"
   → Create. Status moves PROVISIONING → ONLINE (a few minutes).

7. Consume the serving endpoint — standard Model Serving query pattern. Confirm the endpoint name and the exact request/response schema in the console before wiring a client:

from databricks.sdk import WorkspaceClient

w = WorkspaceClient()
resp = w.serving_endpoints.query(
    name="ka-<your-ka-id>-endpoint",          # exact name shown in the console
    messages=[{"role": "user",
               "content": "Can I get a refund on a Basic Economy fare?"}],
)
print(resp)   # answer text plus citation metadata (schema per the endpoint signature)

Or skip the code: use the built-in chat in the tile to test, then Export to Databricks Apps to embed the assistant in a shareable chat UI — that's Topic 10.5.

How to verify: an answer in the tile cites a real source document, the status is ONLINE, and an endpoint query returns the same grounded, cited answer.

05 · Field notes

What to remember

📌 ImportantA Knowledge Assistant gives you the same outcome as a hand-coded RAG chain — grounded, cited answers on a governed endpoint — with essentially no code. The whole build is: pick a knowledge source (UC Volume folder or AI Search index), name it, describe it, add instructions and sample questions, create. Databricks manages chunking/indexing, retrieval, generation, the judge-backed quality loop, and citations. It is GA (Jan 2026).
💡 Tip · fieldYour two highest-leverage inputs are the instructions and the sample questions — the only real levers you have, so spend your effort there ("always cite; say so if it's not in the docs"). Already built the Module 04 index? Attach it as the source instead of re-uploading files: you keep your chunking/embedding choices and still get the managed assistant on top.
⚠️ GotchaDon't present exact console field labels, the endpoint name pattern (ka-<id>-endpoint), or the endpoint's request/response schema as fixed — Agent Bricks UI and endpoint signatures change between releases, and the docs pages are JS-rendered (a static fetch returns nothing). Confirm nav labels, the endpoint name, and the payload shape in the current console before wiring a client or asserting them to a customer. When doc Q&A isn't enough (tools, multi-step planning), the KA is the wrong tool — reach for a custom ResponsesAgent (Module 09) or a Supervisor Agent (10.3).
