Module 10 — Agent Bricks and no/low-code agents · Topics 10.1–10.8 · Theory + Hands-on

Agent Bricks and Low-Code Agents

The no/low-code counterpart to Module 09. Instead of hand-coding a ResponsesAgent, you build the Unity Airways support experience in the Agent Bricks console and AI Playground, then ship it on Databricks Apps — with an escape hatch back to code whenever a tile can't do the job.

01 · The big idea

From a hand-coded agent to console tiles and a hosted app

Module 09 built the engine — a LangGraph loop wrapped in ResponsesAgent, registered to UC, deployed with agents.deploy. Module 10 assembles the car from parts and bolts on a dashboard: prototype in AI Playground, stand up a tile in Agent Bricks, and put a Databricks App in front. Same governed stack underneath, far less fabrication.

  • Prototype no-code (10.1) — AI Playground (GA) compares LLMs and attaches tools (UC functions, AI Search, MCP), then "Export to Databricks Apps."
  • Build with a tile — a Knowledge Assistant (10.2 ★, GA) for cited Q&A, a Multi-Agent Supervisor (10.3, GA) to route across specialists, or Information Extraction / Custom LLM (10.7 / 10.8, Beta) for a narrow task.
  • Escape hatch to code (10.4) — when a tile can't express it, drop back to a Custom Agent (Python ResponsesAgent, Module 09).
  • Front door (10.5 ★) — Databricks Apps hosts a Python chat UI via app.yaml + the databricks apps CLI, wired to endpoints through scoped resources.
📌 The rule that shapes this moduleStart no-code, drop to code only when a tile hits a wall. Default path: AI Playground → Agent Bricks tile → Databricks App; the Custom Agent (Module 09) is the escape hatch. Name the maturity of each tile: GA — AI Playground, Knowledge Assistant, Multi-Agent Supervisor; Beta — Information Extraction, Custom LLM. Agent Bricks itself launched Beta (June 2025). Module 10 reuses the Module 04 AI Search index and can front the Module 09 agent — it doesn't replace them.
02 · The problem

Why "just code it" and "just ship the endpoint" both fail

Module 09's agent was the right amount of work for a differentiated production system. It is far too much for a Tuesday-afternoon proof of value, and it leaves users with an API, not an app. Naive moves break:

  • Hand-code every agent → correct but slow; most early use cases just need cited Q&A, which a Knowledge Assistant does out of the box.
  • Wire the multi-agent system by hand → real maintenance (routing, descriptions, auth); the Multi-Agent Supervisor is the managed version.
  • Ship the endpoint and stop → an endpoint isn't an application; without a Databricks App, only API users can reach it.
  • Treat every tile as GA → Information Extraction and Custom LLM are Beta; promising them on a production date turns demos into escalations.
  • Use tiles for everything → they cover the common shapes, not all; custom logic belongs in a Custom Agent (Module 09), not a bent tile.

04 · Prototype

10.1 — AI Playground: prototyping agents

10.1 AI Playground Hands-on GA

AI Playground is the no-code chat surface where you try ideas before building anything. Compare foundation models side by side (databricks-claude-sonnet-4-5 is the Unity Airways default), then prototype a tool-calling agent with no code by attaching tools and watching the model decide when to call them.

  • Unity Catalog functions — a governed get_flight_status lookup.
  • AI Search retrieval against unity_airways.rag.ua_rag_chunks_index (the Module 04 index).
  • MCP servers (managed or external) for reusable, governed tools.
  • Export when it's good — "Export to Databricks Apps" for a hosted chat UI, or export to code for the Module 09 path.
💡 TipUse Playground as the cheap, fast shape-finding step — decide the model, the tools, and roughly the system prompt here, then move to a Knowledge Assistant (pure Q&A) or export to Apps/code. It is the lowest-cost way to kill a bad idea before you build it.
05 · Cornerstone

10.2 ★ — Knowledge Assistant: cited document Q&A

Full deep-dive — knowledge sources, instructions and examples, evaluation, endpoint, and embedding it in an app — in the Knowledge Assistant deep-dive (knowledge-assistant.md / knowledge-assistant.html).

10.2 Knowledge Assistant Theory Hands-on Cornerstone GA

The fastest path to the Unity Airways FAQ bot — a cited Q&A chatbot you configure, not code.

  • Point it at knowledge — a UC volume folder of PDF/text docs, or the existing AI Search index unity_airways.rag.ua_rag_chunks_index.
  • Shape its behavior — a description, instructions (tone, what to cite), and example questions with guidelines that double as an evaluation set.
  • It builds, evaluates, and serves — provisions an endpoint (PROVISIONING → ONLINE), grounds answers, and returns citations — the Module 05 RAG pattern with no chain code.
📌 ImportantA Knowledge Assistant is the no-code twin of the Module 05 RAG chain — reuse the same AI Search index so retrieval quality (chunking, embeddings) carries over. Weak answers are usually an upstream index problem (Module 03/04), not a tile problem.
06 · Orchestrate + escape

10.3 + 10.4 — Multi-Agent Supervisor, and Custom Agents

10.3 Multi-Agent Supervisor (a.k.a. Supervisor Agent) Theory Hands-on GA

The managed version of Module 09.9's agents-as-tools. You declare specialist sub-agents; it routes and composes across Genie Agents (structured tables), Knowledge Assistants (unstructured docs), and MCP servers (tools/actions). Routing is description-driven — each sub-agent's description tells the supervisor when to pick it, exactly like a good tool_description. Unity Airways: policy → Knowledge Assistant, "my flight / booking" → a Genie Agent over the ops tables, actions → an MCP tool, composed into one cited answer.

10.4 Custom Agents — the escape hatch to code Theory Hands-on

When the tiles can't express the logic, drop back to a Custom Agent: a Python ResponsesAgent + framework (the Module 09.6 artifact), deployed via agents.deploy() or behind a Databricks App. Same UC governance and serving as the tiles — more control. Reach for it for bespoke tools, custom control flow, or non-standard I/O.

⚠️ GotchaDon't force a tile to do a Custom Agent's job. If you're hacking instructions to fake tool logic a tile doesn't support, that's the signal to switch to code (Module 09). The escape hatch is a feature, not a failure. And give Supervisor sub-agents narrow remits + crisp descriptions — vague/overlapping descriptions cause wrong routing.
07 · Cornerstone

10.5 ★ — Build and deploy a GenAI app on Databricks Apps

Full deep-dive — app.yaml structure, the databricks apps CLI (init / validate / deploy / get), resources and scoped auth, and wiring a chat UI to the agent endpoint — in the Databricks Apps deep-dive (databricks-apps.md / databricks-apps.html).

10.5 Databricks Apps Theory Hands-on Cornerstone GA

The managed platform that gives your agent a front door — a Python web app running next to your data and governance.

# app.yaml — the manifest that declares how the app runs
command: ["streamlit", "run", "app.py"]

# the databricks apps CLI loop
databricks apps create ua-support-chat    # create the app (decide the data-access pattern first)
databricks workspace import-dir . /Workspace/Users/me@unity.com/apps/ua-support-chat   # upload source
databricks apps deploy ua-support-chat --source-code-path /Workspace/Users/me@unity.com/apps/ua-support-chat   # ship it
databricks apps get ua-support-chat -o json   # confirm app_status.state: RUNNING
  • Frameworks: Python — Streamlit, Dash, Flask, Gradio. A Streamlit chat UI fronts the Unity Airways support agent.
  • Resources with scoped auth: the app reaches a serving endpoint (your agent, or databricks-claude-sonnet-4-5) and a SQL warehouse through declared resources — the app runs as its own service principal, not your token.
📌 ImportantBefore scaffolding, decide the data-access pattern — most read-only GenAI apps use Analytics (SQL warehouse at read time); reach for Lakebase synced tables only for sub-second lookups (search, typeahead, real-time by-ID). App auth/authz (service principal, OAuth scopes, on-behalf-of-user) is covered in Module 11.9.
08 · End-to-end

10.6 — Get started with AI agents end-to-end

10.6 Get started end-to-end Hands-on

The glue topic — the full low-code journey in one pass, so you can run it as a single demo:

  1. Prototype (10.1): AI Playground — databricks-claude-sonnet-4-5 + retriever + UC function; confirm the shape works.
  2. Build the agent (10.2 / 10.3): a Knowledge Assistant for pure Q&A; a Multi-Agent Supervisor when the use case spans structured + unstructured + actions.
  3. Go custom if needed (10.4): author a Custom Agent (ResponsesAgent, Module 09) for logic the tiles can't express.
  4. Front it (10.5 ★): a Databricks App (Streamlit chat) wired to the endpoint via a resource; deploy with the CLI.
  5. Govern and monitor (Module 11): tracing + monitoring on the endpoint; add a Review App and AI Gateway guardrails.
💡 Tip · fieldRehearse this end-to-end before a customer session. It lands hardest when you start in Playground (fast, visual), graduate to a Knowledge Assistant (cited answers), and finish in an app (a real front door) — each step visibly building on the last.
09 · Beta tiles

10.7 + 10.8 — Information Extraction, and Custom LLM

10.7 Information Extraction Theory Hands-on Beta

Turns unstructured docs / PDFs / images into a clean structured table via a generated JSON schema — no parsing code. Unity Airways: scanned booking confirmations or baggage-claim PDFs → a governed table (booking_ref, passenger_name, flight_number, claim_amount) ready for analytics or a Genie Agent. It's the console cousin of the SQL functions ai_parse_document / ai_extract (Module 03 / 11.10).

10.8 Custom LLM Theory Hands-on Beta

Builds a task-tuned model from your examples for a domain text task — summarize (condense complaint threads), classify (route tickets by category/urgency), transform (rewrite notes), generate (draft policy-grounded replies). Use it for a repeated, well-scoped text task where a general chat model is overkill or inconsistent.

⚠️ Gotcha · verify liveBoth tiles are Beta — label the maturity, expect UI/behavior changes, and verify enrollment before a production timeline. For hardened, GA work today: ai_extract / ai_parse_document for batch extraction, and ai_classify / ai_summarize / ai_gen for scaled text tasks (Module 11.10). 10 · Worked thread

Unity Airways — the same experience, low-code

  1. Prototype (10.1): AI Playground — chat with databricks-claude-sonnet-4-5, attach the AI Search retriever over unity_airways.rag.ua_rag_chunks_index and a get_flight_status UC function; confirm tool calls + citations.
  2. Cited Q&A (10.2 ★): a Knowledge Assistant over the same index (or a UC volume of FAQ PDFs) + instructions + example questions → a cited support bot endpoint, no chain code.
  3. Compose specialists (10.3): a Multi-Agent Supervisor routes policy → the Knowledge Assistant, "my flight / booking" → a Genie Agent over the ops tables, actions → an MCP tool.
  4. Go custom only if needed (10.4): author a Custom Agent (ResponsesAgent, Module 09) for behavior outside the tiles; treat it as a sub-agent or the app backend.
  5. Enrich data (10.7 / 10.8, Beta): Information Extraction turns scanned booking PDFs into a table; a Custom LLM classifies inbound tickets — both labeled Beta.
  6. Front end (10.5 ★): a Databricks App (Streamlit chat) wired to the endpoint via a serving-endpoint resource; databricks apps deploy → confirm RUNNING.
  7. Govern and monitor (Module 11): tracing + monitoring on the endpoints; a Review App for feedback and AI Gateway guardrails at serving time.
