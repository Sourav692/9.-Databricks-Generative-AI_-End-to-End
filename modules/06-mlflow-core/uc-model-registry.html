UC Model Registry

Name it, version it, point at it. A logged run becomes a governed Unity Catalog object with an immutable version history and one movable pointer — the @champion alias — that serving, rollback, and audit all read. Aliases and tags, not stages.

01 · The big idea

From run artifact to governed asset

In 05.6 you logged the Unity Airways RAG chain and got logged.model_uri = runs:/<run_id>/chain. That URI is tied to one run in one workspace — not something production should depend on. Registering it into Unity Catalog gives it a stable name, a version history, a movable "this one is live" pointer, metadata, and access control.

  • Point the registry at UC once: mlflow.set_registry_uri("databricks-uc"). After that every register writes to Unity Catalog, with RBAC, lineage, and cross-workspace access instead of the per-workspace legacy registry.
  • Register with a three-level name: mlflow.register_model("runs:/<run_id>/chain", "unity_airways.rag.ua_rag_chain"). The name is always catalog.schema.model. Each register mints a new immutable version.
  • Aliases replace stages. UC models have no Staging/Production stages. You set a movable pointer — set_registered_model_alias("...", "champion", 3) — and load models:/...@champion. Promote or roll back by repointing it, with zero client changes.
  • Tags + grants make it auditable and safe. Tag versions with eval score, approver, and change ticket; GRANT EXECUTE ON MODEL to just the serving service principal.

03 · How it works

Aliases replaced stages — the one distinction

UC models have no stages. The "which version is live" question is answered by a movable alias, not by a fixed stage ladder. Get this right and everything downstream — serving, rollback, audit — falls out of it.

Roll forward or back by repointing the alias; clients keep the same code and URL.
  • UC beats the legacy Workspace registry. Models become UC securables: access is GRANT/REVOKE, lineage links model → run → data, and one metastore serves dev, staging, and prod workspaces. Workspace-registry models were trapped in one workspace.
  • Each register call is a new immutable version. Register the same three-level name again and you get v2, v3, …; older versions never change. That immutability is what makes rollback trustworthy.
  • The alias is the operational knob. set_registered_model_alias(name, "champion", 4) promotes v4; call it with 3 to roll back. Consumers loading @champion follow instantly.
  • Tags are the audit trail. set_registered_model_tag labels the whole model (team, project); set_model_version_tag labels one build (eval score, approver, change ticket, lifecycle=archived).
  • Grants gate who can serve it. EXECUTE lets a principal load/serve; ownership or MANAGE is needed to move aliases and tags. Registry permissions and endpoint permissions are separate levels of control.
04 · Run this

Register, version, alias, tag, grant

Prereqs: serverless or DBR ML runtime, MLflow ≥ 3.1; the logged chain from 05.6 (logged.model_uri); rights to create a model in unity_airways.rag (USE CATALOG + USE SCHEMA + CREATE MODEL).

%pip install -U mlflow databricks-langchain
dbutils.library.restartPython()

0. Point the registry at Unity Catalog and set the three-level name:

import mlflow
from mlflow import MlflowClient

CATALOG  = "unity_airways"
SCHEMA   = "rag"
UC_MODEL = f"{CATALOG}.{SCHEMA}.ua_rag_chain"   # catalog.schema.model

mlflow.set_registry_uri("databricks-uc")        # ← every register now writes to UC
client = MlflowClient()

1. Register the logged chain — this creates version 1:

mv = mlflow.register_model(
    model_uri=f"runs:/{run.info.run_id}/chain",   # from 05.6; or f"models:/{logged.model_id}"
    name=UC_MODEL,
)
print(mv.name, mv.version)   # unity_airways.rag.ua_rag_chain 1

2. Register improved builds to see versioning (v2, then v3):

mv2 = mlflow.register_model(f"runs:/{run2.info.run_id}/chain", UC_MODEL)  # -> version 2
mv3 = mlflow.register_model(f"runs:/{run3.info.run_id}/chain", UC_MODEL)  # -> version 3

3. Set the champion alias to the approved version — this is the "promotion":

client.set_registered_model_alias(name=UC_MODEL, alias="champion", version=3)

Verify + load by alias (the way apps and endpoints should reference the model):

print(client.get_model_version_by_alias(UC_MODEL, "champion").version)   # 3
chain = mlflow.langchain.load_model(f"models:/{UC_MODEL}@champion")
print(chain.invoke("Can I get a refund on a Basic Economy fare?")[:200])

4. Tag the version (audit trail) and the model (ownership):

client.set_model_version_tag(UC_MODEL, 3, "eval_correctness", "0.91")
client.set_model_version_tag(UC_MODEL, 3, "approver", "s.banerjee")
client.set_model_version_tag(UC_MODEL, 3, "change_ticket", "CHG-2187")
client.set_registered_model_tag(UC_MODEL, "team", "ml-platform")

5. Roll back in one line — no redeploy:

client.set_registered_model_alias(UC_MODEL, "champion", 2)   # champion now = v2
# every models:/...@champion consumer follows instantly; v3 is untouched for later

6. Grant a serving service principal permission to load it (SQL cell or Catalog Explorer):

GRANT USE CATALOG ON CATALOG unity_airways TO `sp-ua-serving`;
GRANT USE SCHEMA  ON SCHEMA  unity_airways.rag TO `sp-ua-serving`;
GRANT EXECUTE     ON MODEL   unity_airways.rag.ua_rag_chain TO `sp-ua-serving`;

The legacy stage call — recognize it, never use it on a UC model:

# LEGACY / Workspace registry — on a UC model there is no stage to move to.
client.transition_model_version_stage(name="ua_rag_chain", version=3, stage="Production")
# Correct UC equivalent:
client.set_registered_model_alias(UC_MODEL, "production", 3)
05 · Field notes

What to remember

📌 ImportantThe whole lesson reduces to three moves: name it (catalog.schema.model), version it (every register call), point at it (an alias, never a stage). Everything downstream — serving, rollback, audit — reads the alias. In the UC Model Registry, model stages do not exist; the lifecycle is aliases + tags.
💡 Tip · fieldStandardize alias names across the org: champion = live, challenger = the next candidate under eval. Attach the eval score, approver, and change ticket to the version as tags at promotion time — you will thank yourself during the next incident review when you can answer "which version, who approved it, and why" from the registry alone.
⚠️ GotchaBooks lag here. 📗B2 Ch5 still describes registration as enabling "controlled stage transitions" through staging/production stages — legacy Workspace-registry / MLflow-2 framing. 📗B2 Ch6 corrects it: "lifecycle management is implemented through aliases and tags rather than deprecated stages," and notes the exam may still use "stage" wording. Teach aliases + tags. Method names (set_registered_model_alias, set_registered_model_tag, set_model_version_tag, get_model_version_by_alias) verified against the current MLflow client API.
