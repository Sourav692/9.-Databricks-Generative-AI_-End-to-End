Module 06 — MLflow for GenAI core · Topic 06.2 · Theory ★

MLflow 2 to 3

The tracking core carries over. The GenAI layer was rebuilt. This is your migration map so you never teach a dead API: mlflow.genai.evaluate over mlflow.evaluate(model_type=...), LoggedModel + tracing, and UC aliases instead of stages.

01 · The big idea

Same spine, new GenAI layer

The two books teach MLflow the 2.x way, and the MLflow book predates the mlflow.genai surface. MLflow 3 kept the core you know and rewrote the GenAI workflow around it. The whole lesson is a translation table you carry in your head.

  • Carries over (unchanged): Experiments, Runs, params, metrics, artifacts, log_model, signatures. Migration from 2.x is mostly additive — "it still preserves its core tracking concepts, making migration from 2.x quick and simple" (📘B1 Ch1).
  • New for GenAI: a first-class LoggedModel (a version that links its own traces, evals, and metrics via mlflow.set_active_model()), first-class OpenTelemetry tracing, and Models-from-Code packaging.
  • Rewritten: evaluation is now mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]) with explicit scorers; the registry is UC-first with @champion/@challenger aliases instead of Staging/Production stages; a Prompt Registry (Beta) is new.
  • Never get wrong: GenAI eval is mlflow.genai.evaluate() — there is no agents.evaluate(), and mlflow.evaluate(model_type="databricks-agent") is the former path.

03 · How it works

Why the version object matters

In MLflow 2 the traces, eval scores, and metrics for a GenAI app floated loose around a Run. In MLflow 3 they all hang off one LoggedModel version, so "which version produced this trace, scored this way" has a definite answer.

Activate a version with mlflow.set_active_model(); traces, evals, and metrics attach to it; then register to UC and promote with an alias.
  • LoggedModel is the hub. mlflow.set_active_model(name="ua_rag_chain_v1") creates/activates a version; subsequent traces and evals link to it. Forget it and lineage still records, but is not tied to a named version. (📘B1 Ch4; naming-conventions §1. Live doc re-check pending — logged-model page is JS-rendered.)
  • Evaluation is explicit now. mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]). You must pass scorers=[...] yourself — 3.x does not auto-select judges the way v2 did (📘B1 Ch1: v2 had "automated judge selection … more manual knobs").
  • Tracing is first-class. Decorate with @mlflow.trace, open mlflow.start_span(), or mlflow.<lib>.autolog(). OpenTelemetry-compatible, GenAI semantic conventions.
  • Packaging changed default. Models-from-Code (mlflow.models.set_model()) over pickling for chains/agents that hold live clients (Module 05.6).
  • Registry moved to UC. set_registry_uri("databricks-uc"), three-level catalog.schema.model, and @champion/@challenger aliases replace Staging/Production stages.
04 · Translate this

Old (do not ship) vs new

[Theory] Illustrative contrasts, not a runnable flow. Assume MLflow ≥ 3.1, CATALOG="unity_airways", SCHEMA="rag".

1. Evaluation — the entry point moved to the mlflow.genai namespace:

# ── MLflow 2 (book style) — deprecated GenAI path ──
mlflow.evaluate(
    model=my_chain,
    model_type="databricks-agent",     # gone for GenAI in 3.x
    data=eval_df,                       # request / response / expected_response
)

# ── MLflow 3 (current) ──
from mlflow.genai.scorers import Correctness, RelevanceToQuery, RetrievalGroundedness
results = mlflow.genai.evaluate(
    data=eval_df,                       # inputs / outputs / expectations
    predict_fn=my_chain.invoke,         # was model=
    scorers=[Correctness(), RelevanceToQuery(), RetrievalGroundedness()],  # MUST be explicit
)

2. Version identity — link traces and evals to a LoggedModel (new in MLflow 3):

active = mlflow.set_active_model(name="ua_rag_chain_v1")
# ...run/trace/eval the app; results attach to this LoggedModel version.

3. Tracing — instrument instead of hand-logging:

mlflow.langchain.autolog()              # auto-capture the whole chain as spans

@mlflow.trace                           # or trace one function
def retrieve(question: str):
    return retriever.invoke(question)

4. Registry — UC three-level name + alias instead of stage:

# ── MLflow 2 / OSS workspace registry (stages) ──
# client.transition_model_version_stage(name="ua_rag_chain", version=1, stage="Production")

# ── MLflow 3 on Databricks (UC + aliases) ──
mlflow.set_registry_uri("databricks-uc")
mv = mlflow.register_model("runs:/<run_id>/model", "unity_airways.rag.ua_rag_chain")

from mlflow import MlflowClient
MlflowClient().set_registered_model_alias(
    name="unity_airways.rag.ua_rag_chain", alias="champion", version=mv.version,
)

Quick check — spot the 2.x pattern, then rewrite: model_type= or agents.evaluate → mlflow.genai.evaluate; stage="Production" → @champion alias; inline prompt to version → Prompt Registry (Beta).

05 · Field notes

What to remember

📌 ImportantMLflow 3 keeps your tracking core (Experiments, Runs, params, metrics, artifacts) and adds/rewrites the GenAI layer. The two facts to never get wrong: GenAI eval is mlflow.genai.evaluate(data=, predict_fn=, scorers=[...]) (there is no agents.evaluate(), and mlflow.evaluate(model_type="databricks-agent") is the former path), and you must pass scorers=[...] explicitly.
💡 Tip · fieldKeep the mapping table open when you read the MLflow book. Translate each GenAI API before you type it: mlflow.evaluate(model_type=...) → mlflow.genai.evaluate(...), stage="Production" → @champion alias, inline prompt → Prompt Registry. The book is right about why; the docs are right about what to call it.
⚠️ GotchaLoggedModel, first-class tracing, and Models-from-Code are new-in-MLflow-3 and GA — but Prompt Registry, production monitoring, and Unity AI Gateway are Beta/Preview. Don't present Beta as GA in a customer demo, and re-verify status at authoring time.
