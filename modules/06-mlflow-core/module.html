Module 06 — MLflow for GenAI Core · Topics 06.1–06.8 · Theory + Hands-on

MLflow for GenAI Core

Every iteration of the Unity Airways assistant needs a record: what code, what data, what config, what model. This module names the machinery behind Module 05's registered chain — Experiments, Runs, the MLflow-3 LoggedModel, and the Unity Catalog registry — and where all of it is stored.

01 · The big idea

From scattered iterations to a governed object model

Module 05 used MLflow to package and version unity_airways.rag.ua_rag_chain. Module 06 explains the objects underneath: an Experiment holds Runs; MLflow 3 adds a LoggedModel per app version; a good version is registered into Unity Catalog and promoted with an alias.

  • Experiments / Runs / Model Registry are MLflow's spine — a container, one tracked execution, and the place a finished model gets versioned and promoted.
  • MLflow 3 kept those and added a GenAI layer: Tracing, mlflow.genai.evaluate(), the Prompt Registry, and the LoggedModel that mlflow.set_active_model() creates.
  • Managed MLflow is the same pip package — Databricks just hosts the tracking server and moves the registry into Unity Catalog, adding governance for free.
  • Storage is split: a backend store for metadata (experiments, runs, params, metrics, LoggedModels) and an artifact store for files (the model, signature, plots).
📌 The rule that shapes this moduleTeach MLflow 3, not MLflow 2. The book (📘B1) is an Early Release and still shows mlflow.evaluate(...), stages, and positional log_model. Standardize on: mlflow.set_active_model() (LoggedModel, new in 3), mlflow.models.set_model() (Models-from-Code), register to UC via set_registry_uri("databricks-uc") + a three-level name, promote with aliases + tags (not stages), and evaluate with mlflow.genai.evaluate().
02 · The problem

Why spreadsheets, re-runs, and stages fail

A GenAI team changes prompt wording, k, the model, the corpus — constantly. A support bot that quotes refund policy has to be auditable. Naive record-keeping breaks:

  • Track it in a spreadsheet → can't link a version to the traces/metrics it produced; diverges from reality the moment two people iterate.
  • "Just re-run the notebook" → data drifts, libraries move, endpoints rename. Without recording environment, code version, and data version, reproducible is a hope.
  • Workspace registry + stages → doesn't share across workspaces, no UC governance; and Staging/Production stages are the deprecated MLflow-2 model.
  • Pickle it onto DBFS → breaks for a system of services (05.6), and a bare file has no signature, lineage, or access control.
  • Copy the book's mlflow.evaluate(model_type=...) → that's MLflow-2; the MLflow-3 GenAI surface is mlflow.genai.evaluate().

04 · Foundations

06.1–06.3 — MLflow, the 2→3 jump, tracking

06.1 What is MLflow? Experiments, Runs, Model Registry Theory

MLflow (open source since 2018) manages the ML/GenAI lifecycle. Its spine: an Experiment is a container for related Runs; a Run logs params, metrics, tags, and artifacts; the Model Registry versions and promotes a finished model.

import mlflow
mlflow.set_experiment("/Users/you@co.com/unity_airways_rag")   # container
with mlflow.start_run(run_name="rag_chain_k5"):                # one execution
    mlflow.log_param("k", 5)                                   # an input
    mlflow.log_metric("groundedness", 0.91)                    # a result
    mlflow.set_tag("dataset_version", "policies_v3")           # a label

06.2 ★ From MLflow 2 → MLflow 3 (what changed for GenAI) Theory Cornerstone

The upgrade preserved core tracking but added a GenAI layer: the LoggedModel + mlflow.set_active_model(), mlflow.genai.evaluate(), comprehensive Tracing, the Prompt Registry, and Models-from-Code. Full walkthrough in the MLflow 2 → 3 deep-dive.

⚠️ Gotcha · book lag📘B1 Ch1 shows mlflow.evaluate() as the "one-liner evaluation function" and frames versioning around stages ("staging vs production"). Both are MLflow-2. Use mlflow.genai.evaluate() with explicit scorers=[...], and aliases + tags instead of stages.

05 · Govern & version

06.4–06.6 — open vs managed, UC registry, lifecycle

Identical mlflow package either way. Databricks hosts the tracking server and moves the registry into Unity Catalog, so governance and cross-workspace sharing come for free (mirrors 📘B1 Fig 1-10).

06.4 Open-source vs Managed MLflow; workspaces ↔ catalogs Theory

Same OSS package, so no vendor lock-in. Databricks hosts the tracking server and puts the registry in Unity Catalog. A workspace binds to one or more catalogs (📘B1 Fig 1-9): two prod workspaces can share prod_catalog, while a dev workspace binds only to dev_catalog and stays isolated. Because the registry lives in UC, a model is reachable — with governance — from any bound workspace.

06.5 ★ Unity Catalog Model Registry — registration, aliases, tags Theory + Hands-on Cornerstone

On Databricks the registry is Unity Catalog: a three-level name, UC governance, cross-workspace reach. Full walkthrough in the UC Model Registry deep-dive.

import mlflow
from mlflow import MlflowClient

mlflow.set_registry_uri("databricks-uc")                       # registry = Unity Catalog
mlflow.register_model(model_uri="runs:/<run_id>/model",
                      name="unity_airways.rag.ua_rag_chain")   # three-level name

client = MlflowClient()
client.set_registered_model_alias(                             # promote with an ALIAS, not a stage
    name="unity_airways.rag.ua_rag_chain", alias="champion", version=3)
client.set_model_version_tag(
    name="unity_airways.rag.ua_rag_chain", version=3, key="eval_passed", value="true")

Aliases (@champion, @challenger) are movable pointers to a version — load with models:/unity_airways.rag.ua_rag_chain@champion. Promoting is re-pointing the alias; rollback re-points it back. Tags add searchable metadata.

⚠️ GotchaOlder MLflow used stages (Staging/Production) and transition_model_version_stage(...). In UC those are removed — translate any stage call to set_registered_model_alias(...).

06.6 Model lifecycle, governance & access control Theory

Registering starts a governed lifecycle: version control, lineage (each version links to its run/LoggedModel/notebook), controlled promotion (move an alias, gate on eval evidence), and governance (UC fine-grained RBAC + audit logs). Access control is a Unity Catalog concern — design promotion so only a reviewer role can move the production alias.

06 · Reproducibility & internals

06.7–06.8 — reproducibility, the two stores, nested runs

06.7 Reproducibility, signatures, metadata, archiving/cleanup Theory + Hands-on

MLflow logs the environment, libraries, and code version automatically. True reproducibility needs you to also pin the data version (log the source Delta table version) and the signature (input/output schema). Keep the registry tidy: tag deprecated versions, drop aliases from retired ones, delete dead versions.

from mlflow.models import infer_signature
signature = infer_signature(model_input={"messages": [...]},
                            model_output="a grounded answer string")
mlflow.set_tag("delta_source_version", "policies_delta@v7")    # data lineage

06.8 MLflow internals — backend store, artifact store, nested runs Theory + Hands-on

MLflow separates metadata from files. The backend (tracking) store holds experiments, runs, params, metrics, tags, and LoggedModels. The artifact store holds files — backed by DBFS, S3/ADLS/GCS, or a UC Volume. log_metric writes to the backend; log_model/log_artifact writes to the artifact store. Nested runs group children under a parent:

with mlflow.start_run(run_name="k_sweep") as parent:
    for k in (3, 5, 10):
        with mlflow.start_run(run_name=f"k={k}", nested=True):
            mlflow.log_param("k", k)
            mlflow.log_metric("groundedness", evaluate_k(k))
⚠️ GotchaBecause the two stores are separate, deleting a run in the UI marks it deleted in the backend store but does not always purge its artifacts from object storage. Reclaiming that space is a separate cleanup step.
07 · Worked thread

Unity Airways, iteration to governed version

  1. Experiment (06.1): set_experiment(".../unity_airways_rag") — the container for every RAG iteration.
  2. LoggedModel (06.2): set_active_model(name="rag_chain") creates the version hub; mlflow.langchain.autolog() streams traces in.
  3. Track (06.3): log k=5, temperature=0; metrics groundedness; tag dataset_version.
  4. Managed (06.4): lands on the workspace's hosted tracking server; registry in the unity_airways catalog.
  5. Register + promote (06.5): set_registry_uri("databricks-uc") → register_model(..., name="unity_airways.rag.ua_rag_chain") → alias @challenger.
  6. Govern (06.6): UC RBAC controls who moves @champion; promotion gated on eval evidence.
  7. Reproducibility (06.7): signature + pinned requirements + Delta source version + Git SHA recorded.
  8. Internals (06.8): metadata in backend store, code + MLmodel in artifact store; a k-sweep used nested runs. Hand @champion to Module 11.
08 · Field notes

What to remember

📌 ImportantThree objects: Experiment → Run → Model Registry. MLflow 3 adds a LoggedModel per version via mlflow.set_active_model(). Register to Unity Catalog (set_registry_uri("databricks-uc") + three-level name) and promote with aliases + tags. Access control is a UC concern.
💡 Tip · fieldStandardize two aliases per model — @champion (serving now) and @challenger (under eval). Serving targets @champion; promotion is a one-line alias move once the challenger wins. Name the LoggedModel by Git SHA so "which commit + data made this trace?" is one glance.
⚠️ GotchaDon't use mlflow.evaluate(model_type="databricks-agent") (MLflow 2) — use mlflow.genai.evaluate(). Don't promote with stages — removed in UC. Don't pickle a chain — use Models-from-Code. A UI-deleted run does not free artifact storage.
