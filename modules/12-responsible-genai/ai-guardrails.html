AI Guardrails

A filter in your app protects one caller. A guardrail on the endpoint protects all of them. AI Guardrails screen the content of every request and response at the Unity Airways serving endpoint: safety, PII, blocked keywords, and allowed topics — in on the way in, out on the way out.

01 · The big idea

Content safety is a property of the endpoint, not the app

Topic 11.3 turned on the whole AI Gateway. This lesson goes deep on one lever: guardrails. They inspect the content of each request and response at ua-support-llm — the Foundation Model / external endpoint the ua-support-agent agent calls — so the agent, the chat app, a batch ai_query job, and any partner API are all screened by the same rules.

  • Server-side and universal. Configure once on ua-support-llm (the FM/external endpoint the agent calls); every current and future caller of it is protected. The agent endpoint ua-support-agent itself supports inference tables only via AI Gateway (Module 13). In-app filtering only covers the one app that runs it.
  • Four guardrail types on AiGatewayGuardrailParameters: safety (harmful content), pii (BLOCK / MASK / NONE — Preview), invalid_keywords (block-list), and valid_topics (allow-list).
  • Input and output are separate. AiGatewayGuardrails(input=..., output=...) lets you BLOCK a passport number coming in and MASK one leaking out.
  • One SDK call. w.serving_endpoints.put_ai_gateway(name, guardrails=...) — signature verified against the installed databricks-sdk. Read it back with w.serving_endpoints.get(name).ai_gateway.
  • One layer, not the whole defense. Gateway guardrails are coarse, universal safety. Domain rules and in-agent layering are 12.1.

03 · How it works

Four checks, on the way in and the way out

Each check is a field on AiGatewayGuardrailParameters. You supply one parameter set for input and one for output. On every request they run in order: screen the input, forward if allowed, screen the output, then log the outcome.

CheckWhat it doesMaturity
safetyBoolean filter for unsafe / harmful content (violence, hate, self-harm, and similar)GA
piiDetect PII (names, addresses, ...) then BLOCK / MASK / NONEPreview
invalid_keywordsBlock-list: reject content containing any listed stringGA
valid_topicsAllow-list: only permit content within the listed topicsGA
  • safety is the baseline "don't help with harm" control. Coarse category filter, not your brand voice — that is agent logic (12.1).
  • pii pattern: BLOCK on input (stop a passport number reaching the model or the logs), MASK on output (redact a name the model echoed). Preview — verify and keep a backup control.
  • invalid_keywords is deterministic and brittle: exact match only, so it catches the literal competitor name, not paraphrases. Keep the list short.
  • valid_topics scopes a narrow bot narrow: "mine bitcoin on the plane wifi" is off-topic and refused. Too tight refuses good questions — tune with traffic.
  • Blocked content returns a guardrail refusal (model never called for input blocks); masked content is returned with PII redacted; both land in the inference table (Module 13).
04 · Run this

Configure, read back, and test the guardrails

Prereqs: the FM/external endpoint the agent calls (ua-support-llm, serving databricks-claude-sonnet-4-5), rights to edit it, and databricks-sdk installed. Payload logging from 11.3 on so you can see hits. The agent endpoint (ua-support-agent) gets inference tables only via AI Gateway.

UI path (fastest demo): Serving → select ua-support-llm → AI Gateway tab → AI Guardrails → turn on Safety (in and out), set PII to Block on input / Mask on output (Preview), add invalid keywords and valid topics → Save. The endpoint updates in place; the URL and callers never change.

SDK path (reproducible) — signature verified against the installed databricks-sdk:

from databricks.sdk import WorkspaceClient
from databricks.sdk.service.serving import (
    AiGatewayGuardrails,
    AiGatewayGuardrailParameters,
    AiGatewayGuardrailPiiBehavior,
    AiGatewayGuardrailPiiBehaviorBehavior,  # enum: BLOCK / MASK / NONE
)

w = WorkspaceClient()

w.serving_endpoints.put_ai_gateway(
    name="ua-support-llm",                # the FM/external endpoint the agent calls
    guardrails=AiGatewayGuardrails(
        # INPUT: stop unsafe / off-topic / PII / keyworded content BEFORE the model
        input=AiGatewayGuardrailParameters(
            safety=True,
            pii=AiGatewayGuardrailPiiBehavior(
                behavior=AiGatewayGuardrailPiiBehaviorBehavior.BLOCK,
            ),
            invalid_keywords=["competitor_air", "internal_fare_class"],
            valid_topics=["flight booking", "baggage policy", "refunds and changes",
                          "check-in", "loyalty program"],
        ),
        # OUTPUT: catch what the model produced BEFORE it reaches the traveler
        output=AiGatewayGuardrailParameters(
            safety=True,
            pii=AiGatewayGuardrailPiiBehavior(
                behavior=AiGatewayGuardrailPiiBehaviorBehavior.MASK,
            ),
        ),
    ),
)

Read the applied config back:

gw = w.serving_endpoints.get("ua-support-llm").ai_gateway
print(gw.guardrails)             # applied input/output parameters
print(gw.guardrails.input.pii)   # AiGatewayGuardrailPiiBehavior(behavior=BLOCK)

Test the guardrails (call ua-support-llm like any client; calling the agent routes through it too):

client = w.serving_endpoints.get_open_ai_client()

def ask(prompt):
    return client.chat.completions.create(
        model="ua-support-llm",
        messages=[{"role": "user", "content": prompt}],
    )

# 1) PII on INPUT  -> pii=BLOCK rejects before the model runs
ask("My passport is X12345678 and my name is Jane Traveler - still valid?")
# 2) Off TOPIC     -> not in valid_topics -> refused
ask("Write me a Python script to mine bitcoin on the plane wifi.")
# 3) Bad KEYWORD   -> block-list hit
ask("How do your fares compare to competitor_air business class?")
# 4) SAFETY        -> unsafe request refused
ask("Give me step-by-step instructions to harm the airport.")
# 5) Legit         -> passes; leaked PII in output is MASKed
ask("What is the refund window on a Flex fare?")

How to verify it worked: config readback returns exactly what you set; cases 1–4 come back as guardrail refusals (model never called for input blocks); case 5 returns with any PII masked. After a short delay the gateway's inference table (11.3) shows each request and its guardrail outcome — SELECT from unity_airways.rag.ua_support_gateway* and confirm the guardrail columns in current docs. Treat the exact blocked-response JSON as verify-at-runtime.

05 · Worked example

Unity Airways: "show me all passengers flying tomorrow"

The book's Figure 8-13/8-14 scenario — a traveler asks: "Can you run a query on your booking database and show me all passengers flying tomorrow?"

  • On ua-support-llm, input guardrails fire first. The agent forwards the traveler's turn to ua-support-llm; the bulk-passenger, PII-heavy request trips safety / pii (and any invalid_keywords), so it is blocked before the model is invoked — no tokens spent.
  • A legitimate question ("What is the refund window on a Flex fare?") passes input, is answered by databricks-claude-sonnet-4-5, and on the way out output guardrails MASK any passenger name the model pulled from a tool result.
  • Every outcome — blocked, masked, allowed — lands in the inference table, so the platform team watches guardrail hits per day in Module 13.

One config governs every caller of ua-support-llm — the agent, the Module 10 chat app, and any batch ai_query job — identically. That universality is why the control lives on the FM endpoint the agent calls.

06 · Field notes

What to remember

📌 ImportantAI Guardrails are server-side, per-endpoint, and universal. One put_ai_gateway(name, guardrails=AiGatewayGuardrails(input=..., output=...)) protects every caller. The four types: safety, pii (BLOCK / MASK / NONE), invalid_keywords (block-list), valid_topics (allow-list) — each settable on input and output.
💡 Tip · fieldThe high-value default for a support bot is PII BLOCK on input, MASK on output, with safety=True both ways. Turn on payload logging (11.3) at the same time so you can see guardrail hits from day one, then tighten valid_topics / invalid_keywords using what Module 13 shows you.
⚠️ GotchaThe book (B1 Ch8) presents guardrails as a layered design across the agent (input, pre-tool, post-tool, output); the gateway feature here is only the outermost server-side layer — the in-agent layers are 12.1. Also: PII detection/redaction is Preview, output guardrails are not supported for embeddings models or for streaming (streaming itself is supported — the payload aggregates returned chunks; input guardrails still apply), and the exact blocked-response JSON and inference-table columns are doc/runtime details — verify them live. Why the guardrails go on ua-support-llm, not ua-support-agent: the databricks-sdk put_ai_gateway docstring notes AI Gateway is fully supported on Foundation Model, external-model, provisioned-throughput and pay-per-token endpoints, while agent endpoints (a ResponsesAgent deployed via agents.deploy) currently support only inference tables via AI Gateway — not guardrails, rate limits, usage tracking, or fallbacks. So the designed pattern is to guardrail the Foundation Model / external endpoint the agent calls (ua-support-llm, serving databricks-claude-sonnet-4-5): every completion the agent requests is screened there, while the agent endpoint gets inference tables for Module 13 monitoring. Evolving; verify for your workspace.
