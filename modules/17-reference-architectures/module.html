Module 17 — Reference architectures and unifying GenAI systems · Topics 17.1–17.7 · Theory + Hands-on

Reference Architectures

The capstone module. Modules 00–16 built the pieces of the Unity Airways stack; here they connect into one system a review board can sign off on. The one idea: MLflow is the integration plane, and the MLflow Trace is the shared artifact that keeps every part honest.

01 · The big idea

Stop thinking in features. Start thinking in planes.

By the time an agent reaches production, calling the model is the easy part. The hard part is getting the app framework, agent runtime, retrieval, tools, background services, IDE assistants, and the eval stack to operate coherently. That is what MLflow becomes: not a tracing library or an eval API in isolation, but the plane that connects them all.

  • The lifecycle is the architecture (17.1) — Develop → Evaluate → Deploy → Monitor → Improve is a loop, and every module you built slots into one phase. B1 Ch2.
  • The trace is the system of record (17.2) — product, platform, agent, and eval teams all read the same MLflow Trace. Consistent span types + searchable tags. B1 Ch10.
  • OTel makes it cross-stack (17.3), Agent Server gives one serving contract (17.4), MCP + Skills bridge assistants (17.5), and third-party judges plug into one eval loop (17.6).
  • 17.7 is Capstone C4 — assemble and defend the whole platform.
📌 The rule that shapes this moduleMLflow is the integration plane. Four load-bearing surfaces: the trace (shared artifact), the serving contract (Agent Server / agents.deploy), the eval loop (native + third-party judges), and the MCP bridge (assistants + Skills) — with OpenTelemetry connecting it to the wider stack. Name maturity: GA — MLflow 3 Traces, Model Serving, agents.deploy, third-party judge integrations; experimental / newer — MLflow Agent Server, MLflow MCP server; Public Preview — managed MCP servers.
02 · The problem

The board sees three demos, not one system

You shipped a capable stack: retrieval (04–05), a governed tool-using agent (09), served and guardrailed (11–12), monitored (13), with a self-serve analytics path (14–15). Each works. But the review board asks the integration questions a demo can't answer:

  • When a request goes wrong, where do I look — app logs, agent runtime, retrieval service, gateway, or the eval report?
  • The support agent, the analytics Genie, and a nightly batch job each emit different signals. How do I trace one user journey across them?
  • A new team wants to call the agent from a mobile backend. Do they re-implement streaming, retries, and logging, or is there one contract?
  • Our platform team already runs OpenTelemetry. Do GenAI traces show up there, or is this a silo nobody watches?
  • A promising third-party evaluator looks better for one dimension. Adopt it and fragment the eval stack, or skip it?

These are integration questions, not feature questions. Answering them is what turns "some AI projects" into "a platform we operate."

04 · Lifecycle

17.1 — The 5-phase GenAI lifecycle as architecture

17.1 The 5-phase lifecycle Theory B1 Ch2

  • Develop — a minimal agent that looks up a booking, answers basic policy questions, cites sources; instrumented with MLflow Traces from the first working path. Goal: a minimum evaluable product.
  • Evaluate — curated scenarios (refunds, changes, baggage, disruptions) with scorers for groundedness and safety (cite official policy; never invent fee waivers). Module 08.
  • Deploy — promote a versioned application behind a controlled serving surface. Modules 11–12.
  • Monitor — collect real questions; reveal drift in refund answers when policy context is missing, or odd behavior when a tool times out. Module 13.
  • Improve — targeted changes (tighten retrieval filters, clarify prompt sections), then validate against the same dataset before promoting. Module 13.7.
💡 Why it's architectureThe phases are held together by durable artifacts and decision points (B1 Table 2-1): consistent tracing schema, versioned eval datasets, business-aligned scorers, promotion gates, versioned prompts/models/configs, lineage for rollback, and cost visibility. The architecture is those shared artifacts.
05 · Trace

17.2 — MLflow Traces as the shared integration artifact

17.2 Trace as shared artifact Theory B1 Ch10 GA

  • The trace is the system of record for what actually happened across the application — the same artifact read by product engineers (request/response), platform teams (latency/failures/paths), agent developers (tool calls, retrieval, decisions), and the eval workflow (scores it later without reconstructing context).
  • It matters more in heterogeneous systems — a request crossing a TypeScript frontend, Python orchestration, a Java retrieval service, and a Go tool backend. A common trace format pays for itself as diversity grows.
  • Conventions make it work: consistent span types (AGENT, TOOL, RETRIEVER), stable names for recurring operations, and searchable tags (user_id, session_id, environment, app_version).
📌 One atom, five jobsThe same Unity Airways trace you instrumented in Module 07 is auto-captured in production (13), scored offline and online with the Module 08 scorers, and — when it fails — curated into the eval set (13.7). Signal, scorer input, dashboard row, alert trigger, next eval-set record.
06 · Trace

17.3 — OpenTelemetry for cross-stack tracing

17.3 OpenTelemetry cross-stack Theory Hands-on OTel-compatible

  • Many apps already run OpenTelemetry (OTel) for distributed tracing across services, gateways, and infrastructure. MLflow's OTel compatibility lets GenAI traces join those workflows instead of living on an island.
  • Two directions: (1) export MLflow traces to an OTel collector so they flow into existing observability platforms; (2) bring OTel-instrumented app activity into Databricks tracing (upstream + downstream visibility).
  • "Dual export" is B1's word for the goal — the same trace reaches both MLflow and another OTel destination. The current docs mechanism isn't a single MLFLOW_ENABLE_DUAL_EXPORT flag: point OTel at your collector, then combine MLflow + OTel spans into one trace via MLFLOW_USE_DEFAULT_TRACER_PROVIDER="false" + mlflow.tracing.set_destination(...).
# Send spans to an OTel collector AND combine with MLflow (current MLflow OTel docs)
import os
import mlflow
from mlflow.entities.trace_location import MlflowExperimentLocation

# 1) Export spans to your external OTel collector via OTLP
os.environ["OTEL_EXPORTER_OTLP_TRACES_ENDPOINT"] = "<collector-endpoint-url>"
os.environ["OTEL_SERVICE_NAME"] = "unity-airways-support"   # optional: groups traces

# 2) Run MLflow AND OTel together so both span sets land in ONE trace
os.environ["MLFLOW_USE_DEFAULT_TRACER_PROVIDER"] = "false"
mlflow.set_tracking_uri("databricks")
mlflow.tracing.set_destination(MlflowExperimentLocation(experiment_id="<experiment-id>"))
⚠️ GotchaTo display OTel traces meaningfully, apps must emit the expected span types and attributes. Those field-level mappings evolve faster than the architecture — B1 deliberately does not reproduce them. Read the current MLflow/Databricks tracing docs. ⚠️ live re-check pending on exact semantic-convention field names.
07 · Serving

17.4 — MLflow Agent Server as a stable serving layer

17.4 Agent Server Theory B1 Ch10 Newer / verify

  • The problem: a web chat, a mobile backend, and an internal tool may all call the same agent, yet each accumulates its own request handling, streaming, retry, and logging — the agent fragments and behaves differently per entry point.
  • MLflow Agent Server is a FastAPI-based server in front of the runtime: exposes agents through an /invocations endpoint, supports decorator registration with @invoke() / @stream(), validates against the Responses API schema, and wires in tracing automatically.
  • The benefit: clients integrate with one serving contract, not agent internals — you can change internal components without forcing consumers to reimplement.
💡 The managed equivalent on Databricksagents.deploy(uc_model_name, version) creates a Model Serving endpoint (our ua-support-agent) plus a Review App, and enables tracing, inference tables, and monitoring. Same idea — one stable contract with tracing built in — managed for you. MLflow Agent Server is a newer MLflow feature; confirm exact status/API on mlflow.org/docs/latest before a customer build.
08 · Bridge

17.5 — MCP as the bridge to assistants and IDEs; MLflow Skills

17.5 MCP + MLflow Skills Theory Hands-on MCP experimental · managed MCP Preview

  • The MLflow MCP server gives assistants and IDE agents a standard, programmatic way to interact with MLflow: search/retrieve trace data, analyze performance, log feedback/assessments, manage tags/metadata — not just as a destination UI.
  • Why it changes the workflow: instead of pasting screenshots into chat, the assistant operates on the underlying trace data directly — find recurring failure patterns, compare related traces, spot which tool call caused a regression. It becomes part of the observability workflow.
  • MLflow Skills — described in B1 Ch10 as a newer, agent-authored playbook set (instrument tracing, analyze traces, reconstruct chat sessions, set up evaluation, query metrics, onboard, search docs), portable across Claude Code, Cursor, Codex, Gemini CLI, and OpenCode. This project bundles skills of exactly this shape (e.g. instrumenting-with-mlflow-tracing) — confirm current availability/naming in the MLflow docs before citing it.
  • On Databricks: managed, UC-governed MCP servers for Genie, AI Search, UC functions, and Databricks SQL (Public Preview).
// Version gate first:  pip install 'mlflow[mcp]>=3.5.1'
// MCP client config (coding-assistant settings), per current MLflow docs.
// Points an assistant at MLflow's MCP server.
{
  "mcpServers": {
    "mlflow": {
      "command": "uv",
      "args": ["run", "--with", "mlflow[mcp]>=3.5.1", "mlflow", "mcp", "run"],
      "env": {
        "MLFLOW_TRACKING_URI": "databricks",
        "DATABRICKS_HOST": "https://<your-workspace-host>",
        "DATABRICKS_TOKEN": "<token>"
      }
    }
  }
}
⚠️ GotchaB1 states plainly: at time of writing the MLflow MCP server does not support interacting with traces stored in Unity Catalog — the direction is clear but the workflow still depends on where traces are stored. MCP server is experimental; managed MCP servers are Public Preview. ⚠️ live re-check pending on the exact launch command and UC-trace support.
09 · Measure

17.6 — Third-party judges and ecosystem integration

17.6 Third-party judges Theory B1 Ch10 GA integrations

  • Evaluation is part of the integration plane. MLflow works as a common scorer interface for evaluation methods that come from outside the core project.
  • The eval ecosystem moves fast — different tools lead on conversational quality, retrieval quality, hallucination detection, safety, summarization, agent behavior. The challenge: adopt them without a second, disconnected eval stack.
  • MLflow lets third-party judges run in the same workflow as native scorers — today including DeepEval, RAGAS, Arize Phoenix, and TruLens. Mix built-in and external evaluators while keeping one evaluation loop.
📌 In our stackThe Module 08 mlflow.genai.evaluate(...) scorecard (Correctness, RetrievalGroundedness, Safety) that gates promotion of ua_support_agent is the same loop a RAGAS retrieval judge or a DeepEval hallucination judge plugs into — no parallel pipeline, one scorecard.

12 · Field notes

What to remember

📌 ImportantMLflow is the integration plane. Four load-bearing surfaces — the trace (17.2), the serving contract (17.4), the eval loop (17.6), and the MCP bridge (17.5) — connected to the wider stack by OTel (17.3). Two planes (Unity Catalog governance + MLflow 3 observability) turn three demos into one platform (17.7).
💡 Tip · fieldOn a whiteboard, draw planes first (UC + MLflow), then hang the boxes (RAG, agent, Genie, monitoring) on them — it reframes "which product?" into "how does it stay coherent?" Standardize span types and tags in dev; they carry to prod unchanged and make cross-release comparison possible.
⚠️ GotchaB1 is an O'Reilly Early Release (RAW & UNEDITED) — Agent Server, MLflow MCP (no UC-trace support yet), and OTel field mappings are current-direction but fast-moving; live re-check before a customer. And remember: AI Gateway on agent endpoints supports only inference tables — guardrails/rate-limits need an FM or external-model endpoint (Module 11/12), so don't over-claim gateway features on ua-support-agent.
