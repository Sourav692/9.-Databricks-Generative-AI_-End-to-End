Module 05 — Building & Versioning a RAG Chain · Topics 05.1–05.7 · Theory + Hands-on

Building and Versioning a RAG Chain

A retriever isn't an app. This module wires the Unity Airways Module 04 retriever to ChatDatabricks, assembles a LangChain RAG chain, then packages, logs, registers, and versions it with MLflow 3 — the point where "we have an index" becomes "we have an app."

01 · The big idea

From a retriever to a versioned RAG chain

Module 04 handed us a retriever: one call returns the five most relevant policy chunks. This module glues retrieve → prompt → generate into one chain, then makes it shippable — a signature, dependent resources, Models-from-Code logging, a UC-registered model, and MLflow-tracked versions.

  • A RAG chain is a small LCEL pipeline: extract the question, retrieve chunks, format them into a prompt, send it to ChatDatabricks, parse the answer.
  • Build incrementally: an LLM-only baseline first, then add the retriever for a full RAG chain — and compare the two versions in MLflow.
  • Keep config (endpoint, index, k, prompt) in a YAML loaded by ModelConfig, so a version is "this code + this config."
  • Package with a signature + dependent resources, log via Models-from-Code (mlflow.models.set_model()), register to UC as unity_airways.rag.ua_rag_chain, and version each iteration with mlflow.set_active_model().
📌 Two naming rules(1) The LangChain integration is databricks-langchain — from databricks_langchain import ChatDatabricks, DatabricksVectorSearch, DatabricksEmbeddings — not langchain-databricks, not langchain_community. (2) On MLflow ≥ 3.1, log chains with Models-from-Code (mlflow.models.set_model()) and version them with mlflow.set_active_model() (LoggedModel).
02 · The problem

Why gluing it by hand — and pickling it — fails

The retriever returns passages, not answers. The support agent needs the system to read those chunks and write a grounded reply, the same way every time, then version and serve it. Naive shortcuts break:

  • Paste chunks into a prompt by hand → no reusable object, no schema, no version history. Fine for one demo, useless for an eval loop or serving.
  • Hard-code endpoint/index names → every experiment is a code edit; you can't tell which config produced which result. Externalize to YAML.
  • Pickle the chain → a RAG chain is a system of live services, and the VectorStoreRetriever inside it isn't serializable — flavor logging throws Failed to save runnable sequence.
  • Deploy and hope auth works → a served chain has no notebook creds; without declared resources=[...] it can't reach the index or the LLM.
  • Track versions in a spreadsheet → manual, and can't link a version to the traces/metrics it produced. MLflow 3's LoggedModel does.

04 · Build the chain

05.1–05.4 — anatomy, integration, LLM→RAG, memory

05.1 What is a "chain"? RAG chain anatomy Theory

A chain is ordered steps composed with LCEL's | pipe. A RAG chain adds a retrieval step: extract query → retrieve → format context → prompt → generate → parse. The context branch and a question passthrough run in parallel, then merge at the prompt.

from operator import itemgetter
from langchain_core.runnables import RunnableLambda
from langchain_core.output_parsers import StrOutputParser

chain = (
    {
        "context":  itemgetter("messages") | RunnableLambda(extract_user_query_string)
                    | retriever | RunnableLambda(format_docs),
        "question": itemgetter("messages") | RunnableLambda(extract_user_query_string),
    }
    | prompt | model | StrOutputParser()   # model = ChatDatabricks
)

The chain's input contract is {"messages":[{"role":"user","content":"..."}]} — the OpenAI chat shape, not a bare string. chain.get_graph().draw_mermaid_png() renders the wiring.

05.2 LangChain ↔ Databricks integration Hands-on

Two classes from databricks-langchain do the wiring. ChatDatabricks wraps a serving endpoint; DatabricksVectorSearch wraps the Module 04 index and returns the retriever.

from databricks_langchain import ChatDatabricks, DatabricksVectorSearch

model = ChatDatabricks(endpoint="databricks-claude-sonnet-4-5", temperature=0, max_tokens=1000)

vs = DatabricksVectorSearch(
    endpoint="unity-airways-vs",
    index_name="unity_airways.rag.ua_rag_chunks_index",
    columns=["chunk_id", "content", "source_doc"],
)
retriever = vs.as_retriever(search_kwargs={"k": 5})   # the SAME object Module 04 built

05.3 ★ LLM-only app → full RAG chain Hands-on Cornerstone

Build in two steps so you can measure what retrieval buys you. Full walkthrough in the LLM-only → RAG chain deep-dive.

import mlflow
mlflow.langchain.autolog()                    # auto-trace every call

# Step 1 — baseline: just the LLM, no context
mlflow.set_active_model(name="llm_only")
# ChatDatabricks answers confidently but ungrounded

# Step 2 — full RAG chain, config in YAML
from mlflow.models import ModelConfig
config = ModelConfig(development_config="../conf/rag_chain_config.yml")
mlflow.set_active_model(name="rag_chain")     # new version to compare against llm_only
chain.invoke({"messages": [{"role": "user",
    "content": "Can I rebook for free if my connection is cancelled?"}]})

Both versions show in the MLflow Versions tab; every question is a trace under its version. Enable the Version column in Traces and confirm rag_chain shows both the retriever and ChatDatabricks spans.

05.4 Memory and context management Theory + Hands-on

The base chain is stateless. Memory preserves earlier turns so a follow-up ("and for business class?") builds on the last one. Inject it via a {chat_history} slot in the prompt, alongside {context} and {question}.

Memory typeKeepsBest for
BufferLast few turns verbatimShort chats; support / form-filling bots
Conversation summaryRunning summary of old turnsLong chats where topic > exact wording
EntityNamed entities (people, order IDs)Personalized flows; CRM / assistant bots

Short-term memory lives in one conversation; long-term persists across sessions in an external store. History affects retrieval too — a bare follow-up may need the earlier subject folded in before it hits the retriever.

05 · Package, log, version

05.5–05.7 — signatures + resources, code vs flavor, versioning

Every arrow into log_model is something the served chain needs to run outside the notebook. set_active_model opens the version and anchors its traces/evals.

05.5 Packaging & logging: signatures, dependent resources Theory + Hands-on

Two pieces are mandatory before logging. A signature (input/output schema) and the dependent resources that enable automatic authentication passthrough when the chain is served.

from mlflow.models import infer_signature
from mlflow.models.resources import DatabricksServingEndpoint, DatabricksVectorSearchIndex

signature = infer_signature(model_input=config.get("input_example"),
                            model_output=config.get("output_example"))

dependent_resources = [
    DatabricksServingEndpoint(endpoint_name="databricks-claude-sonnet-4-5"),
    DatabricksVectorSearchIndex(index_name="unity_airways.rag.ua_rag_chunks_index"),
]
⚠️ GotchaFlavor logging serializes the chain, and the VectorStoreRetriever inside DatabricksVectorSearch is not natively serializable — you'll hit Failed to save runnable sequence ... a loader_fn must be provided. Old fix: a loader_fn(persist_directory). Better fix: Models-from-Code (05.6). And if a served chain 401s but the notebook works, you forgot resources=[...].

05.6 ★ Logging "Model as Code" vs LangChain flavor Theory + Hands-on Cornerstone

Both call mlflow.langchain.log_model; the difference is what you pass as lc_model. Full walkthrough in the Model-as-Code deep-dive.

# rag_chain.py  — build the chain, then mark it:
mlflow.models.set_model(model=chain)     # THE crucial line for Models-from-Code

# log the SCRIPT PATH, not the object:
logged = mlflow.langchain.log_model(
    lc_model="rag_chain.py",
    code_paths=["helpers/"],                     # helper modules the script imports
    model_config="../conf/rag_chain_config.yml", # config saved with the model
    signature=signature,
    resources=dependent_resources,
    registered_model_name="unity_airways.rag.ua_rag_chain",  # register to UC
)

Models-from-Code saves the source instead of pickling the object — no serialization errors, scales to complex apps, and is the GA-recommended path for chains and agents. The artifacts show rag_chain.py + helpers/ + config, not a pickle.

⚠️ Gotcha · scopeThis module stops at LangChain chains. Authoring agents with ResponsesAgent — which also uses mlflow.models.set_model() — is Module 09. Same logging primitive, different authoring interface.

05.7 MLflow app versioning for chains Hands-on

mlflow.set_active_model(name=...) creates a LoggedModel — a metadata hub for one version (new in MLflow 3). Traces, params, evals, and the artifact all link to it automatically.

active = mlflow.set_active_model(name="rag_chain")
app_params = {"model_name": databricks_resources.get("model_name"),
              "retriever_config": retriever_config.get("parameters"),   # {"k": 5}
              "llm_config": llm_config.get("llm_parameters")}
mlflow.log_model_params(model_id=active.model_id, params=app_params)
  • The Versions tab lists each version with its params; the Traces tab shows which version produced each answer.
  • For real reproducibility, name the LoggedModel by Git commit hash and record the Databricks Runtime and source Delta table version as metadata.
  • Versioning is the basis for evaluation (Module 08): scorers attach metrics to a specific version, so you can promote the winner with numbers, not a vibe check.
06 · Worked thread

Unity Airways, retriever to registered chain

  1. Retriever (04): DatabricksVectorSearch(...).as_retriever(search_kwargs={"k":5}) on unity_airways.rag.ua_rag_chunks_index.
  2. Baseline (05.3): set_active_model("llm_only") + ChatDatabricks("databricks-claude-sonnet-4-5") — ungrounded answer.
  3. Config (05.3): endpoint, index, k=5, prompt in rag_chain_config.yml, loaded by ModelConfig.
  4. Chain (05.1): LCEL parallel context/question → PromptTemplate → ChatDatabricks → StrOutputParser. set_active_model("rag_chain").
  5. Memory (05.4): add a {chat_history} buffer of the last N turns for follow-ups.
  6. Package (05.5): infer_signature(...) + resources=[DatabricksServingEndpoint, DatabricksVectorSearchIndex].
  7. Log as code (05.6): set_model(chain) in rag_chain.py; log_model(lc_model="rag_chain.py", ..., registered_model_name="unity_airways.rag.ua_rag_chain").
  8. Version (05.7): each iteration a LoggedModel; compare llm_only vs rag_chain; hand off to Module 09 / 11.
07 · Field notes

What to remember

📌 ImportantImport from databricks-langchain (never langchain-databricks / langchain_community). The chain's input is {"messages":[...]}. Log chains as Models-from-Code (mlflow.models.set_model()) and version them with mlflow.set_active_model(). A served chain needs a signature + resources=[...] to run and authenticate.
💡 Tip · fieldSeparate config into YAML so a version is "this code + this config" — changing k or the prompt is one line + a fresh set_active_model. Name versions by Git SHA, not v1/v2, so "which commit made this trace?" is one glance.
⚠️ GotchaServed-model names churn — confirm databricks-claude-sonnet-4-5 on the supported-models page (the book's databricks-claude-3-7-sonnet is older). Flavor-logging a VectorStoreRetriever throws Failed to save runnable sequence → use Models-from-Code. Don't buffer unlimited memory — it overflows the context window.
