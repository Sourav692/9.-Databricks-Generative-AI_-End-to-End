Module 05 — Building and versioning a RAG chain · Topic 05.3 · Hands-on ★

RAG Chain

Start from a one-line LLM app that confidently invents policy. Add a retrieval step, a prompt, and a parser with LCEL, and the same model answers from Unity Airways' real documents. The chain object you build here is what Module 05.5/05.6 will log and register.

01 · The big idea

Same model, but now it reads your documents

Ask a raw foundation model "Can I get a refund if I miss my connection?" and it answers in fluent, official-sounding language — describing a generic airline policy it never learned from Unity Airways. A RAG chain puts a retrieval step in front of the model: fetch the chunks that actually answer the question, paste them into the prompt as context, and tell the model to answer only from that.

  • An LLM-only app is one line: llm.invoke("..."). It answers from training memory, so it can hallucinate a plausible-but-wrong Unity Airways rule.
  • A RAG chain = retriever + prompt + model + parser, wired with LCEL (the | pipe). The retriever from Module 04 becomes the "R"; ChatDatabricks is the "G".
  • The whole chain is one expression: {"context": retriever | format_docs, "question": RunnablePassthrough()} | prompt | llm | StrOutputParser().
  • Turn on mlflow.langchain.autolog() once and every chain.invoke(...) becomes a trace — you can see the exact chunks retrieved and why the model said what it said.

03 · How it works

What retrieval actually changes

The baseline and the RAG chain use the same ChatDatabricks model and the same question. The only difference is the retrieval step that injects real Unity Airways text before the model answers.

Prove the gap with the baseline first, then wire retrieval in and re-ask the same question.
  • The retriever is the "R" — reused, not rebuilt. You wrap the Module 04 index with DatabricksVectorSearch(...).as_retriever(search_kwargs={"k":5}). k is a real dial: too small misses the answer chunk, too large pads the prompt with noise.
  • format_docs bridges Documents → string. The retriever returns a list of Documents; the prompt slot needs one string, so join each .page_content.
  • The prompt enforces grounding. The system message — "answer only from the context, say you don't know otherwise, cite the source" — is what keeps the model on your documents.
  • The parallel dict feeds two slots. LCEL runs both dict values on the same question: one branch builds {context}, the other passes the raw question to {question}.
  • Tracing makes it observable. mlflow.langchain.autolog() records each invoke as a trace with retriever, prompt, and model spans — open the retriever span to see exactly which chunks answered.
04 · Run this

Baseline, retriever, chain, trace

Prereqs: serverless or DBR ML runtime, MLflow ≥ 3.1; the Module 04 index unity_airways.rag.ua_rag_chunks_index online on endpoint unity-airways-vs; access to a chat serving endpoint.

%pip install -U databricks-langchain databricks-vectorsearch mlflow
dbutils.library.restartPython()
CATALOG       = "unity_airways"; SCHEMA = "rag"
VS_ENDPOINT   = "unity-airways-vs"
INDEX_NAME    = f"{CATALOG}.{SCHEMA}.ua_rag_chunks_index"
CHAT_ENDPOINT = "databricks-claude-sonnet-4-5"   # confirm on the supported-models page

Trace first so even the baseline call is captured:

import mlflow
mlflow.langchain.autolog()   # every LangChain .invoke() now emits an MLflow trace

The LLM-only baseline — watch it answer without your documents:

from databricks_langchain import ChatDatabricks

llm = ChatDatabricks(endpoint=CHAT_ENDPOINT, temperature=0)   # deterministic while developing
print(llm.invoke("Can I get a refund if I miss my connection?").content)
# Fluent and official-sounding — but a GENERIC airline policy, not Unity Airways'.

Build the retriever (wrap the Module 04 index):

from databricks_langchain import DatabricksVectorSearch

vector_store = DatabricksVectorSearch(
    endpoint=VS_ENDPOINT,
    index_name=INDEX_NAME,
    columns=["chunk_id", "content", "source_doc"],   # source_doc so answers are citable
)
retriever = vector_store.as_retriever(search_kwargs={"k": 5})

format_docs + the prompt (the prompt is where grounding is enforced):

from langchain_core.prompts import ChatPromptTemplate

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)   # Documents -> one string

prompt = ChatPromptTemplate.from_messages([
    ("system",
     "You are the Unity Airways policy assistant. Answer using ONLY the context below. "
     "If the context does not contain the answer, say you don't know. Cite the source_doc.\n\n"
     "Context:\n{context}"),
    ("human", "{question}"),
])

Assemble the RAG chain with LCEL, then invoke with a bare string:

from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

answer = chain.invoke("Can I get a refund if I miss my connection?")
print(answer)   # grounded in the Conditions of Carriage, names the source_doc

How to verify it worked: the answer now uses Unity Airways' actual language and cites a source_doc. Open the trace (Experiments → Traces): the retriever span should show the refund/connection chunks. Still generic? Read that span — the fix is upstream in retrieval, not the model. This chain object is exactly what Module 05.5 logs and 05.6 registers as "Model as Code."

05 · Field notes

What to remember

📌 ImportantThe baseline and the RAG chain use the same model. RAG doesn't make the model smarter — it changes what the model sees. All the quality comes from putting the right chunks in the prompt, so when a RAG answer is wrong, suspect retrieval (chunking, k, index columns) before you touch the prompt or swap the model.
💡 Tip · fieldBuild the chain incrementally and test each link with .invoke() before piping: retriever.invoke(q), then (retriever | format_docs).invoke(q), then prompt.invoke({"context":..., "question":q}), then the whole chain. When something breaks you already know which link failed. Keep temperature=0 so runs are reproducible.
⚠️ GotchaBook 1 (Ch 4) builds the same chain in a production-shaped variant: the input is an OpenAI-style {"messages":[...]} object and each branch starts with itemgetter("messages") | RunnableLambda(extract_user_query_string), using PromptTemplate and RunnableLambda(format_docs). That is what a deployed chat endpoint receives in production. This lesson — and the 05.6 packaging deep-dive — use the simpler string-in form (RunnablePassthrough() + ChatPromptTemplate) so the wiring is clear — same chain, simpler input. The book also pins databricks-claude-3-7-sonnet; the catalog moves fast, so confirm databricks-claude-sonnet-4-5 on the supported-models page.
06 · Notes & self-check

Test yourself

Reveal 5-question self-check
  1. In {"context": retriever | format_docs, "question": RunnablePassthrough()}, what input does each branch receive, and what does RunnablePassthrough() do?
  2. Why must the chain end with StrOutputParser(), and what does chain.invoke(...) return without it?
  3. You get a fluent but wrong answer. What do you inspect first, and where in the MLflow trace do you look?
  4. What is the job of format_docs, and what breaks if it returns the raw Document list?
  5. Why invoke this chain with a bare string rather than a dict, and what changes if the input is {"messages":[...]}?
