Module 05 — Building and versioning a RAG chain · Topic 05.6 · Theory + Hands-on ★

Model as Code

One function, two inputs. Hand mlflow.langchain.log_model the in-memory chain and you fight the pickler; hand it a .py file with set_model() and MLflow keeps the recipe — reproducible, deployable, and ready for Unity Catalog.

01 · The big idea

Same door, two inputs

You finished the Unity Airways RAG chain in 05.3 — a DatabricksVectorSearch retriever feeds chunks to ChatDatabricks("databricks-claude-sonnet-4-5"). Now you have to log it so it reloads and deploys identically. MLflow offers two ways in through one call, and the whole lesson is which one you pick.

  • The brittle way: pass the in-memory chain object. MLflow cloudpickles it. The retriever holds a live client that will not pickle, so logging fails with VectorStoreRetriever ... loader_fn must be provided — and the workaround gets messy fast.
  • The recommended way — Models-from-Code: put the chain in a .py file ending with mlflow.models.set_model(chain), then log with lc_model="rag_chain.py". MLflow stores the code. Loading re-runs the script and rebuilds the chain fresh.
  • Attach dependent resources so the deployed model gets scoped credentials automatically: DatabricksVectorSearchIndex + DatabricksServingEndpoint.
  • Add a signature, then register to Unity Catalog as unity_airways.rag.ua_rag_chain. Topic 05.7 layers on LoggedModel and mlflow.set_active_model().

03 · How it works

Why live objects do not pickle

Pick the input by what your chain holds. A chain with any live client — a retriever, a serving-endpoint LLM, a tool — should be logged as code.

Prefer the code-file input. Reach for the object input only for a tiny, fully serializable chain.
  • cloudpickle snapshots data, not connections. A DatabricksVectorSearch retriever is a handle to a running service — client, auth, connection state. There is no meaningful byte snapshot of an open connection, so LangChain marks it non-serializable and MLflow refuses.
  • set_model names the model. The last line of the .py file, mlflow.models.set_model(model=chain), tells MLflow which object in the script is the model to log. It does not log anything by itself.
  • Loading re-executes the script. mlflow.langchain.load_model(uri) runs rag_chain.py again, constructing a fresh retriever and LLM client. Nothing is unpickled, so the fragile step is gone.
  • Resources drive deploy-time auth. You declare the services the chain calls; deployment reads the list and grants the endpoint exactly those permissions. Miss one and it fails at inference, not at logging.
  • The pyfunc twin. For non-LangChain code (a ResponsesAgent), mlflow.pyfunc.log_model(python_model="agent.py", ...) is the same Models-from-Code idea in a different flavor.
04 · Run this

File, resources, log, verify

Prereqs: serverless or DBR ML runtime, MLflow ≥ 3.1; the RAG chain from 05.3; read access to unity_airways.rag.ua_rag_chunks_index; rights to register a model in unity_airways.rag.

%pip install -U mlflow databricks-langchain databricks-vectorsearch
dbutils.library.restartPython()

1. Move the chain into rag_chain.py and end the file with set_model:

# rag_chain.py — the entire chain lives here, no notebook state
import mlflow
from databricks_langchain import ChatDatabricks, DatabricksVectorSearch
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

cfg = mlflow.models.ModelConfig(development_config="rag_chain_config.yml")
retriever = DatabricksVectorSearch(
    index_name=cfg.get("index_name"),
    columns=["chunk_id", "source_doc", "content"],
).as_retriever(search_kwargs={"k": 5})
prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer using only the context.\n\nContext:\n{context}"),
    ("user", "{question}"),
])
llm = ChatDatabricks(endpoint=cfg.get("llm_endpoint"), temperature=0)
chain = ({"context": retriever, "question": RunnablePassthrough()}
         | prompt | llm | StrOutputParser())

mlflow.models.set_model(model=chain)   # ← the crucial line: THIS is the model

Sanity-check the file before logging (catch errors while you can see them):

from rag_chain import chain
print(chain.invoke("Can I get a refund on a Basic Economy fare?")[:200])

2. Signature + dependent resources (resources drive auto-auth on deploy):

from mlflow.models import infer_signature
from mlflow.models.resources import DatabricksVectorSearchIndex, DatabricksServingEndpoint

signature = infer_signature(model_input="Can I change my booking?",
                            model_output="A short grounded answer.")
resources = [
    DatabricksVectorSearchIndex(index_name="unity_airways.rag.ua_rag_chunks_index"),
    DatabricksServingEndpoint(endpoint_name="databricks-claude-sonnet-4-5"),
]

3. Log as Models-from-Code — lc_model= is the file path, not the object:

with mlflow.start_run() as run:
    logged = mlflow.langchain.log_model(
        lc_model="rag_chain.py",                 # ← code file, NOT the chain object
        model_config="rag_chain_config.yml",     # config injected at load time
        signature=signature,
        input_example="Can I change my booking?",
        resources=resources,                     # auto-auth on deploy
        pip_requirements=["mlflow>=3.1", "databricks-langchain", "databricks-vectorsearch"],
        # code_paths=["helpers/"],               # if rag_chain.py imports local helpers
        registered_model_name="unity_airways.rag.ua_rag_chain",   # log + register in one shot
    )
print(logged.model_uri)   # runs:/<run_id>/model

Verify it worked — reload and invoke (this re-executes the code, not a pickle):

loaded = mlflow.langchain.load_model(logged.model_uri)
print(loaded.invoke("What is the checked baggage allowance on Basic Economy?")[:200])

Register separately (alternative to registered_model_name=):

mlflow.set_registry_uri("databricks-uc")
mv = mlflow.register_model(f"runs:/{run.info.run_id}/model", "unity_airways.rag.ua_rag_chain")
print(mv.name, mv.version)   # unity_airways.rag.ua_rag_chain, 1

The brittle object path — shown so you recognize the error, not to ship:

with mlflow.start_run():
    mlflow.langchain.log_model(lc_model=chain, signature=signature, resources=resources)
# -> MlflowException: ... 'VectorStoreRetriever ... a `loader_fn` must be provided.'

# Fix B (do this): log the .py file as in step 3.

05 · Field notes

What to remember

📌 ImportantThe whole lesson is one fork in one function. mlflow.langchain.log_model(lc_model=...) behaves completely differently depending on whether you pass an object (serialize, fragile) or a file path (store code, reproducible). For any real chain with a retriever or a serving-endpoint client, pass the file path.
💡 Tip · fieldMake the .py file the source of truth even during development — build the chain there, from rag_chain import chain, and invoke it in the notebook. You get the same object to iterate on, and logging is then a one-line lc_model="rag_chain.py". Read every environment-specific name via mlflow.models.ModelConfig so the same file logs cleanly in dev, stage, and prod.
⚠️ GotchaMissing resources is a silent trap. The model logs and registers without complaint and only fails when it is deployed and actually tries to call Vector Search or the LLM — an auth error, not a logging error. Declare every Databricks service the chain touches. Class names and kwargs — DatabricksVectorSearchIndex(index_name=...), DatabricksServingEndpoint(endpoint_name=...) from mlflow.models.resources — are confirmed in 📘B1 Ch4; re-verify against current MLflow docs, since resource classes are added over time.
